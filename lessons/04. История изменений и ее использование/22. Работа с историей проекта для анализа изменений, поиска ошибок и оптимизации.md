Приветствую, коллеги. Сегодня мы завершаем большой раздел, посвящённый истории в Git. Я, как архитектор, смотрю на историю не просто как на цепочку коммитов, а как на **живую документацию проекта**. Умение читать и анализировать эту историю — признак зрелого разработчика. Мы собрали целый арсенал инструментов, и сейчас я покажу, как они работают в связке, чтобы превратить расследование бага из паники в отлаженный процесс.

### 1. Обзор инструментов историка

Каждый из этих инструментов отвечает на свой вопрос. Вместе они дают полную картину эволюции кода.

| Инструмент | Вопрос, на который отвечает | Пример использования |
| :--- | :--- | :--- |
| **`git log`** | **Что** происходило в проекте? (Хронология) | `git log --oneline --graph --all` |
| **`git blame`** | **Кто** последним трогал конкретную строку? | `git blame -L 10,20 src/app.js` |
| **`git grep`** | **Где** в истории встречается определённый код? | `git grep "functionName" HEAD~10` |
| **`git bisect`** | **Когда** именно появилась ошибка? (Бинарный поиск) | `git bisect start HEAD v1.0` |
| **`git reflog`** | **Куда** я перемещал HEAD в последнее время? (Локальный журнал ссылок) | `git reflog` |
| **`git show`** | **Что именно** изменилось в конкретном коммите? | `git show a1b2c3d` |
| **`git diff`** | **Чем отличаются** две версии кода? | `git diff main feature -- app.js` |

#### Особое внимание: `git reflog`

`git reflog` (reference log) — это **локальный журнал всех перемещений указателя HEAD** в вашем репозитории. В отличие от `git log`, который показывает историю коммитов, `reflog` показывает ваши **действия**: переключения веток, сбросы (`reset`), перебазирования (`rebase`), слияния (`merge`), даже если коммиты были «потеряны» из-за переписывания истории.

**Зачем это нужно?**
Представьте, что вы сделали `git reset --hard HEAD~3`, а потом поняли, что стёрли важные правки. Обычный `git log` их уже не покажет. Но `git reflog` помнит хеш того состояния, на которое указывал HEAD до сброса. Вы можете восстановить утерянное командой:
```bash
git reset --hard HEAD@{2}   # где 2 — номер записи в reflog
```

`reflog` — это ваша страховка от фатальных ошибок при локальной работе. Записи в нём хранятся по умолчанию 90 дней.

### 2. Анализ эволюции кода и поиск регрессий

История Git — это не просто архив, это источник метрик и понимания развития проекта.

- **Оценка частоты изменений:** `git log --pretty=format:"%ad" --date=short --since="1 year ago" | sort | uniq -c` покажет, в какие дни было больше всего коммитов. Это помогает понять ритм команды.
- **Выявление горячих точек (hotspots):** Файлы, которые часто меняются, — потенциальные источники проблем. `git log --format=oneline --name-only | grep "\\.js$" | sort | uniq -c | sort -rn` выведет список JS-файлов, отсортированный по частоте изменений. На них стоит обратить внимание при рефакторинге.
- **Поиск регрессий:** Именно здесь `git bisect` незаменим. Если производительность упала или тест начал падать, бинарный поиск быстро локализует виновный коммит.

### 3. Методика расследования: «Как найти, когда и почему произошла ошибка»

Вот пошаговый алгоритм, который я использую в работе и рекомендую всей команде.

**Шаг 1. Воспроизведите ошибку.**
Убедитесь, что вы можете стабильно повторять ошибочное поведение. В идеале — напишите скрипт или тест, который падает с понятным сообщением. Это критически важно для автоматизации `bisect`.

**Шаг 2. Найдите коммит, в котором ошибка появилась впервые.**
Используйте `git bisect`. Определите «хороший» коммит (последний, где всё работало) и «плохой» (текущий). Запустите ручной или автоматический поиск.

**Шаг 3. Изучите виновный коммит с помощью `git show`.**
Когда `bisect` укажет на коммит, выполните `git show <hash>`. Посмотрите:
- **Кто автор?** Возможно, стоит сразу обратиться к нему за контекстом.
- **Какие файлы затронуты?** Ограничивает ли это область поиска.
- **Что именно изменилось в коде?** Прочитайте diff, попытайтесь понять логику изменений.

**Шаг 4. Уточните детали с помощью `git blame`.**
Если коммит большой и изменил много строк, а вас интересует конкретная функция, используйте `git blame <file>` для версии файла **после** виновного коммита. Это покажет, какие строки были изменены именно этим коммитом, а какие пришли из предыдущих.

**Шаг 5. Поймите контекст с помощью `git log -p`.**
Посмотрите историю изменений конкретного файла или функции (`git log -L`), чтобы понять, как эта часть кода эволюционировала до и после появления ошибки. Возможно, причина не в одной строке, а в накопившихся изменениях.

### 4. Практическое задание: Путешествие в прошлое Git

Git — это проект с открытым исходным кодом, который управляет собственной историей с помощью Git. Давайте используем наши навыки, чтобы найти самый первый коммит в репозитории Git.

1.  **Клонируйте репозиторий Git (если ещё нет):**
    ```bash
    git clone https://github.com/git/git.git
    cd git
    ```

2.  **Найдите самый старый коммит.**
    Подсказка: `git log` показывает историю от текущего момента к прошлому. Чтобы увидеть **самый первый** коммит, нужно дойти до корня. Используйте опцию `--reverse`, чтобы перевернуть порядок вывода:
    ```bash
    git log --reverse --oneline | head -1
    ```
    Или используйте комбинацию `git rev-list`:
    ```bash
    git rev-list --max-parents=0 HEAD
    ```
    (У самого первого коммита нет родителей).

3.  **Посмотрите информацию об этом коммите:**
    ```bash
    git show <хеш_первого_коммита>
    ```

4.  **Изучите, какие файлы были в том первом коммите.**
    ```bash
    git ls-tree --name-only <хеш_первого_коммита>
    ```
    Вы увидите истоки того инструмента, которым пользуетесь сейчас. Это впечатляющее упражнение для понимания масштаба.

### 5. Вопросы для самопроверки

**1. Чем `git reflog` принципиально отличается от `git log`? Для каких сценариев нужен reflog?**
> **Ответ:** `git log` показывает **историю коммитов** текущей ветки (объекты в репозитории). `git reflog` показывает **историю перемещений указателя HEAD** (ваши действия). Reflog нужен для восстановления коммитов, которые были потеряны из-за `reset --hard`, `rebase` или удаления ветки. Это локальный журнал, он не синхронизируется с сервером.

**2. Почему не стоит бояться переписывать **локальную** историю (например, делать `rebase -i` или `commit --amend`), но категорически нельзя переписывать **публичную**?**
> **Ответ:** Переписывание локальной истории безопасно, потому что она существует только у вас. Если вы ошибётесь, `git reflog` всегда позволит откатиться назад. Переписывание публичной истории (уже запушенной) ломает синхронизацию у всех, кто успел её скачать, так как меняются хеши коммитов. Это приводит к конфликтам и необходимости принудительных пушей (`--force`), что может уничтожить чужую работу.

**3. Вы нашли коммит, в котором перестал работать тест. Какие команды Git помогут понять, **почему** именно этот коммит вызвал ошибку?**
> **Ответ:**
> - `git show <хеш>` — посмотреть полный diff коммита.
> - `git blame <файл>` — посмотреть, какие строки в этом файле были изменены именно этим коммитом.
> - `git log -p <файл>` или `git log -L :<функция>:<файл>` — проследить эволюцию проблемного участка кода.

**4. Как быстро оценить, какие файлы в проекте меняются чаще всего (являются «горячими точками»)?**
> **Ответ:** Использовать комбинацию `git log --name-only` с сортировкой. Пример для всех файлов:
> ```bash
> git log --format= --name-only | sort | uniq -c | sort -nr
> ```
> Для файлов определённого типа:
> ```bash
> git log --format= --name-only | grep "\.py$" | sort | uniq -c | sort -nr
> ```

**5. В чём преимущество использования `git bisect run` перед ручным `git bisect`?**
> **Ответ:** `git bisect run` полностью **автоматизирует** процесс. Он выполняет проверку на каждом шаге без участия человека, что исключает ошибки ручного тестирования и значительно ускоряет поиск. Единственное требование — наличие скрипта, который надёжно определяет «хорошее» и «плохое» состояние (возвращает 0 или не-0).
