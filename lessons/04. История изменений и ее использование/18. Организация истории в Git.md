### 1. Как Git хранит историю: DAG (Направленный Ациклический Граф)

В основе системы хранения истории Git лежит структура данных, называемая **направленным ациклическим графом** (Directed Acyclic Graph, DAG).

- **Узлы (Nodes):** Каждый узел — это **коммит**. Коммит содержит метаданные (автор, дата, сообщение) и **указатель на снимок состояния файлов** (объект дерева).
- **Рёбра (Edges):** Направленные стрелки от коммита к его **родителю (родителям)**. Обычный коммит имеет одного родителя, merge-коммит — двух (или более).
- **Ациклический:** Стрелки всегда направлены в прошлое. Вы не можете создать коммит, который является собственным предком, — это обеспечивает целостность истории.

**Метафора: Генеалогическое древо проекта.**
Представьте, что каждый коммит — это запись о рождении новой версии кода. В записи указано, от какой предыдущей версии она произошла (родитель). Слияние веток — это брак, порождающий ребёнка (merge commit) с двумя родителями. История никогда не зацикливается: вы не можете быть своим собственным дедушкой.

### 2. Идентификация коммитов: SHA-1 хэш

Каждый коммит получает уникальный 40-символьный идентификатор, например `c3b5e2a9f1...`. Это **SHA-1 хэш**, вычисленный на основе содержимого коммита (автор, дата, сообщение, хэш дерева файлов, хэш(и) родителей).

**Важнейшие свойства хэша:**
- **Детерминированность:** Если вы вычислите хэш для тех же данных, вы получите тот же результат.
- **Лавинный эффект:** Малейшее изменение в коммите (даже пробел в сообщении) полностью меняет хэш.
- **Криптографическая стойкость:** Практически невозможно подделать коммит, сохранив тот же хэш.

Именно поэтому **коммит нельзя изменить без последствий**. Если вы перепишете историю (например, через `git commit --amend` или `rebase`), Git создаст **новый коммит с новым хэшем**, а старый коммит останется в репозитории (пока не будет собран сборщиком мусора). Любой, кто уже ссылался на старый хэш, будет дезориентирован. Отсюда золотое правило: **не переписывайте историю публичных веток**.

### 3. Как выглядит дерево коммитов

Вот пример визуализации истории с двумя ветками и слиянием:

```
*   8a3f2e1 (HEAD -> main) Merge branch 'feature'
|\
| * c5b8e3a (feature) Добавлен модуль оплаты
| * 7d4a9f2 Обновлены стили кнопки
|/
* a1b2c3d Второй коммит в main
* e83c516 Первый коммит
```

Каждая звёздочка — коммит. Линии показывают родительские связи. Обратите внимание, что коммиты в ветке `feature` ответвляются от `a1b2c3d`, а затем сливаются обратно в `main` через merge commit `8a3f2e1`.

### 4. Инструменты историка: `git log` с полезными опциями

Базовая команда `git log` выводит полную историю, но её вывод может быть ошеломляющим. Профессиональный историк Git использует фильтры и форматирование.

#### Компактный вывод: `--oneline`

```bash
git log --oneline
```
Выводит каждый коммит в одну строку: сокращённый хэш и сообщение. Идеально для быстрого обзора.

#### Визуализация графа: `--graph`

```bash
git log --graph --oneline --all
```
`--graph` рисует ASCII-граф, `--all` показывает все ветки, а не только текущую. Это ваш главный инструмент для понимания структуры ветвления.

#### Форматирование вывода: `--pretty=format`

Позволяет создать собственный шаблон вывода.

```bash
git log --pretty=format:"%h - %an, %ar : %s"
```

**Полезные плейсхолдеры:**
- `%h` — сокращённый хэш
- `%an` — имя автора
- `%ar` — относительная дата (например, "2 days ago")
- `%s` — тема (первая строка сообщения)
- `%d` — имена веток и HEAD

**Пример красивого формата:**
```bash
git log --pretty=format:"%C(yellow)%h%Creset %C(blue)%ad%Creset | %s %C(green)[%an]%Creset" --date=short
```

#### Фильтрация по дате и автору

```bash
# Коммиты за последнюю неделю
git log --since="1 week ago"

# Коммиты конкретного автора
git log --author="Иван"

# Коммиты в диапазоне дат
git log --since="2024-01-01" --until="2024-01-31"
```

### 5. Важность сообщений коммитов: Conventional Commits

История коммитов — это документация, которую будут читать ваши коллеги (и вы сами через полгода). Плохое сообщение вроде `"fix"` или `"WIP"` бесполезно. Хорошее сообщение объясняет **почему** было сделано изменение.

**Стандарт Conventional Commits** (https://www.conventionalcommits.org) предлагает структуру:

```
<тип>[необязательная область]: <описание>

[необязательное тело]

[необязательная подвал]
```

**Основные типы:**
- `feat`: новая функциональность
- `fix`: исправление ошибки
- `docs`: изменения в документации
- `style`: форматирование, отступы (без изменения логики)
- `refactor`: рефакторинг кода
- `test`: добавление или исправление тестов
- `chore`: рутинные задачи (обновление зависимостей, настройка сборки)

**Примеры:**
```
feat(auth): добавить вход через Google OAuth
fix(parser): исправить обработку отрицательных чисел
chore: обновить версию lodash до 4.17.21
```

**Почему это важно?**
- Автоматическая генерация CHANGELOG.
- Автоматическое определение следующей версии (семантическое версионирование).
- Упрощение навигации по истории и поиска причин изменений.

### 6. Что означает HEAD~3?

`HEAD` — указатель на текущий коммит. Символ тильда `~` с числом позволяет перемещаться назад по истории **по первой родительской линии**.

- `HEAD~1` или просто `HEAD~` — первый родитель (предыдущий коммит).
- `HEAD~3` — прародитель на три поколения назад (дедушка вашего дедушки).

Если коммит имеет нескольких родителей (merge commit), то `HEAD^1` указывает на первого родителя, `HEAD^2` — на второго.

### Практическое задание: Станьте летописцем

1.  Создайте новый репозиторий:
    ```bash
    mkdir history-lab && cd history-lab
    git init
    ```

2.  Создайте 5 коммитов с разными типами изменений по Conventional Commits:
    ```bash
    echo "// Initial code" > app.js
    git add app.js && git commit -m "feat: create initial app structure"

    echo "// Helper functions" >> app.js
    git add app.js && git commit -m "feat(utils): add helper functions"

    echo "// Fix typo" >> app.js
    git add app.js && git commit -m "fix: correct variable name typo"

    echo "// Format code" >> app.js
    git add app.js && git commit -m "style: apply eslint formatting"

    echo "// Update README" > README.md
    git add README.md && git commit -m "docs: add project description"
    ```

3.  Создайте ветку `feature` от второго коммита и сделайте в ней ещё коммит:
    ```bash
    git checkout -b feature HEAD~2
    echo "// Feature work" >> feature.js
    git add feature.js && git commit -m "feat(feature): implement new feature"
    ```

4.  Вернитесь в `main` и выполните слияние:
    ```bash
    git checkout main
    git merge feature -m "Merge branch 'feature'"
    ```

5.  Изучите историю с разными опциями:
    ```bash
    git log --oneline --graph --all
    git log --pretty=format:"%h %ad | %s [%an]" --date=short
    git log --author="Ваше Имя"
    git log --since="1 hour ago" --oneline
    ```

### Вопросы для самопроверки

**1. Почему коммит нельзя изменить без последствий? К чему приведёт попытка переписать уже опубликованный коммит?**
> **Ответ:** Коммит идентифицируется SHA-1 хэшем, вычисленным из его содержимого. Любое изменение создаст **новый коммит с другим хэшем**. Если переписать опубликованный коммит, у коллег, которые уже скачали старую версию, возникнет рассинхронизация: их история будет ссылаться на коммиты, которых больше нет в общем репозитории. Это приводит к необходимости принудительной синхронизации (`--force`), что чревато потерей чужой работы.

**2. Что показывает обозначение `HEAD~3`? Как получить хэш этого коммита?**
> **Ответ:** `HEAD~3` означает «третий прародитель текущего коммита по первой родительской линии», то есть коммит, сделанный за три шага до текущего. Получить его хэш можно командой `git rev-parse HEAD~3`.

**3. Вы хотите найти все коммиты, сделанные коллегой Анной за последний месяц, которые содержат слово "оплата". Как это сделать?**
> **Ответ:** Можно комбинировать опции `git log`:
> ```bash
> git log --author="Анна" --since="1 month ago" --grep="оплата"
> ```
> `--grep` фильтрует по тексту в сообщении коммита.

**4. Зачем нужен стандарт Conventional Commits? Какие практические преимущества он даёт команде?**
> **Ответ:** Структурированные сообщения коммитов позволяют:
> - Автоматически генерировать журнал изменений (CHANGELOG).
> - Автоматически вычислять следующую версию при релизе (семантическое версионирование).
> - Быстро понять контекст изменений, просматривая историю.
> - Упростить код-ревью и поиск виновника бага (`git blame`).

**5. Что такое DAG в контексте Git и почему важно, что граф ациклический?**
> **Ответ:** DAG (направленный ациклический граф) — модель хранения истории, где коммиты — узлы, а ссылки на родителей — направленные рёбра. Ацикличность гарантирует, что история всегда движется вперёд во времени и нет замкнутых кругов (коммит не может быть собственным предком). Это обеспечивает целостность и предсказуемость системы контроля версий.
