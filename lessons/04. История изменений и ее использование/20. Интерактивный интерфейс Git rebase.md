Приветствую, коллеги. Сегодня мы погрузимся в одну из самых мощных и одновременно опасных возможностей Git — **интерактивный rebase** (`git rebase -i`). Это инструмент скульптора: он позволяет придать необработанной массе коммитов форму чистого, логичного и читаемого набора изменений. Но, как и любой острый инструмент, он требует уважения и понимания границ его применения.

### Зачем переписывать историю?

В процессе локальной разработки мы часто делаем коммиты, которые являются «строительным мусором»: `WIP`, `fix typo`, `ещё поправил`, `вроде работает`. Такие коммиты загромождают историю и мешают понять суть изменений при код-ревью или последующем расследовании.

**Цели интерактивного rebase:**
- **Очистка истории перед публикацией:** Собрать несколько черновых коммитов в один логически завершённый.
- **Улучшение сообщений коммитов:** Заменить `"fix"` на осмысленное `"fix(parser): обработка отрицательных чисел"`.
- **Переупорядочивание:** Изменить последовательность коммитов, чтобы история рассказывала связную историю.
- **Удаление лишнего:** Выкинуть коммит с отладочным кодом, который случайно попал в ветку.
- **Разделение больших коммитов:** Разбить один слишком крупный коммит на несколько мелких по разным аспектам.

**⚠️ Золотое правило: Никогда не переписывайте историю публичных веток!**
Если ветка уже была отправлена на сервер (`git push`), и её могли склонировать или слить другие разработчики, **не делайте rebase**. Переписывание опубликованной истории приводит к рассинхронизации и создаёт огромные проблемы коллегам. Интерактивный rebase — это инструмент для наведения порядка **строго до** `git push`.

### Синтаксис и основные команды

Запуск интерактивного режима на последние `N` коммитов текущей ветки:

```bash
git rebase -i HEAD~N
```

После выполнения команды откроется текстовый редактор со списком коммитов и инструкцией. Каждая строка начинается с команды, за которой следует хеш и сообщение коммита. **Коммиты перечислены от самого старого (вверху) к самому новому (внизу)**.

**Основные команды:**

| Команда | Сокращение | Описание |
| :--- | :--- | :--- |
| **`pick`** | `p` | Использовать коммит как есть (оставить без изменений). |
| **`reword`** | `r` | Использовать коммит, но **изменить его сообщение**. |
| **`edit`** | `e` | Использовать коммит, но **остановиться для внесения правок** (amend). Позволяет изменить содержимое коммита или разбить его. |
| **`squash`** | `s` | Объединить этот коммит с **предыдущим**. Сообщения обоих коммитов будут предложены для редактирования. |
| **`fixup`** | `f` | Объединить этот коммит с **предыдущим**, но **отбросить его сообщение**. Идеально для черновых правок. |
| **`drop`** | `d` | **Удалить коммит** (как будто его никогда не было). |

### Пример: Превращаем хаос в порядок

Представьте, что у вас в ветке `feature` есть такая история:

```
c3b5e2a (HEAD) test: проверил на стейдже
a1f8d3b wip: ещё правки
b7e9c4f fix typo в названии переменной
8d2a1e3 feat: добавил основную логику фильтрации
```

Перед созданием Pull Request вы хотите, чтобы это выглядело как один аккуратный коммит:

```
Новый хеш feat(filters): реализовать фильтрацию по дате и статусу
```

**Пошагово:**

1.  Запускаем rebase на 4 последних коммита:
    ```bash
    git rebase -i HEAD~4
    ```

2.  В открывшемся редакторе видим:
    ```
    pick 8d2a1e3 feat: добавил основную логику фильтрации
    pick b7e9c4f fix typo в названии переменной
    pick a1f8d3b wip: ещё правки
    pick c3b5e2a test: проверил на стейдже
    ```

3.  Редактируем файл: первый коммит оставляем `pick`, а остальные три превращаем в `fixup` (или `squash`, если хотим объединить сообщения):
    ```
    pick 8d2a1e3 feat: добавил основную логику фильтрации
    fixup b7e9c4f fix typo в названии переменной
    fixup a1f8d3b wip: ещё правки
    fixup c3b5e2a test: проверил на стейдже
    ```

4.  Сохраняем и закрываем файл. Git применяет изменения. Если мы использовали `fixup`, сообщения черновых коммитов исчезнут. Затем Git откроет редактор для сообщения итогового коммита (так как первый был `pick`). Мы можем заменить его на красивое:
    ```
    feat(filters): реализовать фильтрацию по дате и статусу

    Добавлены компоненты DateRangePicker и StatusSelect.
    Интегрировано с API получения списка транзакций.
    ```

5.  Сохраняем и закрываем. История в ветке теперь содержит только один коммит. Хеш изменится (как и у всех последующих коммитов), потому что родители изменились.

### Конфликты при rebase

При применении каждого шага rebase Git пытается применить патч коммита. Если возникают конфликты, процесс **приостанавливается**.

**Что делать:**
1.  Git сообщит: `error: could not apply ...` и покажет конфликтующие файлы.
2.  Выполните `git status`, чтобы увидеть список файлов с конфликтами.
3.  Разрешите конфликты вручную (удалите маркеры `<<<<<<<`, `=======`, `>>>>>>>`).
4.  Добавьте разрешённые файлы: `git add <file>`.
5.  Продолжите rebase: `git rebase --continue`.

Если вы запутались в конфликтах или поняли, что rebase пошёл не по плану, можно **прервать операцию** и вернуться в исходное состояние:

```bash
git rebase --abort
```

Эта команда откатывает все изменения, сделанные с момента запуска `git rebase -i`, и восстанавливает ветку в точности до начала процесса.

### Как отменить уже завершённый интерактивный rebase?

Если rebase завершился, и вы передумали, можно откатить ветку к состоянию до rebase с помощью **`git reflog`**.

```bash
git reflog
```

Найдите в выводе строку, соответствующую состоянию ветки **до** запуска `rebase`. Обычно она имеет вид `HEAD@{n}: rebase (start): checkout HEAD~4`. Затем выполните сброс:

```bash
git reset --hard HEAD@{n}
```

(Замените `n` на нужный индекс). Это вернёт указатель ветки и рабочий каталог к тому состоянию.

### Практическое задание: Осваиваем `squash`

1.  **Создайте новую ветку и сделайте 4 коммита:**
    ```bash
    mkdir rebase-lab && cd rebase-lab
    git init

    echo "// Main logic" > app.js
    git add . && git commit -m "feat: add main logic"

    echo "// Typo fix" >> app.js
    git add . && git commit -m "fix typo"

    echo "// WIP changes" >> app.js
    git add . && git commit -m "wip"

    echo "// Final test" >> app.js
    git add . && git commit -m "test: check integration"
    ```

2.  **Просмотрите текущую историю:**
    ```bash
    git log --oneline
    # Убедитесь, что видите 4 коммита
    ```

3.  **Запустите интерактивный rebase для последних 3 коммитов:**
    ```bash
    git rebase -i HEAD~3
    ```

4.  **В редакторе измените команды:**
    - Первый из трёх (`fix typo`) оставьте `pick`.
    - Второй (`wip`) измените на `squash` (или `s`).
    - Третий (`test`) измените на `squash`.

5.  **Сохраните и закройте.** Git откроет редактор для сообщения объединённого коммита. Напишите что-то осмысленное, например: `feat: finalize core implementation`.

6.  **Проверьте результат:**
    ```bash
    git log --oneline
    # Должно остаться 2 коммита: начальный "add main logic" и объединённый.
    ```

### Вопросы для самопроверки

**1. Что произойдёт с конфликтами при выполнении интерактивного rebase? Как продолжить или отменить процесс?**
> **Ответ:** Если при применении одного из коммитов возникает конфликт, rebase **приостанавливается**. Git выводит сообщение о конфликте и ждёт действий пользователя. Необходимо:
> - Разрешить конфликты в файлах вручную.
> - Выполнить `git add <разрешённые_файлы>`.
> - Продолжить: `git rebase --continue`.
> - Чтобы отменить rebase на любом этапе и вернуть всё как было, выполнить `git rebase --abort`.

**2. Как отменить уже завершённый интерактивный rebase?**
> **Ответ:** С помощью `git reflog`. Найти запись, предшествующую rebase, и выполнить `git reset --hard HEAD@{номер}`.

**3. В чём разница между `squash` и `fixup` при интерактивном rebase?**
> **Ответ:** Обе команды объединяют коммит с предыдущим. Отличие в обработке сообщения коммита:
> - `squash` объединяет сообщения обоих коммитов и предлагает отредактировать результат.
> - `fixup` просто присоединяет изменения, а сообщение текущего коммита **отбрасывается**, сохраняется только сообщение предыдущего коммита.

**4. Можно ли использовать интерактивный rebase для изменения порядка коммитов?**
> **Ответ:** Да. Для этого достаточно в открывшемся списке просто **поменять строки местами**. Коммиты будут применены в новом порядке. Важно помнить, что это может привести к конфликтам, если изменения в коммитах зависят друг от друга.

**5. Почему категорически не рекомендуется делать интерактивный rebase для коммитов, уже отправленных в общий репозиторий?**
> **Ответ:** Rebase переписывает историю, создавая новые коммиты с новыми хешами. Если коллеги уже скачали старые коммиты, их локальная история будет конфликтовать с переписанной. Для синхронизации им придётся принудительно перезаписывать свои ветки (`--force`), что может привести к потере их собственной работы. Это нарушает основной принцип совместной работы в распределённых системах контроля версий.
