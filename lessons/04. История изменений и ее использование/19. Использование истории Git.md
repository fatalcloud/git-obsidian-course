Добро пожаловать в детективное агентство Git! Сегодня я, ваш ведущий следователь, научу вас раскрывать тайны кодовой базы. В разработке постоянно возникают вопросы: «Кто и зачем написал эту строку?», «Когда появился этот баг?», «Почему эта функция работает именно так?». Git предоставляет целый арсенал инструментов для расследования. Мы пройдём по следам коммитов, найдём авторов злополучных строк и восстановим хронологию событий.

### 1. Осмотр места происшествия: `git show <commit>`

Самая простая, но очень важная команда детектива. Она показывает **полную информацию о конкретном коммите**: автора, дату, полное сообщение и diff изменений.

```bash
# Показать последний коммит
git show

# Показать конкретный коммит по хешу
git show a1b2c3d

# Показать изменения в конкретном файле внутри коммита
git show a1b2c3d -- src/app.js
```

**Пример вывода:**
```
commit a1b2c3d4e5f6...
Author: Анна Иванова <anna@example.com>
Date:   Tue Apr 15 10:30:45 2025 +0300

    fix: исправить расчёт налога

diff --git a/src/utils.js b/src/utils.js
...
```
Это как протокол осмотра: что, где, когда и кем было сделано.

### 2. Сравнение версий: `git diff <commit>^!`

Иногда нужно быстро посмотреть, что именно изменил конкретный коммит, по сравнению с его родителем. Суффикс `^!` означает «этот коммит и его родители», что для одиночного коммита эквивалентно сравнению с первым родителем.

```bash
# Показать изменения, внесённые коммитом abc123
git diff abc123^!
```
Это короче, чем `git diff abc123^ abc123`.

### 3. Поиск виновного: `git blame`

Команда `git blame` (или более дружелюбный псевдоним `git annotate`) показывает **для каждой строки файла** информацию о том, кто, когда и в каком коммите её последний раз изменял.

```bash
git blame src/app.js
```

**Пример вывода:**
```
a1b2c3d4 (Анна Иванова 2025-04-10 14:20:01 +0300  1) function calculateTax(price) {
a1b2c3d4 (Анна Иванова 2025-04-10 14:20:01 +0300  2)   const TAX_RATE = 0.2;
c5b8e3a2 (Борис Смирнов 2025-04-12 09:15:33 +0300  3)   return price * TAX_RATE;
a1b2c3d4 (Анна Иванова 2025-04-10 14:20:01 +0300  4) }
```
Это главный инструмент, когда нужно понять, к кому идти с вопросом о конкретной строке.

**Полезные опции:**
- `git blame -L 10,20 file.js` — показать blame только для строк с 10 по 20.
- `git blame -w` — игнорировать изменения пробельных символов (часто реальный автор логики прячется за коммитом с форматированием).

### 4. Поиск по содержимому в истории: `git grep`

Обычный `grep` ищет только в рабочем каталоге. `git grep` умеет искать **по всему репозиторию в любом коммите**.

```bash
# Найти все вхождения "TAX_RATE" в текущем состоянии
git grep "TAX_RATE"

# Найти в конкретном коммите
git grep "TAX_RATE" a1b2c3d

# Показать, в каких коммитах добавлялось или удалялось слово
git grep -p "TAX_RATE"  # Показывает функцию/метод, содержащий совпадение
```

### 5. Поиск коммита, который добавил/удалил строку: `git log -S`

Это, пожалуй, самый мощный инструмент детектива. Он находит коммиты, в которых **изменилось количество вхождений** заданной строки (появилась или исчезла).

```bash
# Найти коммит, где впервые появилась строка "TAX_RATE"
git log -S "TAX_RATE" --oneline

# Найти коммит, где строка исчезла
git log -S "TAX_RATE" --diff-filter=D --oneline

# В сочетании с -p показать и сами изменения
git log -S "TAX_RATE" -p
```

**Альтернатива: `git log -G`**
`-G` ищет коммиты, в диффе которых **встречается регулярное выражение**, даже если общее число вхождений не изменилось (например, строка была перемещена).

```bash
git log -G "TAX_RATE" --oneline
```

### 6. История одного файла: `git log --follow <file>`

Файлы переименовываются и перемещаются. Обычный `git log file` покажет историю только до момента переименования. Флаг `--follow` заставляет Git отслеживать перемещения файла по всей истории.

```bash
# Проследить за файлом, даже если его переименовали
git log --follow --oneline src/utils.js
```

**Полезное дополнение:** Чтобы увидеть, как менялся файл, можно использовать `-p`:
```bash
git log --follow -p src/utils.js
```

### 7. История конкретных строк: `git log -L`

Это инструмент хирургической точности. Он показывает **эволюцию определённого диапазона строк в файле** на протяжении истории коммитов.

```bash
# Показать историю строк с 15 по 20 в файле file.py
git log -L 15,20:file.py

# Показать историю функции calculateTax (поиск по регулярному выражению)
git log -L :calculateTax:file.js
```

**Что делает `git log -L`?** Он выводит список коммитов, в которых изменялся указанный диапазон строк, с отображением самих изменений. Это невероятно полезно, когда нужно понять, как конкретная функция эволюционировала с течением времени.

### Практическое задание: Раскрываем дело

1.  **Создайте учебный репозиторий:**
    ```bash
    mkdir detective-lab && cd detective-lab
    git init
    ```

2.  **Создайте файл `mystery.js` с несколькими коммитами от разных «персонажей»:**
    ```bash
    # Коммит 1 (Анна)
    git config user.name "Анна"
    echo "function secretFormula() {" > mystery.js
    echo "  return 42;" >> mystery.js
    echo "}" >> mystery.js
    git add mystery.js && git commit -m "feat: initial formula"

    # Коммит 2 (Борис)
    git config user.name "Борис"
    echo "function anotherFunction() {" >> mystery.js
    echo "  return 'hello';" >> mystery.js
    echo "}" >> mystery.js
    git add mystery.js && git commit -m "feat: add another function"

    # Коммит 3 (Вера)
    git config user.name "Вера"
    sed -i 's/42/4200/' mystery.js  # или вручную поменять 42 на 4200
    git add mystery.js && git commit -m "fix: update secret number"

    # Коммит 4 (Анна)
    git config user.name "Анна"
    echo "console.log(secretFormula());" >> mystery.js
    git add mystery.js && git commit -m "feat: log the secret"
    ```

3.  **Расследование:**
    - Используйте `git blame mystery.js`, чтобы узнать, кто изменил строку с `return 4200;`.
    - Используйте `git log -S "42" --oneline`, чтобы найти коммит, где число 42 было впервые добавлено.
    - Используйте `git log -L :secretFormula:mystery.js`, чтобы просмотреть историю функции `secretFormula`.
    - Используйте `git show` на коммит Веры, чтобы увидеть, что именно она изменила.

### Вопросы для самопроверки

**1. Как показать изменения между двумя коммитами `abc123` и `def456`?**
> **Ответ:** `git diff abc123 def456`. Это покажет разницу между состояниями этих двух коммитов. Можно также указать конкретный файл: `git diff abc123 def456 -- file.js`.

**2. Что делает команда `git log -L 15,20:file.py`?**
> **Ответ:** Она показывает историю изменений **только для строк с 15 по 20** в файле `file.py`. Вывод включает список коммитов, в которых затрагивался этот диапазон, и сами изменения (diff) для этих строк. Это позволяет отследить эволюцию конкретного участка кода.

**3. Вы нашли строку с ошибкой в `utils.js` и хотите узнать, кто её написал и в каком коммите. Какие команды дадут эту информацию?**
> **Ответ:**
> - `git blame utils.js` — покажет автора и коммит для каждой строки.
> - `git log -S "строка_с_ошибкой" utils.js` — найдёт коммит, где эта строка была добавлена или удалена.

**4. Чем `git log -S` отличается от `git log -G`?**
> **Ответ:** `-S` ищет коммиты, в которых **изменилось количество вхождений** строки (появилась или исчезла). `-G` ищет коммиты, в диффе которых **встречается** заданное регулярное выражение. `-G` может найти перемещение строки, даже если число вхождений не изменилось.

**5. Как посмотреть полную информацию о коммите (автор, дата, сообщение, diff) по его сокращённому хешу?**
> **Ответ:** `git show <хеш>`. Например, `git show a1b2c3d`.
