Приветствую, коллеги-детективы. Сегодня мы освоим инструмент, который экономит часы (а иногда и дни) отладки — **`git bisect`**. Представьте: вы только что обнаружили, что важный тест перестал проходить. Последний раз вы проверяли его неделю назад, и с тех пор в ветку попало 200 коммитов. Какой именно всё сломал? Просматривать каждый вручную — путь в безумие. К счастью, Git умеет делать бинарный поиск по истории, и именно этим мы займёмся.

### Алгоритм бинарного поиска: как найти иголку в стоге сена

`git bisect` работает по принципу **бинарного поиска**. Он автоматически делит диапазон подозрительных коммитов пополам, переключает вас на коммит из середины и просит проверить, есть ли там ошибка. Вы сообщаете Git результат («хороший» или «плохой»), и алгоритм сужает диапазон вдвое. С каждым шагом количество оставшихся подозреваемых сокращается экспоненциально.

**Временная сложность:** O(log N). Для 1000 коммитов потребуется не более **10 шагов** (2^10 = 1024), а не 1000.

### Пошаговое руководство: ручной режим

Давайте пройдём классический сценарий. У вас есть уверенность, что:
- В коммите `abc123` (или теге `v1.0`) всё работало **хорошо** (good).
- В текущем коммите `HEAD` (или `def456`) что-то **сломалось** (bad).

**Шаг 1. Запуск bisect**
```bash
git bisect start
```

**Шаг 2. Указываем «плохой» коммит**
Обычно это текущее состояние `HEAD`:
```bash
git bisect bad HEAD
# или просто git bisect bad (по умолчанию HEAD)
```

**Шаг 3. Указываем «хороший» коммит**
Тот последний коммит, где всё работало:
```bash
git bisect good abc123
```
Git вычисляет середину между `abc123` и `HEAD` и переключает рабочий каталог на этот коммит (состояние `detached HEAD`). Он выводит сообщение:
```
Bisecting: 99 revisions left to test after this (roughly 7 steps)
```

**Шаг 4. Тестируем текущий коммит**
Теперь ваша задача — определить, есть ли здесь ошибка. Это может быть запуск теста, компиляция проекта, проверка в браузере. В зависимости от результата вы сообщаете Git:

- Если ошибка **есть** (bad): `git bisect bad`
- Если ошибки **нет** (good): `git bisect good`
- Если текущий коммит невозможно протестировать (например, он не компилируется по другой причине): `git bisect skip`

После каждой команды Git снова делит оставшийся диапазон пополам, переключает вас на новый коммит и выводит оставшееся количество шагов.

**Шаг 5. Финал: виновный найден**
Когда диапазон сужается до одного коммита, Git выводит его хеш и сообщение:
```
a1b2c3d is the first bad commit
commit a1b2c3d...
Author: ...
Date: ...
    Сообщение коммита
```
Теперь вы точно знаете, кто и когда внёс ошибку. Можно приступать к исправлению.

**Шаг 6. Завершение bisect и возврат к нормальной работе**
```bash
git bisect reset
```
Эта команда возвращает `HEAD` на ту ветку, с которой вы начинали поиск.

### Автоматизация: `git bisect run`

Ручной режим требует вашего участия на каждом шаге. Но если у вас есть скрипт или команда, которая возвращает код 0 в случае успеха (good) и ненулевой код в случае ошибки (bad), вы можете полностью автоматизировать процесс.

**Синтаксис:**
```bash
git bisect start HEAD abc123
git bisect run <ваша_команда>
```

**Пример:** У вас есть тестовый скрипт `./run_test.sh`, который возвращает 0, если тест проходит, и 1, если падает. Команда будет выглядеть так:
```bash
git bisect start
git bisect bad HEAD
git bisect good v1.0
git bisect run ./run_test.sh
```
Git будет автоматически перебирать коммиты, выполнять скрипт и сужать диапазон. Когда виновный будет найден, процесс остановится.

**Важно:** Скрипт должен быть **детерминированным** и не зависеть от состояния вне репозитория.

### Реальный кейс: перестал работать тест

Представьте, что в проекте на Node.js после утреннего `git pull` упал тест `npm test`. Вы точно помните, что вчера вечером всё работало. Последний коммит перед вашим уходом был `a1b2c3d`.

1. Запускаем поиск:
   ```bash
   git bisect start
   git bisect bad HEAD
   git bisect good a1b2c3d
   ```
2. Git переключился на коммит посередине. Выполняем `npm test`. Если падает — `git bisect bad`. Если проходит — `git bisect good`.
3. Повторяем, пока не найдём виновного.
4. Если вдруг какой-то коммит не запускается (например, сломана сборка из-за отсутствия зависимости), делаем `git bisect skip`.
5. После нахождения выполняем `git bisect reset`.

В крупных проектах этот подход экономит огромное количество времени.

### Практическое задание: найти иголку вручную

Создадим учебный репозиторий с 10 коммитами, в одном из которых спрятана ошибка, и найдём её с помощью `git bisect`.

**1. Подготовка полигона:**
```bash
mkdir bisect-lab && cd bisect-lab
git init

# Создадим файл и сделаем 10 коммитов
echo "function add(a, b) { return a + b; }" > calc.js
git add calc.js && git commit -m "C1: initial add function"

# Добавим ещё функции, в C7 внесём "ошибку" (например, заменим + на -)
# Для простоты используем цикл
for i in {2..10}; do
  if [ $i -eq 7 ]; then
    echo "function sub(a, b) { return a - b; } // BUG: wrong logic" >> calc.js
  else
    echo "// C$i: some comment" >> calc.js
  fi
  git add calc.js && git commit -m "C$i: commit number $i"
done

git log --oneline  # убедимся, что есть 10 коммитов
```

**2. Запускаем bisect:**
```bash
git bisect start
git bisect bad HEAD      # последний коммит (C10) плохой
git bisect good HEAD~9   # первый коммит (C1) хороший
```

**3. Git переключит вас на коммит примерно в середине (C5 или C6).** Проверяем содержимое `calc.js` на наличие строки с `BUG`. Если её нет — коммит хороший:
```bash
grep -q "BUG" calc.js && echo "bad" || echo "good"
```
В зависимости от вывода выполняем `git bisect good` или `git bisect bad`.

**4. Повторяем проверку и команды `good`/`bad`, пока Git не укажет на коммит C7.**

**5. Завершаем:**
```bash
git bisect reset
```

### Вопросы для самопроверки

**1. Сколько шагов потребуется `git bisect` для нахождения ошибки среди 1000 коммитов? Почему?**
> **Ответ:** Максимум **10 шагов**. Бинарный поиск делит диапазон пополам на каждом шаге, и 2^10 = 1024 > 1000. В худшем случае после 10 проверок останется 1 коммит.

**2. Как завершить `git bisect` и вернуться на исходную ветку, если вы нашли виновного (или решили прервать поиск)?**
> **Ответ:** Выполнить команду `git bisect reset`. Она вернёт `HEAD` на ветку, с которой вы начинали, и завершит сеанс bisect.

**3. Что делать, если на одном из шагов bisect коммит невозможно протестировать (например, он не собирается)?**
> **Ответ:** Использовать команду `git bisect skip`. Git исключит этот коммит из рассмотрения и выберет другой из оставшегося диапазона. Важно: если пропущено слишком много коммитов, bisect может не найти точного виновного, но сузит круг.

**4. Можно ли автоматизировать `git bisect` с помощью собственного скрипта? Если да, то какое условие должен выполнять скрипт?**
> **Ответ:** Да, с помощью `git bisect run <скрипт>`. Скрипт должен возвращать код выхода **0**, если тест пройден (коммит хороший), и любое ненулевое значение (например, **1**), если тест упал (коммит плохой).

**5. Вы запустили `git bisect`, но ошиблись при указании плохого коммита. Можно ли начать заново?**
> **Ответ:** Да, проще всего выполнить `git bisect reset`, а затем снова `git bisect start` с правильными параметрами. Также можно внутри сессии переопределить `bad`/`good` для текущего коммита, но сброс надёжнее.
