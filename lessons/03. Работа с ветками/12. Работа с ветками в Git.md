Добрый день, коллеги! Сегодня мы открываем новую главу в изучении Git — **ветвление**. Если до этого мы говорили в основном о сохранении истории линейно, то теперь вы научитесь создавать параллельные реальности вашего проекта. Ветки — это то, что превращает Git из простого архива снимков в мощнейший инструмент для командной и экспериментальной разработки.

### Что такое ветка в Git?

На самом базовом уровне **ветка (branch)** — это **подвижный указатель на один из коммитов** в вашем репозитории. Это не копия папки, не дублирование файлов. Это просто файл внутри `.git/refs/heads/`, который содержит хеш (SHA-1) того коммита, на который ветка указывает.

**Аналогия: Закладки в книге.**
Представьте, что история коммитов — это книга. Вы читаете её и хотите запомнить место. Вы кладёте закладку на текущую страницу. Ветка `main` — это ваша основная закладка. Когда вы создаёте новую ветку `feature`, вы просто кладёте вторую закладку на ту же самую страницу. Вы можете читать дальше, перемещая одну закладку, в то время как другая остаётся на старом месте.

#### Ветка по умолчанию: `main` или `master`

Когда вы инициализируете новый репозиторий (`git init`), Git автоматически создаёт первую ветку. Исторически она называлась **`master`**, но в последние годы сообщество перешло на более нейтральное имя **`main`**. В современных версиях Git по умолчанию создаётся `main`. Это просто имя; с технической точки зрения она ничем не отличается от любой другой ветки.

#### Указатель HEAD: Где я нахожусь?

Если ветки — это закладки в книге, то **HEAD** — это ваш палец, который указывает на ту закладку, которую вы сейчас держите в руках. HEAD говорит Git: «Сейчас я читаю вот здесь, и новые страницы (коммиты) я буду добавлять после этой закладки».

Обычно HEAD указывает на ветку (например, `refs/heads/main`). Это называется **attached HEAD**. Реже HEAD может указывать напрямую на коммит — **detached HEAD**, но это тема для продвинутого урока.

Визуально это можно представить так:
```
HEAD -> main -> Коммит C3
```

### Основные команды для работы с ветками

#### Просмотр списка веток: `git branch`

```bash
git branch
```
Выводит список всех **локальных** веток. Текущая ветка, на которую указывает HEAD, будет выделена звёздочкой (`*`) и часто зелёным цветом.

**Полезные опции:**
- `git branch -v` — показать последний коммит на каждой ветке.
- `git branch -vv` — показать связь с удалёнными ветками (о них позже).
- `git branch --all` — показать все ветки, включая удалённые (с сервера).

#### Создание новой ветки: `git branch <имя_ветки>`

```bash
git branch feature-login
```
Эта команда создаёт новый указатель с именем `feature-login`, который указывает на **тот же самый коммит**, на котором вы сейчас находитесь (на который указывает HEAD).

**Важно:** Создание ветки **не переключает** вас на неё. Вы остаётесь на той же ветке, где были. Чтобы и создать, и сразу переключиться, используйте следующую команду.

#### Переключение между ветками

В Git есть две основные команды для переключения. Рекомендую использовать современный вариант, но вы должны знать оба.

**Современный способ (Git 2.23+): `git switch`**
```bash
# Переключиться на существующую ветку
git switch feature-login

# Создать новую ветку и сразу переключиться на неё
git switch -c feature-signup
```

**Классический способ: `git checkout`**
```bash
# Переключиться на существующую ветку
git checkout feature-login

# Создать новую ветку и сразу переключиться на неё
git checkout -b feature-signup
```

Обе команды делают одно и то же. `git switch` был введён, чтобы разделить функциональность `checkout` (которая умеет и ветки переключать, и файлы восстанавливать) и сделать поведение более предсказуемым для новичков.

### Что происходит с незакоммиченными изменениями при переключении?

Это критически важный момент для понимания.

**Правило:** Git **не позволит** вам переключить ветку, если при этом будут перезаписаны незакоммиченные изменения в рабочем каталоге или индексе.

Возможны три сценария:

1.  **Рабочий каталог чист.** Вы можете свободно переключаться между любыми ветками. Содержимое папки мгновенно изменится, отражая состояние последнего коммита целевой ветки.
2.  **Есть изменения, но они не конфликтуют с целевой веткой.** Например, вы создали новый неотслеживаемый файл `test.txt` или изменили файл, которого вообще нет в целевой ветке. Git разрешит переключение и **перенесёт ваши изменения** в новую ветку. Они так и останутся незакоммиченными.
3.  **Есть изменения, которые конфликтуют с целевой веткой.** Например, вы изменили `app.js` в ветке `feature`, и в целевой ветке `main` этот же файл в том же месте имеет другое содержимое. Git откажется переключаться и выдаст ошибку:
    ```
    error: Your local changes to the following files would be overwritten by checkout:
        app.js
    Please commit your changes or stash them before you switch branches.
    ```

**Что делать в случае конфликта?** Git сам подсказывает: либо закоммитить изменения (если они готовы), либо спрятать их в `stash` (как мы учили в прошлом уроке), затем переключиться, а потом применить stash.

### Зачем нужны ветки? Сценарии использования

Ветки решают три главные задачи в разработке:

1.  **Изоляция функциональности (Feature Branch Workflow).** Вы начинаете работу над новой фичей («Добавить корзину») в отдельной ветке `feature/cart`. Вы можете делать там десятки коммитов, ломать и чинить код, и это никак не повлияет на основную ветку `main`, откуда идёт продакшн. Только когда фича готова и протестирована, вы сливаете (merge) ветку обратно в `main`.

2.  **Эксперименты.** Хотите попробовать новую библиотеку или переписать модуль, но не уверены, что это сработает? Создайте ветку `experiment/new-parser`. Если эксперимент удался — отлично, вливайте в `main`. Если нет — просто удалите ветку, и `main` останется в неприкосновенности.

3.  **Срочные исправления (Hotfix).** Как в примере с `stash`: вы работаете над долгой фичей в `feature/big-update`, а в `main` нашли критический баг. Вы переключаетесь на `main`, создаёте ветку `hotfix/urgent-bug`, чините баг, вливаете обратно в `main`, а затем возвращаетесь к своей фиче. Ваша работа над фичей не была заблокирована.

### Визуализация истории: `git log --graph --oneline --all`

Чтобы увидеть ветвление своими глазами, используйте эту магическую комбинацию флагов:

```bash
git log --graph --oneline --all
```

- `--graph` — рисует ASCII-граф из звёздочек и линий, показывая ветвление и слияния.
- `--oneline` — выводит каждый коммит в одну строку (хеш + сообщение).
- `--all` — показывает историю **всех** веток, а не только текущей.

**Пример вывода:**
```
* 8a3f2e1 (HEAD -> feature-2) Добавлен функционал в ветке 2
| * c5b8e3a (feature-1) Изменения в ветке 1
|/
* a1b2c3d (main) Второй коммит в main
* e83c516 Первый коммит
```
По этому графу видно, что ветки `feature-1` и `feature-2` разошлись от коммита `a1b2c3d`.

### Практическое задание: Создаём свой первый граф

Давайте пошагово создадим структуру с несколькими ветками, чтобы увидеть её своими глазами.

1.  **Создайте папку и инициализируйте репозиторий:**
    ```bash
    mkdir branching-demo
    cd branching-demo
    git init
    ```

2.  **Создайте три коммита в `main`:**
    ```bash
    echo "v1" > file.txt && git add file.txt && git commit -m "C1: first commit"
    echo "v2" >> file.txt && git add file.txt && git commit -m "C2: second commit"
    echo "v3" >> file.txt && git add file.txt && git commit -m "C3: third commit in main"
    ```

3.  **Просмотрите линейную историю:**
    ```bash
    git log --oneline
    ```

4.  **Создайте ветку `feature-1` от текущего состояния и переключитесь на неё:**
    ```bash
    git switch -c feature-1
    # или git checkout -b feature-1
    ```

5.  **Сделайте один коммит в `feature-1`:**
    ```bash
    echo "feature-1 work" > feature1.txt
    git add feature1.txt
    git commit -m "C4: added feature-1 work"
    ```

6.  **Вернитесь на `main` и создайте ветку `feature-2`:**
    ```bash
    git switch main
    git switch -c feature-2
    ```

7.  **Сделайте один коммит в `feature-2`:**
    ```bash
    echo "feature-2 work" > feature2.txt
    git add feature2.txt
    git commit -m "C5: added feature-2 work"
    ```

8.  **Переключитесь обратно на `main` и посмотрите на граф:**
    ```bash
    git switch main
    git log --graph --oneline --all
    ```

**Ожидаемый результат графа:**
```
* (feature-2) C5: added feature-2 work
| * (feature-1) C4: added feature-1 work
|/
* (HEAD -> main) C3: third commit in main
* C2: second commit
* C1: first commit
```
Обратите внимание: `main` остался на коммите C3. Ветки `feature-1` и `feature-2» ушли вперёд, но `main` их «не видит», пока мы не сделаем слияние.

### Вопросы для самопроверки

**1. Что произойдёт с незакоммиченными изменениями при переключении ветки, если они не конфликтуют с файлами в целевой ветке?**
> **Ответ:** Git разрешит переключение и **перенесёт** незакоммиченные изменения в целевую ветку. Они останутся в рабочем каталоге и/или индексе. Это удобно, если вы начали работу не в той ветке и хотите «перевезти» черновик в правильное место.

**2. Как посмотреть список всех локальных веток в репозитории?**
> **Ответ:** Команда `git branch` (без аргументов) покажет все локальные ветки, а звёздочкой отметит текущую. Для более подробной информации можно использовать `git branch -v`.

**3. Чем команда `git switch -c new-branch` отличается от последовательности `git branch new-branch` + `git switch new-branch`?**
> **Ответ:** Результат абсолютно одинаков. `git switch -c` — это просто удобное сокращение (shortcut), объединяющее создание и переключение в одну команду.

**4. (Вопрос на понимание) Вы находитесь на ветке `feature`, сделали коммит. Затем выполнили `git checkout main`. Куда «переедет» указатель `main`?**
> **Ответ:** Указатель `main` останется на том же коммите, на котором был до переключения. `git checkout` (или `switch`) меняет только HEAD (текущую ветку), но не двигает другие ветки. Чтобы сдвинуть `main` вперёд, нужно либо делать коммиты, находясь на `main`, либо выполнять слияние (`git merge`).

**5. Можно ли удалить ветку, на которой вы сейчас находитесь?**
> **Ответ:** Нет. Git выдаст ошибку: `Cannot delete branch 'feature' checked out at '...'`. Сначала нужно переключиться на другую ветку (обычно `main`).
