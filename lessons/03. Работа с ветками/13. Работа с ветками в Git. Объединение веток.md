Добрый день, коллеги. С вами техлид, и сегодня мы разберём одну из ключевых операций в командной разработке — **слияние веток (merge)**. Именно здесь одиночные усилия превращаются в общий результат. Если ветки позволяют изолировать работу, то слияние позволяет собрать разрозненные части в единое целое. Мы детально разберём два принципиально разных типа слияния, поймём, как они выглядят в истории, и научимся управлять этим процессом.

### Что такое `git merge`?

Команда `git merge <имя_ветки>` берёт содержимое указанной ветки-источника и **интегрирует его в вашу текущую ветку**. После выполнения команды текущая ветка будет содержать все изменения, которые были в источнике, плюс ваши собственные (если они были). Git автоматически определяет, каким способом выполнить слияние, исходя из того, насколько разошлись истории веток.

### Fast-Forward Merge (Перемотка вперёд)

Это самый простой и чистый вид слияния. Он возможен только в том случае, если **текущая ветка является прямым предком** сливаемой ветки. Другими словами, с момента создания ветки-фичи в основной ветке (`main`) **не было новых коммитов**.

**Как это выглядит на графе:**
```
      feature
      ↓
* -- * -- * (C3, C4)
     ↑
    main (C2)
```

Ветка `feature` ответвилась от `main` в точке C2, и в `main` с тех пор ничего не менялось. `main` просто отстал на два коммита.

**Выполняем слияние:**
```bash
git switch main
git merge feature
```

**Результат:** Git просто **перемещает указатель `main` вперёд** на тот же коммит, на который указывает `feature`. Никакого нового коммита не создаётся. История остаётся абсолютно линейной.

**После слияния:**
```
      main, feature
      ↓
* -- * -- * -- *
C1    C2    C3   C4
```

**Преимущества:** История чистая, без лишних сущностей.
**Недостаток:** Теряется информация о том, что коммиты C3 и C4 изначально были сделаны в отдельной ветке (хотя это часто и не нужно).

### Трёхстороннее слияние (3-Way Merge) и создание Merge Commit

Этот тип слияния происходит, когда с момента ответвления **обе ветки успели получить новые коммиты**. Ветки «разошлись» (diverged).

**Сценарий:**
1. Вы создали ветку `feature` от `main` (коммит C2).
2. В `main` добавили коммит C3 (исправили баг).
3. В `feature` добавили коммиты C4 и C5 (добавили новую функциональность).

**Граф до слияния:**
```
      feature
      ↓
      C4 -- C5
     /
C1 -- C2
     \
      C3
      ↑
     main
```

Здесь `main` не является прямым предком `feature`, и наоборот. Fast-forward невозможен.

**Выполняем слияние:**
```bash
git switch main
git merge feature
```

**Что делает Git под капотом?**
1.  Определяет **трёх общих предков** для слияния:
    - Коммит, на котором находится `main` (C3) — «наш» коммит.
    - Коммит, на котором находится `feature` (C5) — «их» коммит.
    - **Базовый коммит** (C2) — последний общий предок (merge base).
2.  Сравнивает изменения: (C2 → C3) и (C2 → C5).
3.  Объединяет эти два набора изменений. Если они не конфликтуют, Git создаёт новый, специальный коммит — **Merge Commit** (коммит слияния).

**Результат:** Появляется новый коммит (C6), у которого **два родителя**: C3 и C5.

**Граф после слияния:**
```
      feature
      ↓
      C4 -- C5
     /        \
C1 -- C2 -- C3 -- C6 (merge commit)
                 ↑
                main (HEAD)
```

Merge commit автоматически получает сообщение вида `Merge branch 'feature'`. Он фиксирует сам факт объединения историй. Такая история нелинейна, но она точно отражает реальный ход разработки: кто-то работал над фичей параллельно с правками в `main`, а потом они встретились.

### Флаг `--no-ff`: принудительное создание merge commit

Бывают ситуации, когда формально возможен fast-forward, но вы **хотите явно сохранить информацию о том, что работа велась в отдельной ветке**. Это особенно важно для больших проектов, где каждая фича оформляется отдельным Pull Request'ом, и по истории должно быть видно, где начинается и заканчивается фича.

Для этого используется флаг `--no-ff` (no fast-forward):

```bash
git merge --no-ff feature
```

Даже если `main` не ушёл вперёд, Git **всё равно создаст merge commit**. История станет «вилкой», показывающей точку интеграции.

### Практическое задание: Моделируем оба типа слияния

Сейчас мы руками создадим обе ситуации, чтобы навсегда запомнить разницу.

**Часть 1: Fast-Forward Merge**

1.  **Создайте репозиторий и первый коммит:**
    ```bash
    mkdir merge-ff-demo && cd merge-ff-demo
    git init
    echo "v1" > file.txt && git add file.txt && git commit -m "C1"
    ```

2.  **Создайте ветку `feature` и сделайте в ней коммит:**
    ```bash
    git switch -c feature
    echo "feature" >> file.txt && git add file.txt && git commit -m "C2: work in feature"
    ```

3.  **Вернитесь в `main` и выполните слияние:**
    ```bash
    git switch main
    git merge feature
    ```
    Обратите внимание на вывод: Git напишет `Fast-forward`.

4.  **Проверьте историю:**
    ```bash
    git log --graph --oneline --all
    ```
    Вы увидите линейную последовательность: C1 -> C2. Указатели `main` и `feature` на одном коммите.

**Часть 2: 3-Way Merge с созданием Merge Commit**

5.  **Верните `main` на один коммит назад, чтобы смоделировать расхождение:**
    ```bash
    git reset --hard HEAD~1   # main вернулся на C1
    ```
    *В реальном проекте так делать не стоит, но для учебной симуляции — отлично.*

6.  **Создайте новый коммит в `main` (имитация фикса бага):**
    ```bash
    echo "hotfix" >> file.txt && git add file.txt && git commit -m "C3: hotfix in main"
    ```

7.  **Переключитесь на `feature` и создайте там ещё один коммит (имитация продолжения работы):**
    ```bash
    git switch feature
    echo "more feature" >> file.txt && git add file.txt && git commit -m "C4: more work in feature"
    ```

8.  **Вернитесь в `main` и попробуйте слить `feature`:**
    ```bash
    git switch main
    git merge feature
    ```
    На этот раз откроется редактор для сообщения merge commit'а. Сохраните и закройте его.

9.  **Посмотрите на граф:**
    ```bash
    git log --graph --oneline --all
    ```
    Вы должны увидеть классическую «развилку» и коммит слияния с двумя родителями.

### Вопросы для самопроверки

**1. Когда возникает merge commit при слиянии веток?**
> **Ответ:** Merge commit создаётся в двух случаях:
> 1. Когда ветки **разошлись** (в обеих ветках есть новые коммиты после точки расхождения), и выполняется трёхстороннее слияние.
> 2. Когда разработчик принудительно использует флаг `--no-ff`, даже если возможен fast-forward.

**2. Можно ли запретить fast-forward слияние? Если да, то зачем это может понадобиться?**
> **Ответ:** Да, с помощью флага `--no-ff` (например, `git merge --no-ff feature`). Это заставляет Git создать merge commit даже при линейной истории. Это нужно для того, чтобы **сохранить контекст фичи** — видеть в истории, что определённая группа коммитов была сделана в рамках отдельной задачи и потом интегрирована. Такой подход облегчает навигацию и откат целых функциональных блоков.

**3. В чём ключевое отличие fast-forward merge от трёхстороннего с точки зрения истории коммитов?**
> **Ответ:** Fast-forward **не создаёт нового коммита**, а просто перемещает указатель ветки вперёд, сохраняя линейную историю. Трёхстороннее слияние **создаёт новый merge commit** с двумя родителями, делая историю нелинейной и явно отражая факт параллельной разработки.

**4. (Дополнительно) Вы работаете в ветке `feature`, которая ответвилась от `main`. В `main` тем временем добавили 5 коммитов. Вы хотите «подтянуть» эти изменения в свою `feature`, чтобы быть в курсе последних обновлений. Какое слияние произойдёт, если вы выполните `git merge main` находясь в `feature`?**
> **Ответ:** Произойдёт трёхстороннее слияние с созданием merge commit в ветке `feature`, потому что истории разошлись. Это стандартная практика — периодически подливать `main` в долгоживущие фичи, чтобы избежать гигантских конфликтов в будущем.
