Добрый день, коллеги. С вами снова техлид. В прошлом уроке мы освоили базовое ветвление и слияние. Сегодня мы углубимся в инструменты профессионального управления ветками. Когда в проекте десятки активных веток, умение быстро их переименовывать, безопасно удалять и анализировать состояние слияний экономит часы времени и предотвращает досадные ошибки. Мы рассмотрим команды, которые превращают хаос из множества указателей в стройную систему.

### 1. Удаление веток: `git branch -d` и `git branch -D`

После того как фича слита в `main` или эксперимент признан неудачным, ветку следует удалить, чтобы не засорять список. Git предлагает два режима удаления: безопасный и принудительный.

#### Безопасное удаление: `git branch -d <имя_ветки>`

```bash
git branch -d feature/old-login
```

Git удалит ветку **только в том случае, если она полностью слита с текущей веткой** (или с её upstream-веткой). Это защита от случайной потери несохранённых изменений. Если ветка содержит уникальные коммиты, которые ещё не были слиты, Git выдаст ошибку:

```
error: The branch 'feature/old-login' is not fully merged.
If you are sure you want to delete it, run 'git branch -D feature/old-login'.
```

#### Принудительное удаление: `git branch -D <имя_ветки>`

```bash
git branch -D feature/old-login
```

Флаг `-D` (заглавная буква) удаляет ветку **безусловно**, независимо от статуса слияния. Используйте его, когда вы точно уверены, что работа в этой ветке больше не нужна (например, эксперимент провалился). **Осторожно:** удалённую ветку можно восстановить, если вы знаете хеш её последнего коммита, но лучше не доводить до этого.

### 2. Переименование ветки: `git branch -m`

Название ветки должно отражать её суть. Если вы поняли, что `feature/bug-fix` на самом деле реализует новую функциональность, переименуйте её.

```bash
# Переименовать текущую ветку
git branch -m new-feature-name

# Переименовать произвольную ветку
git branch -m old-branch-name new-branch-name
```

**Важный нюанс:** Если вы переименовываете ветку, которая уже была отправлена на удалённый сервер (`git push`), вам потребуется удалить старую ветку на сервере и отправить новую:

```bash
git push origin --delete old-branch-name
git push origin new-branch-name
git branch --set-upstream-to=origin/new-branch-name new-branch-name
```

### 3. Создание и переключение: `git switch -c` против `git checkout -b`

Мы уже касались этого, но сейчас закрепим с акцентом на командную строку.

| Современный синтаксис (Git ≥ 2.23) | Классический синтаксис |
| :--- | :--- |
| `git switch -c feature/new` | `git checkout -b feature/new` |
| `git switch main` | `git checkout main` |

Обе команды делают одно и то же. `switch` был добавлен для ясности: его единственная задача — работа с ветками. `checkout` же исторически умеет и ветки переключать, и файлы восстанавливать, что иногда приводит к путанице. В профессиональной среде допустимо использовать и то, и другое, но я рекомендую привыкать к `switch` как к более безопасному и семантически чистому инструменту.

**Лайфхак: переключение на предыдущую ветку**
Вам часто приходится метаться между двумя ветками. Используйте дефис `-` как сокращение для «предыдущей ветки»:

```bash
git switch -
# или
git checkout -
```

Git запоминает, где вы были до последнего переключения, и мгновенно возвращает вас обратно.

### 4. Отмена слияния при конфликте: `git merge --abort`

Вы запустили `git merge feature`, и Git сообщил о конфликтах. Вы смотрите на экран и понимаете, что сейчас не готовы их разрешать, или хотите начать слияние заново с другого состояния. **Не паникуйте и не пытайтесь вручную восстанавливать файлы!** У Git есть элегантный способ вернуться к состоянию до начала слияния:

```bash
git merge --abort
```

Эта команда останавливает процесс слияния и возвращает ваш рабочий каталог и индекс ровно к тому состоянию, которое было до выполнения `git merge`. Все артефакты слияния (файлы с маркерами конфликтов) исчезают.

**Альтернативы:**
- `git merge --continue` — продолжить слияние после ручного разрешения конфликтов.
- `git merge --quit` — забыть о текущем слиянии, но оставить рабочий каталог как есть (продвинутый сценарий).

### 5. Сравнение веток: `git diff branch1..branch2`

Чтобы понять, какие изменения привнесёт слияние, полезно сравнить ветки напрямую.

**Синтаксис:**
```bash
git diff main..feature
```

**Что это показывает:** Разницу между последним общим предком веток и состоянием `feature`. Фактически это те изменения, которые будут добавлены в `main`, если вы выполните `git merge feature`.

**Вариации:**
- `git diff main...feature` (три точки) — показывает разницу между последним общим предком и `feature` (то же самое, что и две точки, но более явно).
- `git diff main feature` (пробел вместо точек) — показывает прямую разницу между кончиками веток (менее информативно для оценки слияния).

### 6. Анализ слитых и неслитых веток

В проекте с активной разработкой количество веток растёт как снежный ком. Чтобы навести порядок, нужно знать, какие ветки уже можно удалять, а какие ещё содержат уникальную работу.

**Просмотр веток, слитых с текущей:**
```bash
git branch --merged
```
Выводит список веток, все коммиты которых уже достижимы из текущей ветки. Обычно их можно смело удалять (`git branch -d`). Ветка `main` (или та, на которой вы стоите) будет помечена звёздочкой.

**Просмотр веток, НЕ слитых с текущей:**
```bash
git branch --no-merged
```
Показывает ветки, в которых есть уникальные коммиты. Удалять их через `-d` Git не позволит.

**Просмотр относительно другой ветки:**
Вы можете указать ветку, относительно которой производить проверку:
```bash
git branch --merged main
git branch --no-merged develop
```

Выполните следующие шаги в новом репозитории, чтобы закрепить навыки.

1.  **Подготовка полигона:**
    ```bash
    mkdir branch-management-lab && cd branch-management-lab
    git init
    echo "initial" > README.md && git add . && git commit -m "Initial commit"
    ```

2.  **Создайте 4 ветки:**
    ```bash
    git switch -c feature/login
    echo "login logic" > login.txt && git add . && git commit -m "Add login"
    
    git switch -c feature/payment
    echo "payment logic" > payment.txt && git add . && git commit -m "Add payment"
    
    git switch -c experiment/new-parser
    echo "parser draft" > parser.txt && git add . && git commit -m "Draft parser"
    
    git switch -c hotfix/typo
    echo "fix typo" > README.md && git add . && git commit -m "Fix typo in README"
    ```

3.  **Слейте одну ветку и удалите её безопасно:**
    ```bash
    git switch main
    git merge hotfix/typo
    git branch -d hotfix/typo   # Успешно, так как ветка слита
    ```

4.  **Попробуйте удалить ветку с неслитыми изменениями:**
    ```bash
    git branch -d feature/payment
    ```
    Git откажет, выдав предупреждение. Запомните это поведение.

5.  **Переименуйте ветку:**
    ```bash
    git branch -m feature/login feature/auth
    ```

6.  **Проверьте списки слитых и неслитых веток:**
    ```bash
    git branch --merged main
    git branch --no-merged main
    ```
    Убедитесь, что `feature/auth` и `experiment/new-parser` в списке неслитых.

7.  **Принудительно удалите экспериментальную ветку:**
    ```bash
    git branch -D experiment/new-parser
    ```

8.  **Посмотрите итоговый список веток:**
    ```bash
    git branch
    ```
    Должны остаться `main` и `feature/auth`.

### Вопросы для самопроверки

**1. Что означает флаг `-D` в команде `git branch -D`? Чем он отличается от `-d`?**
> **Ответ:** `-D` — это алиас для `--delete --force`. Он **принудительно удаляет ветку**, даже если она не была слита с текущей. `-d` удаляет ветку только в том случае, если она полностью слита, защищая от потери данных.

**2. Как быстро переключиться на предыдущую ветку, не вспоминая её точное название?**
> **Ответ:** Использовать дефис: `git switch -` или `git checkout -`. Git интерпретирует `-` как имя предыдущей ветки, на которой вы находились до последнего переключения.

**3. Вы выполнили `git merge feature`, возник конфликт, и вы хотите полностью отменить слияние и вернуться к состоянию до команды. Какую команду нужно выполнить?**
> **Ответ:** `git merge --abort`. Она откатывает все изменения, связанные с начатым слиянием, и восстанавливает чистый рабочий каталог и индекс.

**4. Как посмотреть список веток, которые уже были слиты в `main` и, скорее всего, могут быть удалены?**
> **Ответ:** Находясь в ветке `main`, выполните `git branch --merged`. Или из любой ветки: `git branch --merged main`.

**5. Можно ли переименовать ветку, на которой вы сейчас находитесь? Если да, как?**
> **Ответ:** Да, с помощью `git branch -m new-name`. Если вы хотите переименовать другую ветку, нужно указать и старое, и новое имя: `git branch -m old-name new-name`.
