Приветствую, коллеги. Сегодня я выступаю в роли медиатора — специалиста по мирному урегулированию споров между ветками кода. Конфликты слияния пугают новичков, но я покажу вам, что это не катастрофа, а рабочий момент, у которого есть чёткий протокол разрешения. После этого урока вы будете встречать сообщение `CONFLICT` с холодной головой и ясным планом действий.

### Почему возникают конфликты слияния?

Git отлично справляется с автоматическим объединением изменений, если они происходят в **разных файлах** или в **разных частях одного файла**. Конфликт возникает, когда две ветки меняют **одни и те же строки** по-разному, либо когда одна ветка удаляет файл, а другая его изменяет. Git не может решить, какое изменение правильное, и передаёт решение вам — разработчику.

**Классический сценарий:**
- Ветка `main`: изменили заголовок страницы с "Welcome" на "Главная".
- Ветка `feature`: в том же месте изменили заголовок с "Welcome" на "Добро пожаловать".

При попытке слить `feature` в `main` Git останавливается и сообщает о конфликте.

### Как распознать конфликт: вывод `git status` и маркеры

Выполнив `git merge feature`, вы можете увидеть такой вывод:

```
Auto-merging index.html
CONFLICT (content): Merge conflict in index.html
Automatic merge failed; fix conflicts and then commit the result.
```

Команда `git status` покажет:

```
On branch main
You have unmerged paths.
  (fix conflicts and run "git commit")
  (use "git merge --abort" to abort the merge)

Unmerged paths:
  (use "git add <file>..." to mark resolution)
        both modified:   index.html
```

Файл `index.html` помечен как `both modified`. Если открыть его в редакторе, вы увидите специальные маркеры, вставленные Git:

```html
<<<<<<< HEAD
<h1>Главная</h1>
=======
<h1>Добро пожаловать</h1>
>>>>>>> feature
```

**Расшифровка маркеров:**
- `<<<<<<< HEAD` — начало блока из **текущей ветки** (`main`), на которую вы пытаетесь слить изменения.
- `=======` — разделитель между версиями.
- `>>>>>>> feature` — конец блока из **сливаемой ветки** (`feature`).

Всё, что между `<<<<<<< HEAD` и `=======`, — это «наша» версия (ours). Всё, что между `=======` и `>>>>>>> feature`, — это «их» версия (theirs).

### Методы разрешения конфликтов

#### 1. Ручное разрешение в редакторе

Самый надёжный и гибкий способ. Вы открываете файл, анализируете оба варианта и решаете, какой код оставить. Часто нужно не просто выбрать один из вариантов, а написать нечто среднее (например, объединить изменения из обеих веток).

**Действия:**
1. Отредактируйте файл, удалив маркеры конфликта и оставив нужный код.
2. Убедитесь, что файл синтаксически корректен.
3. Сохраните файл.
4. Сообщите Git, что конфликт разрешён: `git add index.html`.
5. Повторите для всех конфликтующих файлов.
6. Завершите слияние: `git commit` (Git автоматически предложит стандартное сообщение merge commit, которое можно оставить или отредактировать).

#### 2. Использование `git mergetool`

Если вы визуал и предпочитаете графический интерфейс для разрешения конфликтов, Git может запустить внешний инструмент слияния (например, VS Code, Meld, KDiff3).

**Настройка (один раз):**
```bash
git config --global merge.tool vscode
git config --global mergetool.vscode.cmd 'code --wait $MERGED'
```

**Запуск:**
```bash
git mergetool
```

Git по очереди откроет конфликтующие файлы в выбранной программе, где вы сможете наглядно сравнить версии и принять решение. После закрытия инструмента Git автоматически пометит файл как разрешённый (выполнит `git add`).

#### 3. Стратегическое разрешение: `--ours` и `--theirs`

Бывают ситуации, когда вы заранее знаете, что хотите принять версию одной из веток целиком, без ручного редактирования. Git предоставляет для этого быстрые опции.

**Принять версию текущей ветки (ours):**
```bash
git checkout --ours index.html
git add index.html
```

**Принять версию сливаемой ветки (theirs):**
```bash
git checkout --theirs index.html
git add index.html
```

**Важно:** После использования этих команд маркеры конфликта исчезают, и файл полностью заменяется на выбранную версию. Это действие нельзя отменить через `undo` в редакторе, но можно восстановить состояние до слияния с помощью `git merge --abort`.

**Что означает `theirs` в контексте слияния?**
Когда вы выполняете `git merge feature`, находясь на `main`, `theirs` указывает на версию из **ветки `feature`**. Если вы выполняете `git rebase`, роли меняются, но об этом позже.

### Отмена слияния: `git merge --abort`

Если вы начали слияние, увидели конфликты и поняли, что сейчас не готовы их разрешать (или ошиблись веткой), вы можете полностью откатить всё к состоянию до команды `git merge`:

```bash
git merge --abort
```

Эта команда удаляет все следы незавершённого слияния и возвращает рабочий каталог и индекс в исходное состояние. Идеальный способ сказать «Давайте попробуем позже».

### Завершение после разрешения: `git merge --continue`

После того как вы разрешили все конфликты и добавили файлы в индекс (`git add`), вместо обычного `git commit` можно использовать:

```bash
git merge --continue
```

Git создаст коммит слияния с автоматически сгенерированным сообщением. Это просто удобная обёртка над `git commit`.

### Практическое задание: Создаём и разрешаем конфликт

Выполните это упражнение шаг за шагом, чтобы почувствовать себя уверенно при реальном конфликте.

1.  **Создайте новый репозиторий и файл:**
    ```bash
    mkdir conflict-lab && cd conflict-lab
    git init
    echo "Hello World" > greeting.txt
    git add greeting.txt && git commit -m "Initial commit"
    ```

2.  **Создайте две ветки от этого коммита:**
    ```bash
    git switch -c feature/spanish
    echo "Hola Mundo" > greeting.txt
    git add greeting.txt && git commit -m "Translate to Spanish"

    git switch -c feature/german main  # создаём ветку german от main (коммита до испанского)
    # Альтернативно: git switch main, git switch -c feature/german
    ```

3.  **Внесите конфликтующее изменение в ветке `feature/german`:**
    ```bash
    # Мы уже на ветке german, но если нет: git switch feature/german
    echo "Hallo Welt" > greeting.txt
    git add greeting.txt && git commit -m "Translate to German"
    ```

4.  **Попробуйте слить обе ветки в `main`:**
    ```bash
    git switch main
    git merge feature/spanish   # Fast-forward, конфликтов нет
    git merge feature/german    # Здесь возникнет конфликт!
    ```

5.  **Посмотрите статус и содержимое файла:**
    ```bash
    git status
    cat greeting.txt
    ```
    Вы увидите маркеры конфликта с `Hola Mundo` (HEAD) и `Hallo Welt` (feature/german).

6.  **Разрешите конфликт вручную:**
    Отредактируйте `greeting.txt`, оставив, например, "Hola Mundo" (или придумайте комбинированное приветствие). Сохраните файл.
    ```bash
    git add greeting.txt
    git commit -m "Merge feature/german, resolved conflict manually"
    ```

7.  **Повторите эксперимент, но используйте `--theirs`:**
    Верните `main` назад до слияния с german:
    ```bash
    git reset --hard HEAD~1  # отменяем merge commit
    git merge feature/german # снова конфликт
    git checkout --theirs greeting.txt
    git add greeting.txt
    git commit -m "Merge feature/german, accepting their version"
    cat greeting.txt  # Убедитесь, что теперь там "Hallo Welt"
    ```

8.  **Попробуйте `git merge --abort`:**
    Верните `main` ещё раз:
    ```bash
    git reset --hard HEAD~1
    git merge feature/german
    # Не разрешайте конфликт, сразу выполните:
    git merge --abort
    git status  # Всё чисто, слияние отменено
    ```

### Вопросы для самопроверки

**1. Как посмотреть список конфликтующих файлов, не заглядывая в каждый файл?**
> **Ответ:** Команда `git status` покажет их в секции `Unmerged paths`. Также можно использовать `git diff --name-only --diff-filter=U`.

**2. Что означает `theirs` при разрешении конфликта слияния? К какой ветке относится эта версия?**
> **Ответ:** `theirs` означает версию из **сливаемой ветки** (той, которую вы передаёте в команду `git merge`). Если вы находитесь на `main` и выполняете `git merge feature`, то `theirs` — это состояние файла в ветке `feature`. `ours` — состояние в текущей ветке `main`.

**3. Вы разрешили конфликты в двух файлах, добавили их через `git add`, но забыли про третий файл. Что произойдёт при попытке выполнить `git commit`?**
> **Ответ:** Git не позволит завершить коммит, так как слияние не завершено. Он выдаст сообщение: `fatal: cannot do a partial commit during a merge.` Нужно разрешить конфликты во **всех** файлах и добавить их, либо прервать слияние через `git merge --abort`.

**4. Можно ли отменить ошибочное разрешение конфликта до того, как сделан `git add`?**
> **Ответ:** Да, просто отредактируйте файл снова или восстановите его из индекса командой `git restore --staged <file>` и затем `git restore <file>` (что вернёт файл с маркерами конфликта). После `git add` восстановить маркеры сложнее, но можно сбросить индекс и перезаписать файл версией с маркерами с помощью `git checkout -m <file>` (старый способ).

**5. В каком случае стоит использовать `git merge --abort`, а не пытаться разрешить конфликт?**
> **Ответ:** Когда вы понимаете, что начали слияние не с той веткой, или когда объём конфликтов слишком велик, и вы хотите подготовиться к слиянию лучше (например, сначала подлить изменения из `main` в свою фичу, разрешив конфликты там). Также если вы случайно запустили слияние и сразу осознали ошибку.
