Приветствую, коллеги. Сегодня мы поговорим о команде, которая требует особой аккуратности и точности — **`git cherry-pick`**. Это скальпель в наборе хирурга Git, позволяющий извлечь один конкретный коммит из истории и применить его к текущей ветке, не трогая всё остальное. В отличие от грубого слияния целых веток, `cherry-pick` даёт вам полный контроль над тем, какие именно изменения попадут в целевую ветку.

### Что такое `git cherry-pick`?

Представьте, что история коммитов — это дерево с ветками. `git merge` сращивает целые ветви, `git rebase` пересаживает ветку на новое основание, а `git cherry-pick` срывает один спелый плод (коммит) и прикрепляет его к другой ветке.

**Определение:** `git cherry-pick` создаёт **новый коммит** в текущей ветке, который вносит **точно такие же изменения**, как и указанный существующий коммит из другой ветки.

**Важно понимать:**
- Cherry-pick **копирует** изменения, а не перемещает коммит. В результате в репозитории появляются два коммита с одинаковыми изменениями, но разными хешами (так как у них разные родители и временные метки). Это называется **дублированием коммитов**.

### Базовый синтаксис

```bash
# Перенести один коммит
git cherry-pick <commit-hash>

# Перенести несколько коммитов (через пробел)
git cherry-pick <hash1> <hash2> <hash3>

# Перенести диапазон коммитов (A — более старый, B — более новый)
git cherry-pick <hashA>..<hashB>   # Не включает A, включает B
git cherry-pick <hashA>^..<hashB>  # Включает A и B
```

### Полезные опции

| Опция | Описание |
| :--- | :--- |
| `-n` или `--no-commit` | Применить изменения к рабочему каталогу и индексу, но **не создавать коммит**. Полезно, если вы хотите объединить несколько cherry-pick'ов в один коммит или доработать изменения перед фиксацией. |
| `-e` или `--edit` | Позволяет **отредактировать сообщение коммита** перед его созданием. По умолчанию Git копирует сообщение исходного коммита. |
| `-x` | Добавляет в сообщение нового коммита строку `(cherry picked from commit <hash>)`. Полезно для отслеживания происхождения изменения в больших проектах. |

### Разрешение конфликтов при cherry-pick

Если изменения из коммита не могут быть применены чисто (например, контекст кода изменился), Git останавливает процесс и сообщает о конфликте.

**Что делать:**

1.  Разрешить конфликты вручную в файлах, удалив маркеры `<<<<<<<`, `=======`, `>>>>>>>`.
2.  Добавить разрешённые файлы в индекс: `git add <file>`.
3.  Продолжить cherry-pick:
    ```bash
    git cherry-pick --continue
    ```
    Git создаст коммит с оригинальным сообщением (или предложит отредактировать).

**Если вы передумали и хотите отменить cherry-pick:**
```bash
git cherry-pick --abort
```
Это вернёт ветку в состояние до начала операции.

### Реальные сценарии использования

**Сценарий 1: Срочный фикс в продакшене**
В ветке `main` обнаружен критический баг. Вы быстро создаёте ветку `hotfix`, исправляете баг в коммите `A`, и вливаете её в `main`. Но в параллельной ветке `release/2.0`, которая ещё не готова к релизу, этот баг тоже присутствует и его нужно исправить **прямо сейчас**, не дожидаясь слияния всей `main`.

```bash
git checkout release/2.0
git cherry-pick <hash-коммита-с-фиксом>
```

Теперь фикс есть в обеих ветках, и вы не затащили в релизную ветку лишние изменения из `main`.

**Сценарий 2: Перенос только нужной функциональности**
В ветке `feature/payment` разработчик сделал 10 коммитов: 8 с логикой оплаты и 2 с полезным рефакторингом вспомогательной функции, который пригодится в `main` уже сейчас. Вместо слияния всей ветки (которая ещё не готова), вы можете перенести только рефакторинг:

```bash
git checkout main
git cherry-pick <hash-рефакторинга>
```

### Сравнение с `merge` и `rebase`

| Команда | Что делает | Когда использовать |
| :--- | :--- | :--- |
| `git merge` | Соединяет **целые ветки**, создавая merge commit (или fast-forward). | Когда нужно интегрировать **все** изменения из одной ветки в другую. |
| `git rebase` | Перемещает **цепочку коммитов** на новое основание, переписывая историю. | Когда нужно сохранить линейную историю и обновить ветку относительно `main`. |
| `git cherry-pick` | Копирует **отдельные коммиты**. | Когда нужно перенести **точечные изменения**, не трогая остальные. |

**Ключевое отличие:** Cherry-pick оперирует **коммитами**, а merge — **ветками**.

### Риски и предостережения

1.  **Дублирование коммитов.** Самый главный риск. Если вы зачеррипикаете коммит из ветки, которую позже всё равно сольёте через `merge`, Git увидит изменения дважды. Это не вызовет конфликта (если изменения одинаковые), но может запутать историю и усложнить `git blame`. Чтобы избежать этого, используйте флаг `-x` для маркировки или старайтесь не смешивать cherry-pick и merge для одних и тех же изменений.

2.  **Изменение хеша.** Cherry-pick создаёт **новый коммит** с новым хешем. Если в команде кто-то ссылается на коммиты по хешу, будьте осторожны.

3.  **Функциональные зависимости.** Выбранный коммит может полагаться на изменения, сделанные в предыдущих коммитах этой ветки. Если вы перенесёте только один коммит, код может не компилироваться или работать некорректно. Всегда проверяйте логику перед cherry-pick.

### Практическое задание: Точечная трансплантация

Выполните эти шаги, чтобы освоить cherry-pick на практике.

1.  **Создайте репозиторий и ветку `feature`:**
    ```bash
    mkdir cherrypick-lab && cd cherrypick-lab
    git init
    echo "Line 1" > file.txt && git add file.txt && git commit -m "C1: base"
    git switch -c feature
    ```

2.  **Создайте три коммита в `feature`:**
    ```bash
    echo "Feature A" >> file.txt && git add file.txt && git commit -m "C2: feature A"
    echo "Feature B" >> file.txt && git add file.txt && git commit -m "C3: feature B"
    echo "Feature C" >> file.txt && git add file.txt && git commit -m "C4: feature C"
    git log --oneline   # Запомните хеш коммита C3 (Feature B)
    ```

3.  **Вернитесь в `main` и перенесите только второй коммит (C3):**
    ```bash
    git switch main
    git cherry-pick <хеш_C3>
    ```

4.  **Проверьте содержимое файла и историю:**
    ```bash
    cat file.txt
    # Ожидаем увидеть: Line 1, Feature B (пропущена Feature A)
    git log --oneline
    # Должно быть два коммита в main: C1 и новый коммит (C5) с сообщением C3
    ```

5.  **Попробуйте диапазон (бонус):**
    ```bash
    git cherry-pick <хеш_C2>..<хеш_C4>  # Перенесёт C3 и C4 (не включая C2)
    ```

### Вопросы для самопроверки

**1. Чем принципиально `git cherry-pick` отличается от `git merge`?**
> **Ответ:** `git merge` объединяет **всю историю** сливаемой ветки с текущей, создавая связь между ветками (merge commit). `git cherry-pick` копирует **отдельные коммиты**, создавая их дубликаты в текущей ветке без какой-либо связи с веткой-источником.

**2. Как перенести диапазон коммитов с помощью cherry-pick? Например, все коммиты от `A` (не включая) до `E` (включая).**
> **Ответ:** Использовать синтаксис `git cherry-pick A..E`. Если нужно включить и `A`, то `git cherry-pick A^..E` или `git cherry-pick A E` (если перечислить явно). Git применит коммиты в топологическом порядке (от более старого к новому).

**3. Вы выполнили `git cherry-pick <hash>`, возник конфликт. Вы разрешили его вручную и выполнили `git add`. Какой командой завершить cherry-pick?**
> **Ответ:** `git cherry-pick --continue`. Git создаст коммит с разрешёнными изменениями.

**4. Что произойдёт, если вы попытаетесь зачеррипикать коммит, изменения из которого уже присутствуют в текущей ветке?**
> **Ответ:** Git проанализирует изменения и, скорее всего, выдаст сообщение: `nothing to commit, working tree clean`. Операция завершится без создания нового коммита, так как применять нечего.

**5. Можно ли использовать `cherry-pick` для переноса коммита с сообщением, которое содержит указание на исходный коммит?**
> **Ответ:** Да, с флагом `-x`: `git cherry-pick -x <hash>`. Это добавит строку `(cherry picked from commit <hash>)` в конец сообщения нового коммита, что служит документацией для будущих разработчиков.
