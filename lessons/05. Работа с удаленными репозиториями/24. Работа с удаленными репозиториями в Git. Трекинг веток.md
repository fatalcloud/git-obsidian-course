Приветствую, коллеги. Сегодня мы поговорим о том, как локальные и удалённые ветки «узнают» друг друга и как наладить между ними правильные отношения. Эта тема называется **upstream tracking** (отслеживание upstream-ветки). Понимание этого механизма избавляет от необходимости постоянно писать `origin/main` в командах `pull` и `push`, а также делает работу с ветками более предсказуемой и удобной.

### Что значит «ветка отслеживает удалённую ветку»?

Когда локальная ветка **отслеживает (tracks)** удалённую ветку, между ними устанавливается связь. Эта связь используется Git для нескольких целей:

1.  **`git status`** показывает, на сколько коммитов ваша локальная ветка опережает или отстаёт от отслеживаемой.
2.  **`git pull`** без аргументов знает, с какой удалённой ветки забирать изменения.
3.  **`git push`** без аргументов знает, в какую удалённую ветку отправлять изменения (если настроен `push.default`).

Проще говоря, отслеживание — это ярлык: «Ветка `feature` на моём компьютере связана с веткой `feature` на сервере `origin`».

### Как настроить отслеживание?

Есть несколько способов установить upstream-связь.

#### 1. При создании ветки на основе удалённой

Если вы создаёте локальную ветку, основываясь на существующей удалённой ветке, Git автоматически настраивает отслеживание:

```bash
git checkout -b feature origin/feature
# или современный аналог
git switch -c feature origin/feature
```

Теперь локальная `feature` отслеживает `origin/feature`.

#### 2. При первом пуше с флагом `-u`

Самый распространённый сценарий: вы создали локальную ветку, сделали коммиты и хотите отправить её на сервер **и сразу настроить отслеживание**.

```bash
git push -u origin feature
```

Флаг `-u` (или `--set-upstream`) делает две вещи:
- Отправляет ветку `feature` на сервер.
- Устанавливает для локальной `feature` upstream `origin/feature`.

После этого можно будет писать просто `git push` и `git pull`.

#### 3. Ручная настройка или изменение отслеживания: `git branch -u`

Если вы забыли флаг `-u` или хотите перенаправить отслеживание на другую ветку, используйте команду:

```bash
git branch -u origin/another-branch
```

**Важно:** Эта команда настраивает upstream для **текущей** ветки. Если вы хотите настроить для другой ветки, сначала переключитесь на неё.

### Как посмотреть отслеживание? `git branch -vv`

Флаг `-vv` (двойная `v` для `verbose`) показывает список локальных веток вместе с информацией об их upstream-связях.

```bash
git branch -vv
```

**Пример вывода:**
```
* feature    a1b2c3d [origin/feature] Add new payment logic
  main       e83c516 [origin/main: behind 2] Initial commit
  dev        c5b8e3a [origin/dev: ahead 1, behind 3] WIP on dev
```

**Расшифровка:**
- `feature` отслеживает `origin/feature` и синхронизирована с ней.
- `main` отслеживает `origin/main` и **отстаёт** на 2 коммита (на сервере есть 2 коммита, которых нет локально).
- `dev` отслеживает `origin/dev` и **опережает** на 1 коммит (есть локальный коммит, не отправленный на сервер) и **отстаёт** на 3 коммита (на сервере есть 3 коммита, которых нет локально).

### Что делает `git pull` без аргументов?

Когда вы выполняете `git pull` в ветке, у которой настроен upstream, Git:

1.  Определяет, какой удалённый репозиторий и какую ветку использовать (из upstream-конфигурации).
2.  Выполняет `git fetch` для этого удалённого репозитория.
3.  Выполняет `git merge` для слияния отслеживаемой ветки в текущую.

Если upstream **не настроен**, Git выдаст ошибку и подскажет, как его настроить.

### Разные upstream для pull и push

По умолчанию `git pull` и `git push` используют **один и тот же upstream**. Но Git позволяет настроить **разные** удалённые репозитории или ветки для получения и отправки изменений. Это редко используемая, но мощная возможность.

Настройка производится через `git config`:

```bash
# Установить отдельный push-URL (например, для форков)
git remote set-url --push origin git@github.com:my-fork/repo.git

# Или настроить push для конкретной ветки
git config branch.dev.pushRemote origin
git config branch.dev.push dev2   # пушить в ветку dev2 вместо dev
```

В этом случае `git pull` будет тянуть с основного репозитория, а `git push` — отправлять в ваш форк или другую ветку.

### Пример: ветка `dev` трекает `origin/dev`, но пушит в `origin/dev2`

Представим, что вы работаете в ветке `dev`, которая отслеживает `origin/dev` для получения обновлений от команды. Но по какой-то причине вы хотите отправлять свои изменения в отдельную ветку `dev2` (например, для тестирования).

**Пошагово:**

1.  Убедимся, что локальная `dev` отслеживает `origin/dev`:
    ```bash
    git checkout dev
    git branch -u origin/dev
    ```

2.  Настроим push для текущей ветки на другую удалённую ветку:
    ```bash
    git config branch.dev.pushRemote origin
    git config branch.dev.merge refs/heads/dev2
    ```
    Или более простым способом, используя команду `git push` с указанием целевой ветки и сохранением настроек:
    ```bash
    git push -u origin dev:dev2
    ```
    Это отправит локальную `dev` в удалённую `dev2` и запомнит эту связь для будущих пушей.

3.  Теперь:
    - `git pull` будет забирать изменения из `origin/dev` и сливать в локальную `dev`.
    - `git push` будет отправлять изменения из локальной `dev` в `origin/dev2`.

### Что означает `origin/HEAD -> origin/main`?

В выводе команды `git branch -r` или `git remote show origin` можно увидеть строку:

```
origin/HEAD -> origin/main
```

**Объяснение:** Удалённый репозиторий тоже имеет указатель `HEAD`, который определяет **ветку по умолчанию** на сервере. Обычно это `main` или `master`. `origin/HEAD` — это локальное отражение этого указателя. Git хранит его как символическую ссылку, чтобы знать, какую ветку использовать по умолчанию при клонировании.

Вы можете обновить или установить `origin/HEAD` локально командой:
```bash
git remote set-head origin -a
```
Это полезно, если ветка по умолчанию на сервере изменилась (например, переименовали `master` в `main`).

### Практическое задание: Тренируем upstream-связи

1.  **Подготовка:**
    ```bash
    mkdir tracking-lab && cd tracking-lab
    git init
    git commit --allow-empty -m "Initial commit"
    # Представим, что у нас есть удалённый репозиторий (симулируем локально)
    git remote add origin /tmp/fake-remote.git  # не обязательно создавать, просто пример
    ```

2.  **Создайте локальную ветку и запушьте её с флагом `-u`:**
    ```bash
    git checkout -b feature
    git push -u origin feature
    ```

3.  **Проверьте связь:**
    ```bash
    git branch -vv
    ```
    Вы должны увидеть `[origin/feature]`.

4.  **Создайте вторую ветку без флага `-u`, а затем настройте отслеживание вручную:**
    ```bash
    git checkout -b experiment
    git push origin experiment   # без -u
    git branch -vv               # upstream не указан
    git branch -u origin/experiment
    git branch -vv               # теперь связь есть
    ```

5.  **Измените отслеживаемую ветку:**
    ```bash
    # Создадим ещё одну удалённую ветку
    git checkout -b temp
    git push origin temp
    
    # Вернёмся в experiment и перенаправим отслеживание на temp
    git checkout experiment
    git branch -u origin/temp
    git branch -vv   # теперь experiment трекает origin/temp
    ```

6.  **Симулируйте разные upstream для pull и push (бонус):**
    ```bash
    git checkout feature
    # Настроим, чтобы push шёл в другую ветку
    git config branch.feature.pushRemote origin
    git config branch.feature.push refs/heads/feature-experimental
    # Теперь git pull будет из origin/feature, а git push в origin/feature-experimental
    ```

### Вопросы для самопроверки

**1. Что означает запись `origin/HEAD -> origin/main` в выводе `git branch -r`?**
> **Ответ:** Это символическая ссылка, показывающая **ветку по умолчанию** на удалённом репозитории. При клонировании Git именно эту ветку извлекает и создаёт локально. `origin/HEAD` — локальное отражение этого указателя.

**2. Как изменить отслеживаемую ветку для текущей локальной ветки?**
> **Ответ:** Использовать команду `git branch -u <новый_удалённый_репозиторий>/<ветка>`. Например: `git branch -u origin/main`. Это перенаправит upstream текущей ветки на `origin/main`.

**3. Вы создали ветку `login`, сделали коммиты и выполнили `git push origin login` (без `-u`). Будет ли работать `git pull` в этой ветке?**
> **Ответ:** Нет, `git pull` выдаст ошибку, потому что upstream не настроен. Git не знает, с какой удалённой веткой синхронизироваться. Нужно либо выполнить `git push -u origin login`, либо вручную настроить трекинг: `git branch -u origin/login`.

**4. Что показывают числа `ahead N` и `behind M` в выводе `git branch -vv`?**
> **Ответ:** `ahead N` означает, что в локальной ветке есть **N коммитов**, которых ещё нет в отслеживаемой удалённой ветке (их нужно запушить). `behind M` означает, что в удалённой ветке есть **M коммитов**, которых нет локально (их нужно запуллить).

**5. Можно ли настроить так, чтобы `git pull` тянул с одного сервера, а `git push` отправлял на другой?**
> **Ответ:** Да, это возможно. Для всей ветки можно задать разные URL для fetch и push через `git remote set-url --push <remote> <url>`, или точечно для ветки через `git config branch.<имя>.pushRemote`. Это часто используется при работе с форками: fetch с оригинального репозитория (`upstream`), push в свой форк (`origin`).
