Приветствую, коллеги! Сегодня мы переходим к разделу, который превращает Git из личного инструмента в командный. До сих пор мы работали исключительно в локальном репозитории — вся история, все ветки и коммиты жили только на вашем компьютере. Теперь мы научимся связывать локальный репозиторий с удалённым, чтобы делиться кодом с коллегами, делать резервные копии и участвовать в совместной разработке.

### Что такое удалённый репозиторий?

**Удалённый репозиторий (remote)** — это версия вашего проекта, сохранённая на сервере в интернете или в локальной сети. Это может быть GitHub, GitLab, Bitbucket, корпоративный сервер или даже другая папка на вашем компьютере. Удалённый репозиторий — это точка синхронизации для всей команды.

Связь с удалённым репозиторием настраивается через **URL** (например, `https://github.com/user/repo.git`) или **SSH-путь** (`git@github.com:user/repo.git`). Каждому удалённому репозиторию вы даёте короткое имя, чтобы не набирать URL каждый раз.

### Имена удалённых репозиториев: `origin` и `upstream`

- **`origin`** — это стандартное имя для вашего **основного** удалённого репозитория. Когда вы клонируете проект (`git clone`), Git автоматически создаёт удалённую ссылку с именем `origin`, указывающую на исходный URL.
- **`upstream`** — общепринятое имя для **родительского** репозитория, из которого вы сделали форк (fork). В open-source разработке вы обычно работаете в своём форке (`origin`), а оригинальный репозиторий добавляете как `upstream`, чтобы периодически подтягивать оттуда обновления.

### Основные команды для работы с удалёнными репозиториями

#### 1. Добавление удалённого репозитория: `git remote add`

Если вы создали локальный репозиторий с нуля (`git init`), вам нужно связать его с пустым репозиторием на GitHub.

```bash
git remote add origin https://github.com/ваш-логин/имя-репозитория.git
```

После выполнения этой команды Git запомнит, что имя `origin` соответствует указанному URL.

**Что происходит в `.git/config`?**
Git добавляет в конфигурационный файл локального репозитория секцию:

```
[remote "origin"]
    url = https://github.com/ваш-логин/имя-репозитория.git
    fetch = +refs/heads/*:refs/remotes/origin/*
```

Это означает: «Для удалённого репозитория с именем `origin` используй этот URL, и когда делаешь `fetch`, забирай все ветки с префиксом `origin/`».

#### 2. Просмотр списка удалённых репозиториев: `git remote -v`

```bash
git remote -v
```

Вывод будет примерно таким:
```
origin  https://github.com/user/repo.git (fetch)
origin  https://github.com/user/repo.git (push)
```

Флаг `-v` (verbose) показывает URL-адреса как для получения (`fetch`), так и для отправки (`push`) данных. Обычно они совпадают, но могут быть настроены раздельно.

#### 3. Переименование удалённого репозитория: `git remote rename`

Если вы ошиблись с именем или хотите привести к единому стандарту:

```bash
git remote rename origin upstream
```

Теперь удалённый репозиторий будет называться `upstream`.

#### 4. Удаление ссылки на удалённый репозиторий: `git remote remove`

Если связь больше не нужна (например, вы удалили проект с GitHub или переехали на другой хостинг):

```bash
git remote remove origin
```

Эта команда удаляет секцию `[remote "origin"]` из `.git/config`. Сами данные удалённого репозитория при этом не удаляются с сервера.

### Просмотр удалённых веток: `git branch -r`

После того как вы добавили удалённый репозиторий и выполнили `git fetch`, Git сохраняет информацию о ветках, существующих на сервере. Их можно посмотреть:

```bash
git branch -r
```

Вывод:
```
origin/main
origin/feature/login
origin/hotfix/bug-123
```

Это **удалённые отслеживающие ветки** (remote-tracking branches). Они доступны только для чтения и отражают состояние веток на сервере на момент последней синхронизации (`fetch` или `pull`). Вы не можете переключиться на них напрямую для коммитов, но можете создать локальную ветку на их основе.

### `git fetch` и `git pull`: в чём разница?

Эти две команды часто путают, но разница принципиальна.

- **`git fetch origin`** — **скачивает** все новые коммиты и ветки с удалённого репозитория, но **не изменяет ваш рабочий каталог и локальные ветки**. Это безопасная операция «разведки». После `fetch` вы можете посмотреть, что изменилось на сервере, сравнить ветки (`git diff origin/main`) и только потом решить, сливать изменения или нет.

- **`git pull origin main`** — это сокращение для двух операций:
  1. `git fetch origin`
  2. `git merge origin/main`

`git pull` сразу загружает изменения **и сливает их в вашу текущую ветку**. Это удобно, но может привести к неожиданным конфликтам, если вы не ожидали изменений. Профессионалы часто предпочитают сначала делать `fetch`, а затем осознанно выполнять слияние.

### Отправка изменений: `git push -u origin main`

Когда вы готовы поделиться своей работой, вы «пушите» локальные коммиты на сервер.

```bash
git push -u origin main
```

**Разбор команды:**
- `git push` — отправить изменения.
- `origin` — имя удалённого репозитория.
- `main` — имя локальной ветки, которую вы отправляете.

**Флаг `-u` (или `--set-upstream`)** устанавливает **связь** между вашей локальной веткой `main` и удалённой `origin/main`. После этого в будущем можно будет писать просто `git push` или `git pull` без указания `origin` и имени ветки. Git будет знать, куда отправлять изменения по умолчанию.

**Важно:** При первой отправке новой ветки флаг `-u` обязателен для установки upstream-связи.

### Практическое задание: Связываем локальный и удалённый репозитории

1.  **Создайте пустой репозиторий на GitHub/GitLab.**
    - Зайдите на сайт, нажмите «New repository».
    - Дайте имя, например `my-first-remote`.
    - **Не создавайте** README, .gitignore и лицензию через интерфейс (чтобы репозиторий был полностью пустым).
    - Скопируйте URL репозитория.

2.  **Создайте локальный репозиторий и свяжите его с удалённым:**
    ```bash
    mkdir my-first-remote && cd my-first-remote
    git init
    echo "# My First Remote Project" > README.md
    git add README.md
    git commit -m "Initial commit"
    
    git remote add origin <вставьте_скопированный_URL>
    git remote -v   # убедитесь, что origin добавился
    ```

3.  **Отправьте локальные изменения на сервер:**
    ```bash
    git push -u origin main
    ```
    Если ваша локальная ветка по умолчанию называется `master`, а на GitHub ожидается `main`, можно переименовать локальную ветку перед пушем:
    ```bash
    git branch -M main
    git push -u origin main
    ```

4.  **Проверьте результат в браузере.** Обновите страницу репозитория на GitHub — там должен появиться ваш `README.md`.

5.  **Имитация работы второго разработчика (бонус):**
    - Создайте ещё одну копию локально: `cd .. && git clone <URL> my-first-remote-clone`
    - Зайдите в неё, внесите изменения в README.md, закоммитьте и запушьте.
    - Вернитесь в первый локальный репозиторий и выполните `git fetch origin`, затем `git log origin/main` — вы увидите новый коммит. После этого сделайте `git merge origin/main` (или просто `git pull`), чтобы синхронизироваться.

### Вопросы для самопроверки

**1. Что именно хранится в файле `.git/config` после выполнения команды `git remote add origin <url>`?**
> **Ответ:** Добавляется секция `[remote "origin"]` с полями `url` и `fetch`. В `url` записывается переданный URL, а в `fetch` — спецификация для отслеживания веток (обычно `+refs/heads/*:refs/remotes/origin/*`). Это указывает Git, как сопоставлять удалённые ветки с локальными отслеживающими.

**2. Как полностью удалить ссылку на удалённый репозиторий? Затрагивает ли это сам удалённый репозиторий на сервере?**
> **Ответ:** Команда `git remote remove <имя>` (например, `git remote remove origin`). Она удаляет только **локальную ссылку** на удалённый репозиторий (записи в `.git/config` и локальные отслеживающие ветки вида `origin/*`). Сам удалённый репозиторий на GitHub/GitLab **не затрагивается** и продолжает существовать.

**3. Чем `git fetch` отличается от `git pull`? В каком случае стоит предпочесть `fetch`?**
> **Ответ:** `git fetch` скачивает изменения с удалённого репозитория, но **не изменяет рабочую копию** и локальные ветки. `git pull` = `git fetch` + `git merge`, то есть сразу пытается слить изменения. `fetch` предпочтителен, когда вы хотите сначала оценить, что изменилось на сервере, сравнить ветки, и только потом осознанно выполнить слияние. Это снижает риск неожиданных конфликтов.

**4. Что делает флаг `-u` в команде `git push -u origin main`? Нужно ли его указывать при последующих пушах?**
> **Ответ:** Флаг `-u` (или `--set-upstream`) устанавливает **upstream-связь** между локальной веткой `main` и удалённой `origin/main`. После однократного использования при последующих пушах и пуллах из этой ветки достаточно просто писать `git push` или `git pull` без дополнительных аргументов. Связь сохраняется в конфигурации (`.git/config`, секция `[branch "main"]`).

**5. Вы склонировали репозиторий с GitHub. Какое имя удалённого репозитория Git создаст автоматически?**
> **Ответ:** По умолчанию создаётся удалённый репозиторий с именем **`origin`**, указывающий на URL, с которого был сделан `git clone`.
