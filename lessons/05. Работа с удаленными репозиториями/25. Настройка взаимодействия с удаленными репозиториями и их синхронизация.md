Приветствую, коллеги. Сегодня мы завершаем раздел о взаимодействии с удалёнными репозиториями. Мы уже умеем отправлять и получать изменения, но сейчас соберём полную картину: от момента, когда коллега запушил новый код, до момента, когда ваш код успешно интегрирован. Я, как DevOps-инженер, покажу вам не только «как», но и «как правильно», чтобы ваша работа в команде была плавной, а история репозитория — чистой.

### Полный цикл синхронизации: от сервера до сервера

Представьте стандартный рабочий день. Вы приходите утром, и ваша задача — синхронизировать локальный репозиторий с общим, внести свои изменения и отправить их обратно. Этот цикл состоит из трёх ключевых этапов.

#### 1. Получение изменений: `git fetch`

Всё начинается с разведки. Вместо того чтобы сразу дёргать `git pull` и рисковать получить конфликт в неподходящий момент, выполните:

```bash
git fetch origin
```

**Что происходит:** Git связывается с удалённым репозиторием `origin` и скачивает все новые коммиты и ветки. Ваш **рабочий каталог и локальные ветки остаются нетронутыми**. Обновляются только **удалённые отслеживающие ветки** (`origin/main`, `origin/feature`).

**Зачем:** Вы получаете актуальную картину мира, не меняя свой код. Вы можете посмотреть, что изменилось: `git log origin/main` или `git diff main origin/main`.

#### 2. Интеграция изменений: `git merge` или `git rebase`

Теперь, когда вы знаете, что нового на сервере, нужно влить эти изменения в свою локальную ветку. У вас есть выбор: **merge** или **rebase**.

- **`git merge origin/main`** — создаёт **merge commit**, объединяя две истории. Это безопасно и сохраняет хронологию. Подходит для публичных веток и когда вы хотите явно зафиксировать факт синхронизации.

- **`git rebase origin/main`** — переписывает ваши локальные коммиты так, будто вы начали работу **после** всех изменений на сервере. История становится линейной, без лишних «узелков». **Идеально для обновления личных фиче-веток перед созданием Pull Request.**

**Золотое правило:** Если вы обновляете свою личную фиче-ветку, используйте `rebase` для чистоты истории. Если вы работаете в общей ветке (например, `develop`) — используйте `merge`.

#### 3. Отправка изменений: `git push`

После успешной интеграции и тестирования вы отправляете результат на сервер:

```bash
git push origin <имя-ветки>
```

Если upstream настроен (флаг `-u` был использован ранее), достаточно просто `git push`.

### Разрешение конфликтов при `git pull`

Если вы всё же используете `git pull` (который = `fetch` + `merge`), и на сервере есть изменения, конфликтующие с вашими локальными, Git остановит слияние.

**Алгоритм действий при конфликте во время `pull`:**
1.  Git помечает конфликтующие файлы маркерами `<<<<<<<`, `=======`, `>>>>>>>`.
2.  Выполните `git status`, чтобы увидеть список конфликтующих файлов.
3.  Вручную разрешите конфликты в каждом файле, удалив маркеры и оставив корректный код.
4.  Добавьте разрешённые файлы: `git add <файлы>`.
5.  Завершите слияние: `git commit`. Git предложит стандартное сообщение для merge commit.
6.  Теперь можно выполнить `git push`.

**Совет:** Если вы не готовы разрешать конфликт прямо сейчас или ошиблись веткой, всегда можно отменить слияние: `git merge --abort`.

### Сила и опасность: `git push --force` и `--force-with-lease`

Иногда история локальной ветки расходится с удалённой настолько, что обычный `git push` отклоняется с ошибкой:

```
 ! [rejected]        main -> main (non-fast-forward)
error: failed to push some refs to 'origin'
```

Это происходит, когда вы переписали историю (например, сделали `rebase` или `commit --amend` для уже запушенных коммитов). Git защищает вас от случайной перезаписи удалённой ветки.

#### `git push --force` (Красная кнопка)

```bash
git push --force origin feature
```

Эта команда **принудительно заменяет** удалённую ветку вашей локальной версией. Все коммиты, которые были на сервере, но отсутствуют у вас локально, будут **безвозвратно потеряны**.

**Когда это допустимо?**
- **Только на личных ветках**, где вы единственный разработчик, и вы точно знаете, что никто другой не работает с этой веткой.
- **Никогда на общих ветках (`main`, `develop`, `release`).**

**Риск:** Если коллега в этот момент работал на основе старой версии вашей ветки, его работа будет десинхронизирована, и при следующем пуше он может случайно перетереть ваши изменения. Это ведёт к хаосу и потерянным данным.

#### `git push --force-with-lease` (Безопасная альтернатива)

```bash
git push --force-with-lease origin feature
```

Эта команда работает как `--force`, но с **проверкой**. Она отправит изменения **только в том случае**, если удалённая ветка находится именно в том состоянии, которое вы ожидаете (которое вы последний раз видели при `fetch`).

**Как это защищает?**
Если после вашего последнего `fetch` кто-то другой успел запушить коммиты в эту же ветку, `--force-with-lease` **отклонит** пуш и выдаст ошибку. Вы будете вынуждены сначала сделать `fetch`, посмотреть на новые коммиты коллеги, интегрировать их (возможно, через `rebase`) и только потом пушить.

**Рекомендация DevOps:** Всегда, когда вам кажется, что нужен `--force`, используйте `--force-with-lease`. Это спасёт вас от случайной потери чужих изменений.

### Рабочий процесс с форками (Forking Workflow)

Форки — основа open-source и многих корпоративных моделей разработки. Вы не имеете прав на прямую запись в основной репозиторий, поэтому создаёте его **копию (fork)** в своём аккаунте.

**Типичная конфигурация remote:**
- **`origin`** — ваш личный форк на GitHub/GitLab. Сюда вы можете пушить свободно.
- **`upstream`** — оригинальный репозиторий, откуда вы получаете обновления.

**Полный цикл работы с форком:**

1.  **Начальная настройка (один раз):**
    ```bash
    git clone git@github.com:your-username/repo.git
    cd repo
    git remote add upstream https://github.com/original-owner/repo.git
    git remote -v   # проверяем: origin -> ваш форк, upstream -> оригинал
    ```

2.  **Синхронизация с оригиналом (регулярно):**
    ```bash
    git fetch upstream
    git checkout main
    git merge upstream/main   # или git rebase upstream/main
    git push origin main      # обновляем свой форк на GitHub
    ```

3.  **Работа над фичей:**
    ```bash
    git checkout -b feature-xyz
    # ... работа, коммиты ...
    git push -u origin feature-xyz
    ```

4.  **Создание Pull Request:** Через веб-интерфейс GitHub/GitLab из ветки вашего форка (`origin/feature-xyz`) в ветку оригинального репозитория (`upstream/main`).

5.  **Обновление фиче-ветки после изменений в upstream (если PR долго висит):**
    ```bash
    git fetch upstream
    git checkout feature-xyz
    git rebase upstream/main   # перекладываем свою работу на свежий main
    git push --force-with-lease origin feature-xyz  # обновляем PR
    ```

### Чек-лист перед `git push`

Повесьте этот список перед глазами. Он занимает 30 секунд, но экономит часы командного времени.

- [ ] **1. Сделать `git fetch`.** Убедиться, что локальная копия удалённых веток актуальна.
- [ ] **2. Обновить локальную ветку.** Выполнить `git merge` или `git rebase` из `origin/<имя-ветки>`. Разрешить конфликты, если есть.
- [ ] **3. Запустить тесты.** Убедиться, что после слияния ничего не сломалось (`npm test`, `pytest`, `go test`).
- [ ] **4. Проверить статус:** `git status`. Нет ли забытых файлов, которые не должны попасть в репозиторий?
- [ ] **5. Просмотреть дифф:** `git diff origin/<имя-ветки>`. Убедиться, что изменения именно те, что ожидались.
- [ ] **6. Если ветка публичная (над ней работают несколько человек) — НИКОГДА не использовать `--force`.**
- [ ] **7. Если ветка личная и требуется перезапись — использовать `--force-with-lease`.**
- [ ] **8. Выполнить `git push`.**

### Практическое задание: Синхронизация форка

Это упражнение можно выполнить с любым публичным репозиторием на GitHub. Например, возьмите репозиторий [github/gitignore](https://github.com/github/gitignore).

1.  **Создайте форк** репозитория `github/gitignore` в своём аккаунте GitHub (кнопка "Fork" в правом верхнем углу).
2.  **Клонируйте свой форк локально:**
    ```bash
    git clone https://github.com/ваш-логин/gitignore.git
    cd gitignore
    ```
3.  **Добавьте оригинальный репозиторий как `upstream`:**
    ```bash
    git remote add upstream https://github.com/github/gitignore.git
    git remote -v
    ```
4.  **Синхронизируйте свой форк:**
    ```bash
    git fetch upstream
    git checkout main
    git merge upstream/main
    git push origin main
    ```
5.  **Создайте ветку для тестового изменения:**
    ```bash
    git checkout -b add-my-gitignore
    # Внесите любое безобидное изменение, например, создайте файл Test.gitignore с одной строкой "# Test"
    echo "# Test comment" > Test.gitignore
    git add Test.gitignore
    git commit -m "Add test gitignore file"
    git push -u origin add-my-gitignore
    ```
6.  (Опционально) Создайте Pull Request из этой ветки в оригинальный репозиторий (вам его, скорее всего, не примут, но вы увидите процесс).

### Вопросы для самопроверки

**1. Как отменить уже отправленный `git push`?**
> **Ответ:** Отменить можно, но нужно действовать аккуратно, особенно если ветка публичная. Самый безопасный способ — создать **новый коммит, отменяющий изменения** (через `git revert <хеш-нежелательного-коммита>`), и запушить его. Если ветка личная и вы уверены, что никто не успел её склонировать, можно переписать историю локально (`git reset --hard <хеш-до-пуша>`) и принудительно запушить: `git push --force-with-lease`. **Никогда не делайте `--force` на `main` или другие общие ветки!**

**2. Что делать, если `git push` отклонён с ошибкой `non-fast-forward`?**
> **Ответ:** Эта ошибка означает, что удалённая ветка содержит коммиты, которых нет у вас локально.
> 1. **Не применяйте `--force` сразу!**
> 2. Выполните `git fetch origin`.
> 3. Посмотрите историю удалённой ветки: `git log origin/<имя-ветки>`.
> 4. Интегрируйте удалённые изменения в свою локальную ветку: `git rebase origin/<имя-ветки>` (предпочтительно для личных веток) или `git merge origin/<имя-ветки>`.
> 5. Разрешите конфликты, если они возникнут.
> 6. Снова выполните `git push`. Если вы делали `rebase` для личной ветки, может потребоваться `git push --force-with-lease`.

**3. Чем `git push --force-with-lease` принципиально отличается от `git push --force`?**
> **Ответ:** `--force` слепо заменяет удалённую ветку вашей локальной, игнорируя любые изменения, которые могли появиться на сервере. `--force-with-lease` выполняет замену **только в том случае**, если удалённая ветка находится в том состоянии, которое вы ожидаете (на основе вашего локального `origin/<ветка>`). Это защищает от случайной перезаписи работы коллег.

**4. Как правильно синхронизировать форк с оригинальным репозиторием? Опишите последовательность команд.**
> **Ответ:**
> ```bash
> git fetch upstream
> git checkout main
> git merge upstream/main   # или git rebase upstream/main
> git push origin main
> ```

**5. Вы работаете в команде над веткой `feature/shared`. Почему нельзя использовать `git push --force`?**
> **Ответ:** Потому что другие разработчики могли уже запушить свои коммиты в эту ветку или основывать на ней свою работу. `--force` перезапишет удалённую ветку, удалив их коммиты, и создаст рассинхронизацию в их локальных репозиториях. Это приведёт к потере данных и конфликтам. В общих ветках история должна только дополняться (через merge или rebase с осторожностью), но не переписываться.
