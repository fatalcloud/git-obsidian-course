Этот урок построен как пошаговое руководство, которое проведет вас от установки Git до создания ваших первых алиасов, независимо от того, работаете ли вы на Windows, macOS или Linux.

### Шаг 1: Проверка наличия и установка Git

Прежде чем что-то настраивать, нужно убедиться, что Git установлен в вашей системе. Откройте **терминал** (на macOS или Linux) или **Git Bash** (на Windows) и выполните команду:

```bash
git --version
```

Эта команда — ваш главный «детектор» Git. Если программа установлена, вы увидите что-то вроде `git version 2.43.0`. Если же Git не найден, следуйте инструкциям ниже для вашей операционной системы.

#### Windows

1.  Перейдите на официальный сайт Git: [git-scm.com](https://git-scm.com/).
2.  Скачайте установщик для Windows (обычно это 64-битная версия).
3.  Запустите загруженный файл и следуйте инструкциям мастера установки. **Важно:** На одном из этапов убедитесь, что выбраны опции:
    *   `Use Git Bash as default shell` — для комфортной работы с командами.
    *   `Integrate Git with the Windows Shell` — чтобы Git был доступен из командной строки.
4.  Остальные настройки можно оставить по умолчанию. После завершения установки перезапустите Git Bash.

Альтернативный способ для опытных пользователей Windows — установка через менеджер пакетов `winget`:

```bash
winget install --id Git.Git -e --source winget
```

#### macOS

Самый удобный способ — использовать менеджер пакетов Homebrew.

1.  Установите Homebrew, если он еще не установлен. Для этого в терминале выполните команду с официального сайта [brew.sh](https://brew.sh/):
    ```bash
    /bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"
    ```
2.  После установки Homebrew, установите Git:
    ```bash
    brew install git
    ```
3.  Альтернативный способ — установка через Xcode Command Line Tools, которые включают Git. Выполните в терминале:
    ```bash
    xcode-select --install
    ```
    После этого следуйте инструкциям на экране.

#### Linux

На Linux установка Git выполняется через стандартный менеджер пакетов вашего дистрибутива. Откройте терминал и выполните соответствующую команду:

*   **Debian/Ubuntu:**
    ```bash
    sudo apt update
    sudo apt install git
    ```
*   **Fedora/CentOS/RHEL:**
    ```bash
    sudo dnf install git  # Для новых версий Fedora
    # или
    sudo yum install git  # Для старых версий и CentOS
    ```
*   **OpenSUSE:**
    ```bash
    sudo zypper install git
    ```
*   **Arch Linux:**
    ```bash
    sudo pacman -S git
    ```

После установки любым из способов, снова выполните `git --version`, чтобы убедиться, что всё прошло успешно.

### Шаг 2: Первое знакомство с `git config`

Теперь, когда Git установлен, нам нужно его настроить. Вся настройка в Git выполняется с помощью одной главной команды — `git config`. Прежде чем мы начнем, важно понять, что у Git есть **три уровня конфигурации**, и они имеют разный приоритет:

1.  **System (`--system`)**: Настройки, применяемые ко всем пользователям на данном компьютере. Хранятся в `/etc/gitconfig` (на Linux/macOS) или в `C:\ProgramData\Git\config` (на Windows). Изменяются редко и обычно администратором.
2.  **Global (`--global`)**: Настройки для конкретного пользователя операционной системы. Применяются ко **всем** вашим Git-проектам на этом компьютере. Это самый часто используемый уровень. Хранится в `~/.gitconfig` (на Linux/macOS) или `C:\Users\<имя_пользователя>\.gitconfig` (на Windows).
3.  **Local (`--local`)**: Настройки, специфичные для одного конкретного репозитория. Хранятся в папке `.git/config` внутри вашего проекта. Этот уровень имеет **наивысший приоритет**: если настройка задана на локальном уровне, она переопределит глобальную и системную.

**Золотое правило**: Начинайте с `--global`. Если для какого-то проекта нужны особые настройки (например, другая рабочая почта), переопределяйте их на уровне `--local` уже внутри этого проекта.

### Шаг 3: Настройка имени и email

Это самые важные настройки. Git добавляет эту информацию к каждому создаваемому вами коммиту, чтобы другие разработчики знали, кто и когда внес изменения.

Выполните следующие команды в терминале, заменив `"Ваше Имя"` и `"ваша.почта@example.com"` на свои данные:

```bash
git config --global user.name "Ваше Имя"
git config --global user.email "ваша.почта@example.com"
```

**Важно:** Рекомендуется использовать тот же адрес электронной почты, который вы используете для своего аккаунта на GitHub, GitLab или другом хостинге репозиториев.

### Шаг 4: Выбор текстового редактора по умолчанию

В процессе работы Git может попросить вас ввести сообщение для коммита или разрешить конфликт слияния, и для этого он откроет текстовый редактор. По умолчанию Git часто использует редактор **Vim**. Если вы с ним не знакомы, выход из него может стать квестом. Поэтому лучше сразу настроить понятный вам редактор.

Вот примеры настройки популярных редакторов:

*   **Nano** (простой консольный редактор):
    ```bash
    git config --global core.editor "nano -w"
    ```
*   **VS Code** (один из самых популярных):
    ```bash
    git config --global core.editor "code --wait"
    ```
*   **Sublime Text** (на macOS):
    ```bash
    git config --global core.editor "subl -n -w"
    ```
*   **Vim** (если вы решили его освоить):
    ```bash
    git config --global core.editor "vim"
    ```

Убедитесь, что выбранный редактор установлен в вашей системе, и его исполняемый файл доступен из командной строки.

### Шаг 5: Настройка обработки окончаний строк (`core.autocrlf`)

Это критически важная настройка для тех, кто работает в команде, где используются разные операционные системы. Дело в том, что Windows, macOS и Linux по-разному обозначают конец строки в текстовых файлах. Windows использует символы `CRLF` (`\r\n`), а macOS и Linux — только `LF` (`\n`). Без настройки это может привести к тому, что Git будет показывать изменения там, где их нет, или создавать ненужные конфликты.

*   **Если вы работаете на Windows:**
    ```bash
    git config --global core.autocrlf true
    ```
    Эта настройка говорит Git: при сохранении в репозиторий (коммите) преобразовывай `CRLF` в `LF`, а при извлечении из репозитория — обратно из `LF` в `CRLF`.

*   **Если вы работаете на macOS или Linux:**
    ```bash
    git config --global core.autocrlf input
    ```
    Эта настройка говорит Git: при сохранении в репозиторий преобразовывай `CRLF` в `LF`, если они вдруг встретятся. При извлечении файлов никаких преобразований не делай (так как `LF` — это "родной" формат для этих ОС).

**Примечание для опытных пользователей:** Самый надежный способ управлять окончаниями строк — создать файл `.gitattributes` в корне вашего репозитория и закоммитить его. Тогда все участники проекта будут использовать единые правила, независимо от их локальных настроек `core.autocrlf`.

### Шаг 6: Создание полезных алиасов

Алиасы (псевдонимы) — это ваши личные сокращения для часто используемых команд Git. Они помогают печатать меньше, а работать быстрее.

Вот несколько базовых алиасов, с которых стоит начать:

```bash
# Сокращение для 'git status' — самой частой команды
git config --global alias.st status

# Сокращение для 'git checkout' — переключения между ветками
git config --global alias.co checkout

# Сокращение для 'git branch' — просмотра и создания веток
git config --global alias.br branch

# Сокращение для 'git commit' — создания коммита
git config --global alias.ci commit
```

После создания этих алиасов вместо `git status` можно будет писать просто `git st`, а вместо `git checkout` — `git co`.

### Шаг 7: Просмотр текущей конфигурации

После того как вы всё настроили, полезно проверить, что именно записал Git.

*   **Просмотр ВСЕХ настроек** (системных, глобальных и локальных) в том виде, в котором их видит Git в данный момент:
    ```bash
    git config --list
    ```
    Вы увидите длинный список пар `ключ=значение`. Обратите внимание, что некоторые настройки могут повторяться, но Git будет использовать ту, что имеет более высокий приоритет.

*   **Просмотр только ГЛОБАЛЬНЫХ настроек** (ваши личные):
    ```bash
    git config --global --list
    ```
    Это лучший способ проверить, правильно ли вы указали имя, почту и редактор.

*   **Просмотр только ЛОКАЛЬНЫХ настроек** (для текущего проекта):
    ```bash
    git config --local --list
    ```
    Эта команда сработает, только если вы находитесь внутри Git-репозитория.

*   **Просмотр конкретной настройки**:
    ```bash
    git config user.name
    # Выведет: Ваше Имя
    ```

    *   Проверьте работу алиаса: наберите `git st`. Если вы не в репозитории, вы увидите сообщение об ошибке, но команда должна "поняться" как `git status`.
