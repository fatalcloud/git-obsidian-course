Добрый день, коллеги. Сегодня мы разберём две команды, которые отвечают за перемещение и удаление файлов в Git: **`git mv`** и **`git rm`**. На первый взгляд может показаться: зачем учить специальные команды, если можно просто перетащить файл мышкой в проводнике или нажать Delete? Однако в экосистеме Git прямой путь не всегда самый короткий. Неправильное удаление или переименование может привести к путанице в индексе и даже потере данных. Давайте разберёмся, как делать это правильно.

### Как Git видит переименование и удаление

Для Git файл — это не имя, а **содержимое** (точнее, хеш содержимого). Git не хранит в репозитории явной записи «файл А переименован в файл Б». Вместо этого, когда вы делаете коммит, Git анализирует изменения и пытается **эвристически определить переименование**. Если вы удалили `old.js` и добавили `new.js` с очень похожим (или идентичным) содержимым, Git с высокой вероятностью скажет при просмотре истории: «Этот файл был переименован».

Но для того чтобы Git мог корректно отследить эту связь, **индекс должен быть проинструктирован правильно**. Вот здесь и вступают в игру `git mv` и `git rm`.

### `git mv` — Переименование и перемещение файлов

Команда `git mv` — это **удобная комбинация** трёх действий:
1.  Перемещение/переименование файла в файловой системе (`mv old new`).
2.  Удаление старого имени из индекса (`git rm --cached old`).
3.  Добавление нового имени в индекс (`git add new`).

**Синтаксис:**
```bash
git mv <старое_имя> <новое_имя>
```

**Что произойдёт, если просто переименовать файл через ОС?**
Представим, вы в проводнике переименовали `old.js` в `new.js`. Выполнив `git status`, вы увидите:
```
Changes not staged for commit:
  deleted:    old.js

Untracked files:
  new.js
```
Git видит удаление старого файла и появление нового неотслеживаемого. Чтобы Git понял, что это переименование, вам придётся вручную выполнить `git add new.js` и (если старый файл был в индексе) `git rm old.js`. После этого `git status` покажет:
```
Changes to be committed:
  renamed:    old.js -> new.js
```

Использование `git mv` делает это **одной командой** и сразу подготавливает изменение для коммита.

**Пример: Переименование `legacy_module.js` в `new_module.js`**
```bash
# Создадим файл для примера
echo "console.log('старый код');" > legacy_module.js
git add legacy_module.js
git commit -m "Добавлен легаси модуль"

# Переименовываем правильно
git mv legacy_module.js new_module.js
git status
# Вывод: renamed: legacy_module.js -> new_module.js

git commit -m "Переименован модуль в new_module"
```

**Важный нюанс:** Если вы вносили правки в файл и не сохранили их (не сделали коммит или хотя бы `git add`), а затем выполнили `git mv`, несохранённые изменения **останутся в новом файле**. Но будьте осторожны: если файл изменён, а вы делаете `git mv`, а затем `git checkout -- old.js`, можно потерять часть истории. Лучше фиксировать изменения до переименования.

### `git rm` — Удаление файлов

Команда `git rm` удаляет файл из индекса и, опционально, из рабочего каталога. У неё есть два основных режима.

#### 1. `git rm <файл>` — Полное удаление

Эта команда **удаляет файл и из рабочей папки, и из индекса**. После выполнения файла физически не будет на диске, и Git подготовит это удаление к следующему коммиту.

**Пример: Удаление ненужного временного файла**
```bash
# Создаём файл, который потом решим удалить
echo "временные данные" > temp.txt
git add temp.txt
git commit -m "Добавлен временный файл"

# Удаляем его полностью
git rm temp.txt
# Файл temp.txt исчез из папки проекта

git status
# Changes to be committed:
#   deleted:    temp.txt

git commit -m "Удалён временный файл"
```

**Предупреждение:** Если у вас есть **незакоммиченные изменения** в файле, Git откажется удалять его командой `git rm`, чтобы вы случайно не потеряли работу. Вы увидите ошибку:
```
error: the following file has local modifications:
    temp.txt
```
Чтобы принудительно удалить файл вместе с несохранёнными правками, используйте флаг `-f`:
```bash
git rm -f temp.txt
```
**Делайте это только если абсолютно уверены, что правки не нужны.**

#### 2. `git rm --cached <файл>` — Удаление только из индекса

Этот режим удаляет файл из-под контроля версий, **но оставляет его в рабочей папке**. Файл становится **Untracked**. Это идеальный вариант, когда вы случайно закоммитили `config.yml` с паролями и хотите прекратить его отслеживание, но сохранить локально для работы.

**Пример: Убираем файл из репозитория, оставляя локально**
```bash
# Предположим, config.yml уже отслеживается
git rm --cached config.yml

# Добавляем его в .gitignore, чтобы больше не попадал
echo "config.yml" >> .gitignore

git status
# Changes to be committed:
#   deleted:    config.yml
#
# Untracked files:
#   config.yml

git add .gitignore
git commit -m "Убрал config.yml из репозитория, добавил в игнор"
```
После этого файл `config.yml` останется на вашем диске, но Git перестанет за ним следить.

### Частые ошибки и как их избежать

1.  **Случайное удаление файла через ОС с последующим `git add .`**
    *   **Симптом:** Файл пропал, Git показывает `deleted: file.txt` в staged changes.
    *   **Проблема:** Вы не хотели удалять его навсегда, а просто перенесли в другую папку без `git mv`.
    *   **Решение до коммита:** `git restore --staged file.txt` (отменить удаление из индекса), затем восстановить файл из последнего коммита: `git restore file.txt`. Если нужно было переместить, используйте `git mv`.

2.  **Использование `git rm` на файле с несохранёнными изменениями**
    *   **Проблема:** Git блокирует удаление, но если вы применили `-f`, изменения потеряны навсегда.
    *   **Рекомендация:** Всегда проверяйте `git status` перед удалением. Если файл изменён, либо сделайте коммит, либо спрячьте изменения (`git stash`).

3.  **Забыли добавить файл после ручного переименования**
    *   **Симптом:** `git status` показывает `deleted` и `untracked file`.
    *   **Как исправить:** Вручную выполните `git add <новое_имя>` и `git rm <старое_имя>`, либо (если файл ещё не был закоммичен) просто используйте `git add -A`.

### Практическое задание: Сравниваем способы переименования

Это задание поможет вам на практике почувствовать разницу между «ручным» переименованием и использованием `git mv`.

1.  Создайте новую папку и инициализируйте репозиторий:
    ```bash
    mkdir rename_demo && cd rename_demo
    git init
    ```

2.  Создайте два файла:
    ```bash
    echo "file A" > a.txt
    echo "file B" > b.txt
    git add a.txt b.txt
    git commit -m "Initial commit"
    ```

3.  **Способ 1: Использование `git mv`**
    ```bash
    git mv a.txt aa.txt
    git status
    # Вы должны увидеть: renamed: a.txt -> aa.txt
    ```

4.  **Способ 2: Ручное переименование через ОС**
    *   Переименуйте `b.txt` в `bb.txt` через проводник или командой `mv b.txt bb.txt` (без git).
    *   Выполните `git status`.
    *   **Вопрос:** Что показывает статус? (Увидите `deleted: b.txt` и `Untracked: bb.txt`).
    *   Теперь выполните команду, которая исправит ситуацию: `git add -u` (она добавит в индекс все изменения для отслеживаемых файлов, включая удаления).
    *   Снова `git status`. Теперь должно быть `renamed: b.txt -> bb.txt`.

5.  **Закоммитьте оба переименования:**
    ```bash
    git commit -m "Renamed files"
    ```

6.  **Сравните историю:**
    ```bash
    git log --oneline --name-status
    ```
    Вы увидите, что оба переименования зафиксированы одинаково — как `R100` (100% similarity).

### Вопросы для самопроверки

**1. Что делает команда `git rm` с файлом в рабочей папке?**
> **Ответ:** Команда `git rm` (без флагов) **удаляет файл физически с диска** (из рабочего каталога) и одновременно удаляет его из индекса, подготавливая удаление к коммиту. Если файл был изменён и не закоммичен, Git выдаст ошибку, защищая данные.

**2. Как отменить `git rm --cached` до того, как был сделан коммит?**
> **Ответ:** Используйте команду `git restore --staged <имя_файла>`. Она вернёт файл в индекс (восстановит его отслеживаемое состояние) из последнего коммита. Сам файл в рабочей папке при этом не изменится (он и так там остался после `--cached`).

**3. (Дополнительно) В чём разница между `git rm --cached` и простым добавлением файла в `.gitignore`?**
> **Ответ:** `git rm --cached` удаляет файл из индекса и **прекращает его отслеживание** для будущих коммитов. Добавление в `.gitignore` только указывает Git игнорировать **неотслеживаемые** файлы с таким именем. Если файл уже отслеживается, одного `.gitignore` недостаточно — Git продолжит следить за изменениями. Нужно обязательно выполнить `git rm --cached`, чтобы «забыть» файл.

**4. (Со звёздочкой) Вы случайно удалили важный файл через `git rm` и даже сделали коммит. Можно ли его восстановить?**
> **Ответ:** Да, Git хранит всю историю. Вы можете восстановить файл из предыдущего коммита командой `git restore --source=HEAD~1 <имя_файла>` (где `HEAD~1` — коммит перед удалением). Или использовать `git revert` для отмены коммита с удалением. Данные не потеряны, пока не перезаписана история принудительно (`git gc` не собрал мусор).
