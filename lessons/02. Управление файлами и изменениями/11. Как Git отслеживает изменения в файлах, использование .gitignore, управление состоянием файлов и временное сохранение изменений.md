### 1. Как Git отслеживает изменения: снимки, хэши и три состояния

В отличие от многих старых систем контроля версий, которые хранят только **разницу между файлами** (дельты), Git делает **полный снимок (snapshot)** состояния всего проекта в момент каждого коммита. Если файл не менялся, Git не копирует его заново, а создаёт ссылку на предыдущую версию. Это делает навигацию по истории быстрой и надёжной.

Каждый снимок идентифицируется уникальным **хешем SHA-1** (например, `c3b5e2a`). Хеш вычисляется на основе содержимого файлов и метаданных, что гарантирует: если что-то изменилось — хеш тоже изменится. Это позволяет Git мгновенно определять, совпадает ли текущее состояние файла с сохранённым.

Ключевое понятие, которое мы изучали — **три состояния файла**:
- **Изменён (Modified)** — файл в рабочем каталоге отличается от версии в репозитории, но ещё не добавлен в индекс.
- **Индексирован (Staged)** — файл помечен для включения в следующий коммит.
- **Зафиксирован (Committed)** — файл сохранён в локальной базе данных Git (папка `.git`).

Понимание этих трёх состояний — фундамент для осмысленного выполнения `git add`, `git commit` и `git restore`.

### 2. `.gitignore`: паттерны и применение

Файл **.gitignore** — это ваш главный инструмент для поддержания чистоты репозитория. Он указывает Git, какие файлы и папки следует **намеренно игнорировать**.

**Основные правила синтаксиса:**
- `*.log` — игнорировать все файлы с расширением `.log` в любой папке.
- `build/` — игнорировать папку `build` и всё её содержимое.
- `!important.log` — отменить игнорирование для конкретного файла.
- `**/temp` — игнорировать папку `temp` на любом уровне вложенности.

**Типичные сценарии использования:**
- Файлы скомпилированного кода (`*.class`, `*.pyc`).
- Папки с зависимостями (`node_modules/`, `venv/`).
- Локальные конфигурации и секреты (`.env`, `config.local.yml`).
- Файлы операционной системы (`.DS_Store`, `Thumbs.db`).

**Важное предостережение:** `.gitignore` действует **только на неотслеживаемые файлы**. Если файл уже попал под версионный контроль, сначала нужно прекратить его отслеживание командой `git rm --cached <файл>`.

### 3. `git stash`: временное убежище для незавершённой работы

**Stash** (тайник) позволяет временно «спрятать» изменения из рабочего каталога и индекса, чтобы быстро переключиться на другую задачу, не создавая незавершённый коммит.

**Ключевые команды:**
- `git stash push -m "описание"` — сохранить изменения в тайник.
- `git stash list` — посмотреть все сохранённые тайники.
- `git stash apply` — применить последний тайник, оставив его в списке.
- `git stash pop` — применить и удалить тайник.
- `git stash drop` — удалить тайник без применения.
- `git stash clear` — удалить все тайники.

**Когда использовать?**
- Вам нужно срочно переключиться на другую ветку для исправления бага.
- Вы хотите «прибраться» в рабочем каталоге перед `git pull`.
- Вы экспериментируете с кодом и хотите иметь возможность быстро откатиться к сохранённому состоянию.

**Важный нюанс:** По умолчанию `git stash` **не сохраняет неотслеживаемые файлы**. Чтобы включить их, используйте флаг `-u` (или `--include-untracked`).

### Сравнительная таблица состояний файла в Git

| Состояние | Где находится файл? | Видит ли его Git? | Попадёт в следующий коммит? | Как туда попасть? |
| :--- | :--- | :--- | :--- | :--- |
| **Untracked** | Рабочий каталог | Видит, но не следит | Нет | Создать новый файл или `git rm --cached` для отслеживаемого |
| **Modified** | Рабочий каталог | Видит как изменённый | Нет | Изменить отслеживаемый файл |
| **Staged** | Индекс | Видит как подготовленный | **Да** | `git add <файл>` |
| **Stashed** | Тайник (`.git/refs/stash`) | Спрятан из рабочего каталога | Нет (он вне коммитов) | `git stash push` |
| **Committed** | Репозиторий (`.git/objects`) | Хранит в истории | Уже в истории | `git commit` |

### Большой практический кейс: Управление пятью файлами

Представьте, что вы работаете над веб-проектом. В вашем рабочем каталоге находятся пять файлов, с каждым из которых нужно поступить по-разному. Ваша задача — выполнить все действия в правильной последовательности, используя только команды Git.

**Исходная ситуация:**
- Вы находитесь в ветке `feature/update-profile`.
- Файлы в проекте:
    1.  `secret.keys` — содержит реальные ключи API. Его нужно **игнорировать** (сейчас он уже отслеживается Git по ошибке).
    2.  `unfinished.js` — вы начали писать новую функцию, но пришла срочная задача. Его нужно **временно спрятать** в stash.
    3.  `styles.css` — вы поправили отступы и хотите включить эти правки в **следующий коммит**.
    4.  `experiment.html` — вы испортили его тестовыми правками и хотите **отменить все изменения**, вернув к последней сохранённой версии.
    5.  `old_name.txt` — его нужно **переименовать** в `new_name.txt`.

**Требуется:** написать последовательность команд Git, которая решит все пять задач. После выполнения коммит должен содержать изменения только из `styles.css` и переименование файла, а все остальные файлы должны оказаться в нужных состояниях.

**Пример решения (эталонная последовательность):**

```bash
# 1. Игнорируем secret.keys, который уже отслеживается
echo "secret.keys" >> .gitignore
git rm --cached secret.keys
git add .gitignore
# Теперь файл удалён из индекса, но остался локально, и Git его игнорирует

# 2. Прячем незавершённый unfinished.js в stash
git stash push -m "WIP: незаконченная функция" -- unfinished.js
# или просто git stash push -m "..." и добавить только этот файл

# 3. Добавляем правки в styles.css в индекс
git add styles.css

# 4. Отменяем изменения в experiment.html до последнего коммита
git restore experiment.html

# 5. Переименовываем old_name.txt в new_name.txt
git mv old_name.txt new_name.txt

# 6. Проверяем статус перед коммитом
git status

# 7. Делаем коммит (в него войдут styles.css и переименование)
git commit -m "Обновлены стили и переименован файл"
```

После этого можно продолжить работу: переключиться на другую ветку для срочной задачи, а затем вернуться и применить stash: `git stash pop`.
