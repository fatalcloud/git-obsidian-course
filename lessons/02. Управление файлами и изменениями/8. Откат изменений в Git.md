За десять лет практики я вытащил из цифрового небытия сотни файлов, распутал тысячи конфликтов и отменил столько неудачных коммитов, что хватило бы на небольшую библиотеку.

Главное, что вы должны запомнить прежде, чем мы начнём: **в Git почти всё можно отменить, если знать нужную команду и действовать хладнокровно**. Но есть один красный рубильник, который стирает данные безвозвратно — и я покажу, где он находится, чтобы вы случайно на него не нажали.

### Три фронта отмены: Рабочий каталог, Индекс, Репозиторий

Помните три состояния файлов? Отмена изменений происходит на одном из этих трёх уровней. И для каждого уровня есть свой инструмент.

**1. Отмена неиндексированных изменений (Рабочий каталог)**
Вы изменили файл, но ещё не делали `git add`. Внезапно вы понимаете, что эксперимент не удался, и проще начать с чистого листа — вернуть файл к тому состоянию, которое было в последнем коммите.

**Старая школа (команда `git checkout`):**
```bash
git checkout -- <имя_файла>
```
Два дефиса перед именем файла говорят Git: «Я точно имею в виду файл, а не ветку».

**Новая школа (команда `git restore`, рекомендуется с версии 2.23):**
```bash
git restore <имя_файла>
```

**Что произойдёт:** Git возьмёт версию файла из индекса (если файл был в индексе) или из последнего коммита и затрёт ею ваш рабочий файл. **Все несохранённые правки будут потеряны безвозвратно.**

**2. Отмена индексации (Staging Area / Индекс)**
Вы сделали `git add`, добавили файл в индекс, но передумали включать его в следующий коммит. Вам нужно убрать файл из индекса, но оставить изменения в рабочем каталоге (чтобы доработать позже).

**Старая школа:**
```bash
git reset HEAD <имя_файла>
```

**Новая школа:**
```bash
git restore --staged <имя_файла>
```

После этой команды файл перестанет быть «зелёным» в `git status` и вернётся в состояние «изменён, но не в индексе» (красный).

**3. Отмена последнего коммита (Репозиторий)**
Вы сделали `git commit`, и только потом заметили, что забыли добавить важный файл, или написали сообщение с опечаткой. Или — упаси боже — закоммитили пароль в открытом виде. Здесь начинается зона повышенной ответственности.

### `git reset` — Машина времени для локальной истории

Команда `git reset` перемещает указатель текущей ветки (`HEAD`) на указанный коммит. В зависимости от флагов, она по-разному обращается с индексом и рабочим каталогом. Представьте, что вы пролистываете ленту времени назад.

#### `git reset --soft HEAD~1` — Мягкий откат

Это самый безопасный способ «отменить» коммит, **не потеряв ни строчки кода**.

```bash
git reset --soft HEAD~1
```

**Что происходит:**
1.  Указатель ветки (`HEAD`) перемещается на один коммит назад.
2.  **Индекс** остаётся точно таким же, каким он был на момент отменённого коммита (все файлы, которые были в том коммите, теперь снова в индексе — зелёные).
3.  **Рабочий каталог** не трогается вообще.

**Сценарий использования:** Вы сделали коммит, но забыли добавить ещё один файл. Делаете `git reset --soft HEAD~1`, затем `git add забытый_файл.txt` и делаете новый коммит с правильным составом. В истории это будет выглядеть как один нормальный коммит.

#### `git reset --mixed HEAD~1` (по умолчанию)

Если вы опустите флаг, `git reset` работает в режиме `--mixed`.

```bash
git reset HEAD~1
# или
git reset --mixed HEAD~1
```

**Что происходит:**
1.  Указатель ветки перемещается на один коммит назад.
2.  **Индекс очищается** (становится как в том, предыдущем коммите).
3.  **Рабочий каталог остаётся нетронутым.**

**Результат:** Все файлы из отменённого коммита переходят в статус «изменённые, но не добавленные в индекс» (красные). Это удобно, если вы хотите не просто отменить коммит, но и полностью пересобрать его заново, разбив на несколько маленьких.

#### `git reset --hard HEAD~1` — ⚠️ Красная кнопка!

**Внимание! Это команда, которая может безвозвратно уничтожить ваши несохранённые данные.**

```bash
git reset --hard HEAD~1
```

**Что происходит:**
1.  Указатель ветки перемещается на один коммит назад.
2.  **Индекс** синхронизируется с этим предыдущим коммитом.
3.  **Рабочий каталог** **принудительно заменяется** состоянием предыдущего коммита.

**Результат:** Все изменения, сделанные в отменённом коммите, а также **вообще все незакоммиченные правки в рабочем каталоге**, будут **стёрты**. Восстановить их можно будет только с помощью сложных манипуляций с `git reflog` (и то не всегда).

**Когда использовать:** Только когда вы точно знаете, что последний коммит — полная ерунда, и вы хотите вернуть всё как было час назад, **при этом в рабочем каталоге нет ничего ценного**. Я рекомендую новичкам перед `--hard` всегда делать `git stash`, чтобы подстраховаться.

### `git revert` — Безопасная отмена для командной работы

Если `git reset` переписывает историю (перемещает указатель ветки назад), то `git revert` создаёт **новый коммит**, который отменяет изменения, внесённые указанным коммитом.

```bash
git revert <хеш_коммита>
# или для последнего коммита
git revert HEAD
```

**Почему это важно для публичных веток?**
Представьте, что вы запушили коммит `A` на GitHub, и ваши коллеги уже скачали его себе. Если вы сделаете `git reset --hard HEAD~1` и принудительно запушите (`git push --force`), вы **сломаете** репозитории коллег. Их локальная история будет ссылаться на коммит, которого больше нет на сервере.

`git revert` не удаляет старый коммит. Он добавляет **новый**, который говорит: «Сделать всё наоборот». Это полностью безопасно для синхронизации: коллеги просто получат ещё один коммит сверху.

**Пример:**
1.  В коммите `abc123` вы случайно удалили важную функцию.
2.  Выполняете: `git revert abc123`
3.  Git открывает редактор для сообщения коммита (по умолчанию "Revert 'Удалил функцию...'").
4.  После сохранения появляется новый коммит `def456`, в котором удалённая функция возвращена.

### Таблица спасения: Ситуация → Команда

Держите эту таблицу под рукой. Она сэкономит вам часы гугления.

| Ситуация | Что делать | Команда (современный вариант) |
| :--- | :--- | :--- |
| Испортил файл, хочу вернуть как было в последнем коммите | Откат неиндексированных изменений | `git restore <file>` |
| Случайно добавил файл в индекс (`git add`) | Убрать из индекса, оставить изменения | `git restore --staged <file>` |
| Закоммитил, но забыл добавить файл / поправить сообщение | Отменить коммит, сохранив всё в индексе | `git reset --soft HEAD~1` |
| Закоммитил, хочу разобрать коммит на части | Отменить коммит, оставить изменения в рабочем каталоге | `git reset HEAD~1` |
| Закоммитил полную чушь, нужно вернуть всё назад и забыть | **Осторожно!** Полный откат с удалением правок | `git reset --hard HEAD~1` |
| Отменить коммит, который уже улетел на GitHub/GitLab | Безопасная отмена для публичной истории | `git revert HEAD` |

### Практическое задание: Тренируемся на кошках

Создадим учебный полигон, чтобы безопасно освоить `reset` и `restore`.

1.  **Создайте папку и репозиторий:**
    ```bash
    mkdir git-undo-lab
    cd git-undo-lab
    git init
    ```

2.  **Создайте файл и сделайте первый коммит:**
    ```bash
    echo "Строка 1" > important.txt
    git add important.txt
    git commit -m "Первый коммит"
    ```

3.  **Имитируем работу и ошибку:**
    ```bash
    # Добавим хорошую правку
    echo "Строка 2" >> important.txt
    git add important.txt
    git commit -m "Добавил строку 2"

    # Теперь "испортим" файл (например, напишем бред в рабочем каталоге)
    echo "АБВГДЕЙКА" >> important.txt
    git status
    # Вы увидите, что файл изменён (красный)
    ```

4.  **Откатываем неиндексированные изменения:**
    ```bash
    cat important.txt  # Убедитесь, что там есть "АБВГДЕЙКА"
    git restore important.txt
    cat important.txt  # "АБВГДЕЙКА" исчезла, файл вернулся к состоянию последнего коммита
    ```

5.  **Создадим ещё один коммит, чтобы было что отменять:**
    ```bash
    echo "Строка 3" >> important.txt
    git add important.txt
    git commit -m "Добавил строку 3"
    git log --oneline  # Посмотрите три коммита
    ```

6.  **Отменяем последний коммит через `git reset --soft`:**
    ```bash
    git reset --soft HEAD~1
    git status
    # Вы увидите, что important.txt находится в индексе (зелёный), 
    # а коммита "Добавил строку 3" больше нет в логе.
    git log --oneline  # Осталось два коммита
    ```

7.  **Проверьте содержимое файла:**
    ```bash
    cat important.txt
    # Строка 3 всё ещё там! Она просто ждёт нового коммита в индексе.
    ```

### Вопросы для самопроверки

**1. Чем отличается `git restore` от `git reset`?**
> **Ответ:** `git restore` работает на уровне **файлов** и предназначена для восстановления состояния рабочего каталога или индекса из коммита. `git reset` работает на уровне **коммитов** (перемещает `HEAD`) и затрагивает всю ветку целиком. Грубо говоря, `restore` — это ластик для файла, `reset` — машина времени для всего проекта.

**2. Зачем нужен `git revert`, если можно использовать `git reset --hard` и принудительный пуш?**
> **Ответ:** `git reset --hard` + `push --force` **переписывает историю**. Если репозиторий публичный и у него есть клоны у других разработчиков, это вызовет рассинхронизацию и большие проблемы при слиянии. `git revert` создаёт новый коммит, отменяющий изменения, и сохраняет историю линейной и безопасной для всех участников. В командной работе `revert` — золотой стандарт, `reset --hard` для публичных веток — моветон.

**3. (Дополнительно) Что выведет команда `git status` после выполнения `git reset --soft HEAD~1`?**
> **Ответ:** Она покажет файлы из отменённого коммита в секции **"Changes to be committed:"** (зелёные). Они уже в индексе и готовы к новому коммиту.

**4. (Со звёздочкой) Как отменить `git reset --hard HEAD~1`, если вы поняли, что стёрли нужное?**
> **Ответ:** Пока не собран мусор (`git gc`), можно использовать `git reflog`. Найдите хеш того коммита, на который указывал `HEAD` до сброса, и выполните `git reset --hard <найденный_хеш>`. Это вернёт всё на место. Именно поэтому `reflog` называют «спасательным кругом» Git.
