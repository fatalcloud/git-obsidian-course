В своей практике я не раз видел, как в репозиторий по ошибке улетали секретные ключи, гигабайты логов или скомпилированные бинарники. Этот урок — ваша прививка от подобных инцидентов.

### Что такое .gitignore и зачем он нужен?

**.gitignore** — это обычный текстовый файл, который вы кладёте в корень вашего репозитория (или в любую подпапку). Внутри этого файла вы пишете правила — какие файлы и папки Git должен **намеренно игнорировать**.

**Ключевой принцип:** Игнорируются только **неотслеживаемые (Untracked)** файлы. Если файл уже попал под версионный контроль (был закоммичен), просто добавить его в `.gitignore` недостаточно — Git продолжит отслеживать изменения в нём. О том, как это исправить, поговорим в разделе про `git rm --cached`.

### Синтаксис .gitignore: Пишем правила грамотно

Файл `.gitignore` читается построчно. Пустые строки и строки, начинающиеся с `#` (комментарии), игнорируются. Вот основной синтаксис, который нужно знать как таблицу умножения.

| Шаблон | Что игнорирует | Пример совпадения | Пример НЕ совпадения |
| :--- | :--- | :--- | :--- |
| `*.log` | Все файлы с расширением `.log` в любой папке. | `error.log`, `logs/debug.log` | `log.txt`, `log1.backup` |
| `build/` | Папку с именем `build` и всё её содержимое. | `build/output.txt`, `src/build/` | `build` (имя файла без `/` не подходит) |
| `!exception.log` | **Отмена игнорирования.** Исключает файл из правила. | `exception.log` будет отслеживаться, даже если есть `*.log`. | — |
| `[0-9].txt` | Файлы от `0.txt` до `9.txt`. | `5.txt`, `0.txt` | `10.txt`, `a.txt` |
| `**/temp` | Папка `temp` на любом уровне вложенности. | `temp/`, `src/temp/`, `a/b/c/temp/` | — |
| `*.log` + `!important.log` | Игнорировать все `.log`, кроме `important.log`. | `error.log` игнор, `important.log` трекается. | — |

**Пояснение ключевых моментов:**
*   **Слеш в конце (`folder/`)** — указывает, что это именно **директория**.
*   **Звёздочка (`*`)** — заменяет любое количество символов (кроме слеша).
*   **Две звёздочки (`**`)** — заменяют любое количество символов, включая слеш (удобно для вложенных папок).
*   **Восклицательный знак (`!`)** — отрицание предыдущего правила. **Важно:** Порядок правил имеет значение. Если сначала игнорировать всё `*.log`, а потом сделать исключение для `!access.log`, то `access.log` будет отслеживаться.

### Как игнорировать файл, который уже попал в репозиторий?

Это, пожалуй, самая частая проблема, с которой ко мне приходят разработчики: *"Я добавил секретный ключ в `.gitignore`, но Git всё равно показывает, что файл изменён!"*.

Всё верно. Git продолжает отслеживать файл, потому что тот уже находится под версионным контролем. Чтобы **прекратить отслеживание, но оставить файл локально**, используйте команду:

```bash
git rm --cached <имя_файла>
```

**Что делает эта команда?**
*   Удаляет файл из **индекса** Git (staging area).
*   **Не удаляет** сам файл с вашего жёсткого диска (рабочего каталога).
*   В следующем коммите Git зафиксирует удаление файла из репозитория. После этого добавление имени файла в `.gitignore` начнёт работать как положено.

**Пример: Убираем случайно закоммиченный `config/database.yml`**

1.  Добавьте `config/database.yml` в `.gitignore`.
2.  Выполните:
    ```bash
    git rm --cached config/database.yml
    ```
3.  Сделайте коммит, фиксирующий удаление:
    ```bash
    git commit -m "Удалён отслеживаемый конфиг, добавлен в .gitignore"
    ```

**Важно:** Если вы работаете в команде, все ваши коллеги после `git pull` получат удаление этого файла у себя. Убедитесь, что файл действительно не нужен в репозитории (например, вместо него в репозитории должен лежать шаблон `database.yml.example`).

### Глобальный .gitignore и локальный exclude

Бывают файлы, которые специфичны только для вашей операционной системы или вашего любимого редактора, и добавлять их в `.gitignore` каждого проекта неудобно. На этот случай есть два механизма.

**1. Глобальный .gitignore (для всех репозиториев)**
Вы можете создать файл, например, `~/.gitignore_global`, и указать Git использовать его глобально. Это удобно для:
*   Файлов macOS: `.DS_Store`
*   Файлов Windows: `Thumbs.db`, `desktop.ini`
*   Файлов вашего редактора: `*.swp` (для Vim)

**Настройка:**
```bash
# Создаём файл
touch ~/.gitignore_global

# Добавляем в него правила, например:
echo ".DS_Store" >> ~/.gitignore_global

# Указываем Git использовать его
git config --global core.excludesfile ~/.gitignore_global
```

**2. Исключения на уровне локального репозитория (`.git/info/exclude`)**
Если вы не хотите, чтобы ваши личные правила игнорирования улетали в общий репозиторий (например, у вас есть тестовая папка `local_tests/`, которая нужна только вам), используйте файл `.git/info/exclude`. Он находится внутри папки `.git` вашего проекта и работает точно так же, как `.gitignore`, но **никогда не попадает в коммит** и не синхронизируется с другими разработчиками.

### Практическое задание: Наводим порядок в проекте

Давайте отработаем полученные знания на практике.

1.  Создайте папку `gitignore_demo` и инициализируйте там Git (`git init`).
2.  Создайте файлы и папки, имитирующие "мусор":
    *   Папку `temp_build/` с файлом `output.o` внутри.
    *   Файл `error.log`.
    *   Файл `important_error.log`.
    *   Файл `app.py` (пустой).
3.  Напишите файл `.gitignore` со следующими правилами:
    *   Игнорировать все файлы с расширением `.log`.
    *   **НЕ** игнорировать `important_error.log`.
    *   Игнорировать папку `temp_build/`.
4.  Выполните `git status`.
    *   **Ожидаемый результат:** Git должен видеть только `.gitignore`, `app.py` и `important_error.log`. Файл `error.log` и папка `temp_build` не должны отображаться в статусе.

### Мини-тест: Игнорировать или нет? (5 примеров)

Проверьте свою интуицию. Напротив каждого пункта напишите для себя "Да" или "Нет", а затем сверьтесь с ответами ниже.

1.  Файл `src/config/secrets.env` с реальными паролями.
2.  Файл `src/config/secrets.env.example` с описанием переменных, но без паролей.
3.  Файл `dist/bundle.js` — результат работы сборщика Webpack.
4.  Папка `node_modules/` с библиотеками, описанными в `package.json`.
5.  Файл `.vscode/settings.json` с настройками цвета темы вашего личного редактора.

**Ответы и обоснование:**
1.  **ДА, игнорировать.** Любые секреты, ключи, пароли в репозитории — угроза безопасности.
2.  **НЕТ, НЕ игнорировать.** Файлы-примеры (`*.example`, `*.sample`, `*.dist`) **должны** быть в репозитории. Они служат документацией для других разработчиков о том, как должен выглядеть конфиг.
3.  **ДА, игнорировать.** Артефакты сборки генерируются автоматически. Их коммит засоряет историю и может вызывать конфликты слияния.
4.  **ДА, игнорировать.** Зависимости восстанавливаются командой `npm install`. Коммитить папку `node_modules` — дурной тон, она весит сотни мегабайт.
5.  **ДА, игнорировать (через глобальный `.gitignore`).** Настройки личного редактора не должны навязываться всей команде. Для общих настроек проекта используйте расширения EditorConfig или явные конфиги в корне (например, `.vscode/extensions.json` для рекомендаций).
