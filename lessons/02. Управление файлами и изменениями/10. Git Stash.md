Приветствую! Сегодня я, как опытный разработчик, расскажу вам о команде, которая не раз спасала мои нервные клетки — **`git stash`**. Это ваш карманный «отложенный ящик» для незавершённой работы. Бывало ли такое: вы увлечённо пишете новую фичу, и тут прилетает срочная задача — исправить критический баг прямо сейчас. Код в беспорядке, половина файлов не готова, коммитить рано, а переключиться на другую ветку нужно немедленно. Вот здесь и вступает в игру `git stash`.

### Что такое `git stash`?

**Stash** (в переводе «тайник», «укрытие») — это механизм Git для **временного сохранения изменений** из рабочего каталога и индекса, чтобы очистить рабочую область и иметь возможность быстро переключить контекст. Это как нажать «паузу» в игре, отложить контроллер в сторону, разобраться с делами, а потом вернуться и продолжить с того же места.

**Что попадает в stash?**
По умолчанию Git сохраняет:
- Все **отслеживаемые изменённые файлы** (modified).
- Все **файлы в индексе** (staged).

**Что НЕ попадает в stash?**
- **Неотслеживаемые файлы** (untracked) и **игнорируемые файлы** (по правилам `.gitignore`). Чтобы сохранить и их, нужны дополнительные флаги: `-u` (или `--include-untracked`) для неотслеживаемых, и `-a` (или `--all`) для всех, включая игнорируемые.

### Основные команды работы со stash

#### 1. Создание stash: `git stash push`

Команда для сохранения изменений в тайник.

```bash
# Просто сохранить всё отслеживаемое изменённое и проиндексированное
git stash

# То же самое, но с понятным описанием (рекомендую всегда так делать)
git stash push -m "WIP: добавил черновую валидацию формы"

# Сохранить вместе с неотслеживаемыми файлами
git stash -u -m "Включая новые файлы"

# Сохранить вообще всё, даже игнорируемые файлы
git stash -a
```

После выполнения команды рабочий каталог и индекс становятся **чистыми**, как будто вы только что сделали `git checkout` последнего коммита.

#### 2. Просмотр списка stash: `git stash list`

Ваш склад может хранить не один «отложенный ящик». Посмотреть все сохранённые stash можно командой:

```bash
git stash list
```

Пример вывода:
```
stash@{0}: On main: WIP: валидация формы
stash@{1}: On feature/payment: сохранение перед переключением
stash@{2}: WIP on experiment: 8a3f2e1 Начало эксперимента
```

Каждый stash идентифицируется по индексу `stash@{n}`, где `n=0` — самый свежий. Stash работает по принципу **стека** (LIFO): последний сохранённый — первый применяется по умолчанию.

#### 3. Применение stash: `git stash apply` и `git stash pop`

Чтобы вернуть изменения из тайника обратно в рабочий каталог, есть две команды.

**`git stash apply`** — применяет изменения из stash, **но не удаляет** запись из списка stash.

```bash
# Применить последний stash (stash@{0})
git stash apply

# Применить конкретный stash по имени
git stash apply stash@{1}
```

**`git stash pop`** — применяет изменения из stash и **сразу удаляет** запись из списка (при успешном применении без конфликтов).

```bash
git stash pop
# Эквивалентно git stash apply && git stash drop stash@{0} (при условии успеха)
```

**Важная опция `--index`**
По умолчанию `apply` и `pop` **не восстанавливают состояние индекса** (staging area). Все файлы, которые были в индексе, после применения окажутся просто изменёнными (modified), но не staged. Чтобы восстановить и индекс тоже, используйте флаг:

```bash
git stash apply --index
# или
git stash pop --index
```

#### 4. Удаление stash: `git stash drop` и `git stash clear`

Если вы уверены, что изменения из stash больше не нужны (например, вы их уже закоммитили в другом месте), тайник можно удалить.

```bash
# Удалить последний stash
git stash drop

# Удалить конкретный stash
git stash drop stash@{2}

# Удалить ВСЕ stash сразу (осторожно!)
git stash clear
```

#### 5. Применение stash к другой ветке

Интересный факт: stash не привязан к ветке, в которой был создан. Вы можете сохранить изменения, работая в `feature/login`, переключиться на `main`, сделать срочный коммит, а потом применить этот же stash уже в ветке `feature/registration`. Git попытается наложить изменения на текущее состояние файлов.

```bash
# Находясь в любой ветке
git checkout other-branch
git stash apply
```

Это работает, потому что stash — это просто набор патчей (изменений), которые можно применить к любому базовому состоянию, если нет конфликтов.

### Конфликты при применении stash

Как и при слиянии веток, применение stash может привести к **конфликтам**. Это происходит, если с момента сохранения stash файлы в рабочем каталоге изменились так, что Git не может автоматически определить, как объединить правки.

**Пример:** Вы сохранили изменения в файле `utils.js`. Потом переключились, сделали коммит, который также меняет `utils.js`. При попытке `git stash pop` Git остановится и сообщит о конфликте.

**Что делать?**
1.  Git пометит конфликтующие файлы стандартными маркерами (`<<<<<<<`, `=======`, `>>>>>>>`).
2.  Разрешите конфликты вручную, отредактировав файлы.
3.  После разрешения выполните `git add <файл>`.
4.  **Важно:** После ручного разрешения конфликтов stash **не удаляется автоматически** (даже при использовании `pop`). Вам нужно будет вручную удалить его: `git stash drop`.

### Реальный сценарий: Срочный баг во время разработки фичи

Это классическая ситуация, где `stash` проявляет себя во всей красе.

**Исходные данные:**
- Вы на ветке `feature/new-dashboard`.
- У вас куча изменений в рабочем каталоге: что-то в индексе, что-то ещё не готово.
- Приходит сообщение: «На проде упала главная страница! Нужно срочно поправить в `main`».

**Пошаговый план действий с Git:**

1.  **Сохраняем текущую работу:**
    ```bash
    git stash push -m "WIP: дашборд на стадии вёрстки"
    ```
    Рабочий каталог становится чистым.

2.  **Переключаемся на ветку `main` и подтягиваем последние изменения:**
    ```bash
    git checkout main
    git pull origin main
    ```

3.  **Создаём ветку для срочного фикса и исправляем баг:**
    ```bash
    git checkout -b hotfix/critical-homepage
    # ... вносим правки в файлы ...
    git add .
    git commit -m "Исправлен баг с 500 ошибкой на главной"
    ```

4.  **Отправляем фикс и сливаем через Pull Request (или напрямую в main).**
    После того как фикс принят, возвращаемся к своей фиче:

5.  **Переключаемся обратно на ветку фичи:**
    ```bash
    git checkout feature/new-dashboard
    ```

6.  **Возвращаем сохранённые изменения:**
    ```bash
    git stash pop
    ```

7.  **Продолжаем работу как ни в чём не бывало.**

Если при применении stash возникли конфликты (например, фикс в `main` затронул те же файлы, что и ваша незаконченная работа), разрешаем их и удаляем stash.

### Практическое задание: Воспроизводим сценарий

1.  **Создайте новый репозиторий и перейдите в него:**
    ```bash
    mkdir stash-demo && cd stash-demo
    git init
    ```

2.  **Создайте файл `index.html` с начальным содержимым и закоммитьте его:**
    ```bash
    echo "<h1>Заголовок</h1>" > index.html
    git add index.html
    git commit -m "Начальный коммит с заголовком"
    ```

3.  **Создайте ветку `feature` и переключитесь на неё:**
    ```bash
    git checkout -b feature/add-text
    ```

4.  **Внесите изменения в `index.html` (имитация работы над фичей):**
    ```bash
    echo "<p>Новый параграф в фиче</p>" >> index.html
    # Проверьте статус: файл изменён, но не закоммичен
    git status
    ```

5.  **Внезапно приходит задача исправить баг в `main`. Сохраните изменения:**
    ```bash
    git stash push -m "Черновик параграфа для фичи"
    git status  # теперь чисто
    ```

6.  **Переключитесь на `main` и создайте хотфикс:**
    ```bash
    git checkout main
    # Внесите "срочную правку" в index.html
    echo "<footer>Экстренное сообщение</footer>" >> index.html
    git add index.html
    git commit -m "Добавлен срочный футер"
    ```

7.  **Вернитесь на ветку `feature` и примените stash:**
    ```bash
    git checkout feature/add-text
    git stash pop
    ```

8.  **Посмотрите содержимое `index.html`. Скорее всего, конфликта не будет, так как правки были в разных частях файла. Проверьте `git status` — изменения из stash вернулись, файл изменён.**

9.  **Убедитесь, что stash больше нет в списке:**
    ```bash
    git stash list   # должно быть пусто
    ```

### Вопросы для самопроверки

**1. Чем принципиально отличается `git stash apply` от `git stash pop`?**
> **Ответ:** `apply` применяет изменения из stash, но **оставляет** запись в списке stash. `pop` применяет изменения и **удаляет** запись при успешном применении без конфликтов.

**2. Можно ли с помощью `git stash` сохранить неотслеживаемые файлы (untracked)? Если да, то как?**
> **Ответ:** Да, с флагом `-u` или `--include-untracked`. Например: `git stash -u -m "Сохранение с новыми файлами"`. Для сохранения вообще всех файлов, включая игнорируемые, используется `-a` или `--all`.

**3. Что произойдёт, если при выполнении `git stash pop` возникнет конфликт? Будет ли stash удалён?**
> **Ответ:** При конфликте Git остановит операцию `pop`, не удаляя stash из списка. Вам нужно будет разрешить конфликты вручную и затем явно удалить stash командой `git stash drop`.

**4. Вы работаете над фичей, сделали `git stash`, переключились на другую ветку, внесли там коммит. Можете ли вы применить этот stash в другой ветке, не той, где он был создан?**
> **Ответ:** Да, stash не привязан к конкретной ветке. Вы можете переключиться на любую ветку и выполнить `git stash apply`. Git попытается наложить сохранённые изменения на текущее состояние файлов.

**5. Как восстановить не только изменения в файлах, но и состояние индекса (то, что было за staged) из stash?**
> **Ответ:** Использовать флаг `--index` с командами `apply` или `pop`: `git stash apply --index` или `git stash pop --index`.

Освоив `git stash`, вы получили в своё распоряжение мощный инструмент для гибкого переключения контекста без потери незавершённой работы. В следующем уроке мы перейдём к работе с ветками и слияниями.
