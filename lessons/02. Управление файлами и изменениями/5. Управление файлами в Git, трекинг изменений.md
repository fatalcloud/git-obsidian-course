Добрый день, коллеги. Сегодня мы переходим от теории к практике — к той самой работе с файлами, ради которой всё и затевалось. В прошлых уроках мы настроили инструмент и создали репозиторий. Теперь настало время научиться говорить Git: *«Вот этот файл важен, сохрани его состояние, пожалуйста»*.

Этот урок посвящён **жизненному циклу файла** и ключевым командам `git add` и `git commit`. Приготовьтесь, будет много практических примеров и важная аналогия, которая раз и навсегда прояснит роль таинственной Staging Area.

### Жизненный цикл файла в Git: от незнакомца до архивной записи

Представьте, что вы — редактор в издательстве. К вам на стол попадают рукописи. В Git у каждого файла есть чёткий «статус» в зависимости от того, знаком ли он уже издательству.

1.  **Untracked (Неотслеживаемый).** Вы только что создали файл `новая_глава.txt`. Git видит его в папке, но пока не обращает на него внимания. Он как рукопись, которую принёс курьер, но вы ещё не внесли её в реестр входящих документов. Если файл неотслеживаемый, Git не будет сохранять его историю и не сможет восстановить его в случае удаления.
2.  **Tracked (Отслеживаемый).** Вы взяли рукопись, прочитали название и внесли в базу. С этого момента Git знает о файле и следит за каждым его изменением. Отслеживаемые файлы могут находиться в трёх подсостояниях:
    *   **Modified (Изменённый):** Вы взяли ручку и начали править текст в уже известном файле. Git видит, что содержимое файла в рабочем каталоге отличается от того, что сохранено в репозитории.
    *   **Staged (Индексированный/Подготовленный):** Вы закончили правку и положили листы в стопку «В печать». Вы сказали Git: «Вот эту версию файла я хочу сохранить в истории в следующем коммите».
    *   **Committed (Зафиксированный):** Стопка ушла в типографию, и вы получили сигнальный экземпляр. Данные надёжно сохранены в базе данных Git (папке `.git`), и у этого состояния есть уникальный номер (хеш коммита).

Визуально этот путь выглядит так:

```
[Untracked] --(git add)--> [Staged] --(git commit)--> [Committed / Unmodified]
    ^                           |
    |                           |
    +-------(Изменение файла)---+
```

### Индекс (Staging Area): Зачем он нужен?

Очень часто новички спрашивают: *«Почему нельзя сразу сделать коммит? Зачем лишняя команда `git add`? Это же неудобно!»*

Поверьте моему десятилетнему опыту: **индекс — это суперсила Git**.

**Аналогия: Сборка идеального букета.**
Вы — флорист. У вас на столе (Рабочий каталог) лежат десятки разных цветов и веток. Вы не хватаете всё подряд и не заворачиваете в бумагу. Вы выбираете *только* три розы и веточку гипсофилы, чтобы сделать один конкретный букет «Нежность».

Индекс — это ваш верстак, где вы формируете **логический снимок** (коммит).
- Вчера вы правили 15 файлов: в 3 из них вы починили баг с авторизацией, в 7 — начали новую фичу, а в 5 — просто поправили отступы.
- Без индекса вы бы сделали один коммит с сообщением «Всякое по мелочи». Через месяц, когда фича сломается, вы не поймёте, где искать проблему.
- **С индексом** вы сначала сделаете `git add` для тех 3 файлов, которые касаются бага, и сделаете коммит `"Исправил ошибку входа"`. Потом отдельно добавите файлы с фичей и сделаете коммит `"Начало разработки виджета"`. История проекта остаётся кристально чистой.

### Основные команды: Инструменты флориста

Давайте разберём синтаксис и нюансы.

#### `git add` — Добавление в Индекс

Эта команда обновляет индекс, добавляя туда текущее содержимое указанных файлов.

**Базовое использование:**
```bash
# Добавить конкретный файл
git add app.js

# Добавить все файлы в текущей папке и подпапках (расширение .txt)
git add *.txt

# Добавить конкретную папку со всем содержимым
git add src/
```

**Три полезных флага для добавления всего:**
1.  **`git add -A`** или **`git add --all`**
    Добавляет **все** изменения в репозитории: новые файлы (untracked), изменённые (modified) и даже информацию об удалении файлов. Самый надёжный способ сказать «Я хочу закоммитить вообще всё, что наработал».

2.  **`git add .`** (точка)
    Добавляет изменения в **текущей директории и ниже**. Будьте внимательны: если вы находитесь не в корне проекта, файлы в папках уровнем выше добавлены не будут.

3.  **`git add -u`**
    Добавляет только **изменённые и удалённые** файлы, которые **уже отслеживаются**. Новые (untracked) файлы проигнорирует.

#### `git commit` — Сохранение снимка

Эта команда берёт всё, что накопилось в Индексе, и создаёт коммит в репозитории.

**Стандартный подход:**
```bash
git commit -m "Краткое, но ёмкое описание сделанной работы"
```
Флаг `-m` позволяет передать сообщение прямо в командной строке. Если его опустить, Git откроет редактор, который вы настроили ранее (Vim, Nano, VS Code), чтобы вы написали более подробное описание.

**Комбинированный подход (использовать с осторожностью!):**
```bash
git commit -a -m "Сообщение"
```
Флаг `-a` (или `--all`) говорит Git: *«Перед тем как сделать коммит, автоматически добавь в Индекс все **изменённые и удалённые** отслеживаемые файлы»*.

**Ловушка для новичка:** `git commit -a` **не добавляет** новые (Untracked) файлы. Если вы создали `new_feature.js` и сделали `git commit -a`, этот файл останется за бортом коммита. Именно поэтому я рекомендую новичкам **всегда использовать явный `git add`**, чтобы руками контролировать, что попадает в историю.

### Практический пример: Разработка файла `app.js`

Давайте пройдём полный цикл работы с одним файлом, чтобы закрепить понимание.

**Шаг 1. Создаём файл (Состояние: Untracked)**
```bash
echo "console.log('Hello World!');" > app.js
git status
```
**Результат `git status`:**
```
On branch main
Untracked files:
  (use "git add <file>..." to include in what will be committed)
        app.js
```
Git видит файл, но не следит за ним (красный цвет в терминале).

**Шаг 2. Добавляем в Индекс (Состояние: Staged)**
```bash
git add app.js
git status
```
**Результат:**
```
On branch main
Changes to be committed:
  (use "git restore --staged <file>..." to unstage)
        new file:   app.js
```
Git сообщает, что готов создать новый файл в репозитории (зелёный цвет).

**Шаг 3. Фиксируем первый снимок (Состояние: Committed)**
```bash
git commit -m "Добавлен базовый скрипт инициализации"
git status
```
**Результат:**
```
On branch main
nothing to commit, working tree clean
```
Рабочий каталог чист. Файл `app.js` теперь надёжно сохранён в `.git`.

**Шаг 4. Изменяем файл (Состояние: Modified)**
Вносим правки в файл. Допустим, мы поменяли `Hello World` на `Привет, Мир`.
```bash
git status
```
**Результат:**
```
On branch main
Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
        modified:   app.js
```
Файл отслеживаемый, но его текущая версия отличается от той, что в репозитории (красный цвет).

**Шаг 5. Снова добавляем в Индекс**
```bash
git add app.js
git status
```
Теперь изменения снова в зелёной зоне и готовы к следующему коммиту.

### Инструменты разведчика: `git status` и `git diff`

#### `git status` — Ваш компас
Это главная команда для понимания того, что происходит *прямо сейчас*.
- **Красные файлы:** Изменены, но не добавлены в индекс (или не отслеживаются).
- **Зелёные файлы:** Добавлены в индекс, готовы к коммиту.
- **Сообщение "working tree clean":** Всё спокойно, изменений нет.

#### `git diff` — Ваше увеличительное стекло
Эта команда показывает **построчные различия**.

- **`git diff`** (без параметров): Показывает различия между **Рабочим каталогом** и **Индексом**. То есть то, что вы изменили, но ещё не добавили (`git add`).
- **`git diff --staged`** (или `--cached`): Показывает различия между **Индексом** и последним **Коммитом**. То есть то, что попадёт в репозиторий, если вы прямо сейчас выполните `git commit`.

Пример вывода `git diff`:
```diff
diff --git a/app.js b/app.js
index 8d3e5e9..c5f8b3a 100644
--- a/app.js
+++ b/app.js
@@ -1 +1 @@
-console.log('Hello World!');
+console.log('Привет, Мир!');
```
Это невероятно полезно перед коммитом: вы видите, что именно меняется, и можете избежать случайной отправки отладочного вывода `console.log('Я ТУТ')` в общую историю.

### Практическое задание: Осваиваем селективное добавление

Это задание поможет вам прочувствовать концепцию индекса на кончиках пальцев.

1.  Создайте новую папку и инициализируйте в ней репозиторий: `mkdir git_practice && cd git_practice && git init`.
2.  Создайте два файла:
    ```bash
    echo "Текст документа 1" > doc1.txt
    echo "Текст документа 2" > doc2.txt
    ```
3.  Выполните `git status`. Убедитесь, что оба файла красные (Untracked).
4.  **Добавьте в индекс ТОЛЬКО `doc1.txt`**:
    ```bash
    git add doc1.txt
    ```
5.  Проверьте `git status`. Убедитесь, что `doc1.txt` зелёный (готов к коммиту), а `doc2.txt` остался красным.
6.  **Сделайте коммит**:
    ```bash
    git commit -m "Добавил doc1, doc2 пока не готов"
    ```
7.  Снова проверьте `git status`. Что вы видите?
    *   `doc1.txt` больше не показывается (он зафиксирован, изменений в нём нет).
    *   `doc2.txt` **всё ещё красный и неотслеживаемый**. Потому что мы его специально не добавляли.

Это упражнение наглядно показывает, что **коммит сохраняет только содержимое индекса**, а не всего рабочего каталога.

### Вопросы для самопроверки (с ответами)

**1. Зачем нужна промежуточная область (индекс)? Разве нельзя было сделать так, чтобы `git commit` сразу сохранял всё, что изменилось?**
> **Ответ:** Индекс позволяет формировать **осмысленные атомарные коммиты**. Без индекса в один коммит попадала бы мешанина из правок к разным задачам. С индексом мы можем разбить изменения на логические блоки: "Рефакторинг кнопки", "Исправление орфографии", "Новая функция экспорта". Это делает историю проекта читаемой и упрощает поиск ошибок в будущем.

**2. Что произойдёт, если я создам новый файл и сразу выполню `git commit -a -m "..."`?**
> **Ответ:** **Ничего.** Файл останется неотслеживаемым (Untracked) и не попадёт в коммит. Флаг `-a` автоматически добавляет в индекс только те файлы, которые **уже были в прошлых коммитах** и были изменены или удалены. Новые файлы требуют явного `git add`.

**3. Я изменил файл, сделал `git add`, потом снова изменил этот же файл и сразу сделал `git commit`. Какая версия файла сохранится?**
> **Ответ:** Сохранится та версия, которая была на момент выполнения `git add`. Второе изменение останется в рабочем каталоге как **Modified**. Git фиксирует именно **снимок индекса** на момент команды `git commit`. Чтобы сохранить последнюю версию, нужно перед коммитом сделать `git add` ещё раз.

**4. (Вопрос со звёздочкой) Как убрать файл из индекса, если я случайно добавил его туда (`git add`), но ещё не делал коммит?**
> **Ответ:** Git сам подсказывает ответ в выводе `git status`: `use "git restore --staged <file>..." to unstage`. Команда `git restore --staged doc1.txt` вернёт файл в состояние "Modified" (красный), но не удалит его из рабочего каталога.
