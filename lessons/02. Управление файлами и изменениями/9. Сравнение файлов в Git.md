Добрый день, коллеги. Сегодня я надену очки аналитика, потому что мы погрузимся в мир `git diff` — команды, без которой невозможно представить осознанную работу с кодом. `git diff` показывает, **что именно изменилось** между двумя состояниями вашего проекта. Это ваш главный инструмент для проверки собственной работы перед коммитом и для анализа чужого кода при код-ревью. Освоив `git diff`, вы перестанете гадать «а что я там написал вчера?» и начнёте точно знать ответ.

### Что сравнивает `git diff`?

По умолчанию Git оперирует тремя основными областями: **рабочий каталог**, **индекс** (staging area) и **репозиторий** (последний коммит, на который указывает `HEAD`). `git diff` может сравнивать любые две из них, а также произвольные коммиты и даже отдельные файлы из разных веток.

Результат выводится в унифицированном формате (unified diff), где удалённые строки помечаются минусом `-`, а добавленные — плюсом `+`. Контекст (неизменённые строки) отображается для лучшего понимания расположения изменений.

### 1. Рабочий каталог против Индекса: `git diff`

Самая частая команда перед выполнением `git add`. Вы изменили несколько файлов и хотите убедиться, что не внесли случайных отладочных выводов или опечаток.

**Команда:**
```bash
git diff
```

**Что она делает:** Сравнивает содержимое файлов в вашем **рабочем каталоге** с их версиями в **индексе** (staging area). Если файл не был добавлен в индекс, в индексе лежит его версия из последнего коммита.

**Пример использования:**
Вы работаете над `app.js` и хотите проверить правки перед `git add`.

```bash
# Вносим изменения в файл
echo "console.log('debug');" >> app.js

# Смотрим, что изменилось
git diff
```

Вывод покажет что-то вроде:
```diff
diff --git a/app.js b/app.js
index 8d3e5e9..c5f8b3a 100644
--- a/app.js
+++ b/app.js
@@ -1,3 +1,4 @@
 function init() {
   console.log('Start');
+  console.log('debug');
 }
```

Вы видите, что добавили `console.log('debug');`. Если это была случайность — вы успеваете удалить строку до индексации.

### 2. Индекс против Последнего коммита: `git diff --staged`

Выполнив `git add`, вы поместили изменения в индекс. Теперь `git diff` (без флагов) уже ничего не покажет, потому что рабочий каталог и индекс синхронизированы. Но перед коммитом полезно проверить, **что именно попадёт в репозиторий**.

**Команда:**
```bash
git diff --staged
# или устаревший вариант
git diff --cached
```

**Что она делает:** Сравнивает **индекс** с последним коммитом (`HEAD`). Это именно те изменения, которые будут зафиксированы следующим `git commit`.

**Пример:**
```bash
# Добавляем файл в индекс
git add app.js

# Смотрим, что уйдёт в коммит
git diff --staged
```

Вывод будет идентичен предыдущему, но теперь это подтверждение того, что именно эти правки войдут в историю.

### 3. Сравнение двух коммитов

Иногда нужно понять, что изменилось между двумя конкретными точками в истории, например, между вчерашним релизом и сегодняшним состоянием.

**Синтаксис:**
```bash
git diff <коммит1> <коммит2>
```

В качестве коммитов можно использовать:
- Полный или сокращённый хеш (`a1b2c3d`, `e83c5163`)
- Имя ветки (`main`, `feature/new-logic`)
- Относительные ссылки (`HEAD~2`, `HEAD^`)
- Специальный указатель `HEAD` (текущий коммит)

**Примеры:**
```bash
# Сравнить последний коммит с предпоследним
git diff HEAD~1 HEAD

# Сравнить две ветки
git diff main feature/experiment

# Посмотреть изменения в конкретном файле между двумя коммитами
git diff abc123 def456 -- src/utils.js
```

**Реальный сценарий:** Вы проверяете пулл-реквест коллеги. Ветка `feature/new-button` расходится с `main`. Выполнив `git diff main...feature/new-button` (с тремя точками), вы увидите только те изменения, которые были сделаны именно в ветке `feature`, исключая общие коммиты с `main`.

### 4. Сравнение файлов из разных веток

Часто нужно сравнить не всю ветку целиком, а конкретный файл, например, конфигурационный файл, который менялся в обеих ветках.

**Синтаксис:**
```bash
git diff <источник1>:<путь/к/файлу> <источник2>:<путь/к/файлу>
```

**Пример:**
```bash
# Сравнить файл config.yml в ветке main и в ветке develop
git diff main:config.yml develop:config.yml
```

Это можно прочитать так: «Покажи разницу между файлом `config.yml` в том виде, в каком он существует в ветке `main`, и тем же файлом в ветке `develop`».

### 5. Полезные опции для `git diff`

Сырой построчный вывод не всегда удобен. Git предоставляет опции для изменения формата отображения.

| Опция | Описание | Пример использования |
| :--- | :--- | :--- |
| `--stat` | Показывает краткую статистику: список изменённых файлов и количество добавленных/удалённых строк. | `git diff --stat HEAD~3 HEAD` |
| `--name-only` | Выводит только имена изменённых файлов. Удобно для скриптов. | `git diff --name-only main feature` |
| `--word-diff` | Показывает изменения **на уровне слов**, а не целых строк. Очень полезно для текстов (документация, Markdown). | `git diff --word-diff` |
| `--color-words` | Аналог `--word-diff`, но с более компактным цветовым выделением. | `git diff --color-words` |

**Пример работы `--word-diff`:**
Исходная строка: `The quick brown fox jumps.`
Изменённая строка: `The quick red fox jumps.`

Обычный `git diff` покажет удаление всей строки и добавление новой. `--word-diff` выделит только заменённое слово:
```
The quick [-brown-]{+red+} fox jumps.
```

Это незаменимо при работе с длинными абзацами в документации или Markdown-файлами.

### 6. Что показывает `git diff HEAD`?

Это частый вопрос, и ответ на него объединяет несколько концепций.

**Команда:**
```bash
git diff HEAD
```

**Что она делает:** Сравнивает **рабочий каталог** (НЕ индекс!) с последним коммитом (`HEAD`). Она показывает **все** изменения, которые вы сделали с момента последнего коммита, **независимо от того, добавлены они в индекс или нет**.

**Отличие от `git diff` и `git diff --staged`:**
- `git diff` → Рабочий каталог vs Индекс.
- `git diff --staged` → Индекс vs HEAD.
- `git diff HEAD` → Рабочий каталог vs HEAD (сумма двух предыдущих).

Если вы изменили файл, добавили его в индекс, а потом снова изменили, `git diff HEAD` покажет **все** изменения (и те, что в индексе, и те, что ещё не добавлены). Это удобно, чтобы окинуть взглядом всю работу, проделанную после последнего коммита.

### 7. Сравнение файла с версией двухкоммитной давности

Вопрос из самопроверки: «Как сравнить текущий файл с версией из двух коммитов назад?»

Ответ зависит от того, что вы подразумеваете под «текущим файлом».

**Вариант 1: Сравнить файл в рабочем каталоге с версией из `HEAD~2`**
```bash
git diff HEAD~2 -- <имя_файла>
```

**Вариант 2: Сравнить файл в последнем коммите (`HEAD`) с `HEAD~2`**
```bash
git diff HEAD~2 HEAD -- <имя_файла>
```

**Вариант 3: Сравнить файл из индекса с `HEAD~2`**
```bash
git diff --staged HEAD~2 -- <имя_файла>
```

Двойное тире `--` используется для явного отделения опций команды от путей к файлам, чтобы Git не перепутал имя файла с названием ветки или хешем.

### Практическое задание: Исследуем изменения

Выполните эти шаги, чтобы натренировать глаз и руку на `git diff`.

1.  **Создайте новый репозиторий и файл:**
    ```bash
    mkdir diff-lab && cd diff-lab
    git init
    echo "Первая строка" > poem.txt
    git add poem.txt && git commit -m "Начало поэмы"
    ```

2.  **Внесите первое изменение (в рабочем каталоге):**
    ```bash
    echo "Вторая строка" >> poem.txt
    # Проверьте разницу между рабочим каталогом и индексом (т.е. последним коммитом)
    git diff
    ```

3.  **Добавьте изменение в индекс и проверьте:**
    ```bash
    git add poem.txt
    # Теперь git diff ничего не покажет (рабочий каталог = индекс)
    # Проверяем, что уйдёт в коммит:
    git diff --staged
    ```

4.  **Сделайте коммит:**
    ```bash
    git commit -m "Добавлена вторая строка"
    ```

5.  **Внесите ещё одно изменение и сравните с коммитом двух шагов назад:**
    ```bash
    echo "Третья строка" >> poem.txt
    git add poem.txt && git commit -m "Добавлена третья строка"
    
    # Сравните текущий файл в рабочем каталоге (он чистый после коммита, поэтому покажет пустоту)
    # Лучше сравнить последний коммит с тем, что был два коммита назад:
    git diff HEAD~2 HEAD -- poem.txt
    ```

6.  **Попробуйте `--stat` и `--word-diff`:**
    ```bash
    # Сделайте ещё коммит с изменением слова
    sed -i 's/Первая/Самая первая/' poem.txt  # или ручная правка
    git add poem.txt && git commit -m "Уточнил первую строку"
    
    # Посмотрите статистику изменений между тремя последними коммитами
    git diff --stat HEAD~3 HEAD
    # Посмотрите разницу в словах
    git diff --word-diff HEAD~1 HEAD
    ```

### Вопросы для самопроверки

**1. Что показывает команда `git diff HEAD`? Чем она отличается от `git diff` и `git diff --staged`?**
> **Ответ:** `git diff HEAD` показывает разницу между **рабочим каталогом** и последним коммитом (`HEAD`), игнорируя состояние индекса. Она объединяет изменения, которые уже в индексе, и те, что ещё не добавлены. `git diff` показывает только незаиндексированные изменения, а `git diff --staged` — только заиндексированные.

**2. Как с помощью `git diff` сравнить текущее состояние файла `config.json` (в рабочем каталоге) с версией этого файла из коммита с хешем `a1b2c3d`?**
> **Ответ:**
> ```bash
> git diff a1b2c3d -- config.json
> ```
> Или если нужно сравнить с версией, которая была два коммита назад:
> ```bash
> git diff HEAD~2 -- config.json
> ```

**3. Вы изменили 10 файлов, хотите быстро оценить масштаб бедствия перед коммитом, но не хотите видеть все строки кода. Какую опцию использовать?**
> **Ответ:** `git diff --stat`. Она выведет компактную таблицу с именами файлов и количеством добавленных/удалённых строк.

**4. Как посмотреть, что изменилось в ветке `feature` относительно `main` только в файлах, лежащих в папке `src/`?**
> **Ответ:**
> ```bash
> git diff main feature -- src/
> ```
> Можно также использовать `git diff main..feature -- src/` (две точки) или `...` для изменений с момента ответвления.

**5. Что выведет `git diff --staged`, если индекс пуст?**
> **Ответ:** Ничего. Команда завершится без вывода, так как нет разницы между пустым индексом и последним коммитом (индекс содержит те же версии файлов, что и `HEAD`).
