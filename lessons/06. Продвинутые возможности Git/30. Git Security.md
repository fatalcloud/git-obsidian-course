Приветствую, коллеги. Сегодня я надену шляпу security-эксперта и проведу для вас урок по безопасности Git — теме, о которой часто забывают, пока не становится слишком поздно. За годы работы я расследовал десятки инцидентов, связанных с утечками секретов через репозитории, и могу с уверенностью сказать: большинство из них можно было предотвратить с помощью простых, но последовательных практик. Git помнит всё, и это одновременно его сила и его главная уязвимость: секрет, попавший в историю, остаётся там навсегда, если не предпринять специальных мер. Сегодня мы разберём, как защитить ваш код и вашу репутацию.

### 1. Защита учётных данных: Не храните пароли и токены в коде

Это правило номер один. Никогда, ни при каких обстоятельствах не коммитьте в репозиторий файлы, содержащие пароли, API-ключи, токены доступа, приватные SSH-ключи или любые другие секреты.

**Почему это критично?**
Даже если вы быстро удалите секрет в следующем коммите, он навсегда останется в истории Git. Злоумышленник, получивший доступ к репозиторию (или к его клону), может просто пройтись по истории коммитов и найти ваш секрет. Это может привести к компрометации серверов, утечке данных пользователей и серьёзным финансовым потерям.

**Как предотвратить утечку на этапе разработки:**

1.  **`.gitignore`** — ваш главный помощник. Всегда добавляйте в `.gitignore` файлы, которые могут содержать секреты, **до** первого коммита. Типичные кандидаты: `.env`, `*.local`, `secrets.yml`, `config/credentials.yml.enc` (для Rails, но без ключа).

2.  **`git-secrets`** — инструмент от AWS Labs, который сканирует коммиты, сообщения коммитов и даже `git merge` на предмет известных паттернов секретов (ключи AWS, GitHub, Google и др.). Установите его глобально или для конкретного репозитория, и он будет блокировать коммиты, содержащие подозрительные строки.

    ```bash
    # Установка (пример для macOS)
    brew install git-secrets
    
    # Регистрация стандартных паттернов AWS в глобальной конфигурации
    git secrets --register-aws --global
    
    # Установка хуков для конкретного репозитория
    cd /path/to/your/repo
    git secrets --install
    git secrets --register-aws
    ```

3.  **`truffleHog`** — ещё один мощный сканер, который ищет секреты с высокой энтропией (то есть похожие на случайно сгенерированные ключи) в истории коммитов и в текущем состоянии репозитория.

4.  **GitHub Push Protection** — для публичных репозиториев на GitHub эта функция включена по умолчанию. Она сканирует коммиты перед отправкой на сервер и блокирует пуш, если обнаруживает секреты (поддерживается более 200 типов токенов).

**Что делать, если секрет уже попал в историю?**
1.  **Немедленно отзовите скомпрометированный секрет.** Сгенерируйте новый ключ/пароль/токен. Это самое важное и срочное действие.
2.  Используйте **BFG Repo-Cleaner** или `git filter-repo`, чтобы полностью удалить секрет из всей истории репозитория.
3.  Сделайте принудительный пуш (`git push --force`) и предупредите всех участников команды, чтобы они переклонировали репозиторий.

### 2. Подпись коммитов с помощью GPG: Докажите, что это вы

По умолчанию Git идентифицирует автора коммита по полям `user.name` и `user.email`, которые можно подделать. GPG (GNU Privacy Guard) позволяет создавать криптографическую цифровую подпись для каждого коммита, доказывая, что он действительно был создан владельцем приватного ключа.

**Зачем это нужно?**
- **Аутентичность:** Вы подтверждаете, что коммит исходит именно от вас, а не от злоумышленника, подделавшего ваш email.
- **Целостность:** Подпись гарантирует, что код не был изменён после того, как вы его подписали.
- **Доверие:** На GitHub и GitLab подписанные коммиты получают зелёную метку **"Verified"**, что повышает доверие к вашему коду.

**Пошаговая настройка подписи коммитов:**

**Шаг 1. Установите GPG**
- **macOS:** `brew install gnupg`
- **Linux (Debian/Ubuntu):** `sudo apt install gnupg`
- **Windows:** Скачайте [Gpg4win](https://www.gpg4win.org/).

**Шаг 2. Сгенерируйте GPG-ключ**
```bash
gpg --full-generate-key
```
- Выберите тип ключа: `RSA and RSA` (по умолчанию).
- Размер ключа: `4096` бит.
- Срок действия: `1y` (рекомендуется, для безопасности).
- Имя и email: **обязательно используйте email, привязанный к вашему GitHub/GitLab аккаунту!**
- Задайте надёжный пароль для защиты приватного ключа.

**Шаг 3. Получите ID ключа и экспортируйте публичную часть**
```bash
# Вывести список секретных ключей с длинными ID
gpg --list-secret-keys --keyid-format=long
```
В выводе найдите строку вида `sec rsa4096/AB1234567890CDEF`. `AB1234567890CDEF` — это ID вашего ключа.

```bash
# Экспортировать публичный ключ в ASCII-формате
gpg --armor --export AB1234567890CDEF
```
Скопируйте **весь** вывод, включая строки `-----BEGIN PGP PUBLIC KEY BLOCK-----` и `-----END PGP PUBLIC KEY BLOCK-----`.

**Шаг 4. Добавьте публичный ключ на GitHub/GitLab**
- GitHub: **Settings** → **SSH and GPG keys** → **New GPG key**. Вставьте скопированный публичный ключ.

**Шаг 5. Настройте Git для использования GPG-ключа**
```bash
git config --global user.signingkey AB1234567890CDEF
git config --global commit.gpgsign true
```
Теперь Git будет автоматически подписывать все ваши коммиты с помощью указанного ключа.

**Шаг 6. Подписывайте коммиты**
- Автоматически: `git commit -m "Your message"` (если включили `commit.gpgsign true`).
- Для отдельного коммита: `git commit -S -m "Your signed message"`

**Проверка подписи:**
- Локально: `git log --show-signature`
- На GitHub: у подписанных коммитов будет зелёная плашка "Verified".

### 3. Безопасная работа с удалёнными репозиториями: SSH и Host-ключи

При взаимодействии с удалёнными репозиториями (GitHub, GitLab) вы аутентифицируетесь либо по HTTPS, либо по SSH.

| Протокол | Плюсы | Минусы | Рекомендация |
| :--- | :--- | :--- | :--- |
| **SSH** | Высокая безопасность, не нужно вводить пароль при каждом пуше, шифрование с открытым ключом. | Сложнее в настройке, может блокироваться корпоративными файрволами. | **Предпочтительно для постоянной разработки** |
| **HTTPS** | Проще в настройке, работает через стандартный порт 443, подходит для сред с ограничениями. | Менее безопасен (если не используется токен), может требовать частого ввода учётных данных. | **Для быстрого доступа или в ограниченных средах** |

**Рекомендация security-эксперта:** Используйте **SSH-ключи** для всех репозиториев, с которыми работаете регулярно. Сгенерируйте пару ключей (`ssh-keygen -t ed25519 -C "your_email@example.com"`), добавьте публичный ключ в настройки GitHub/GitLab, и используйте SSH-URL для клонирования (`git clone git@github.com:user/repo.git`).

**Проверка host-ключей:** При первом подключении к новому SSH-серверу вы увидите fingerprint его host-ключа. **Всегда сверяйте его с официально опубликованными значениями** (например, [GitHub's SSH keys fingerprints](https://docs.github.com/en/authentication/keeping-your-account-and-data-secure/githubs-ssh-key-fingerprints)), чтобы избежать атак «человек посередине» (MITM).

### 4. Ограничение прав на ветки: Защищённые ветки (Protected Branches)

Даже с подписанными коммитами и безопасным подключением, человеческий фактор остаётся. Защита веток предотвращает случайные или злонамеренные изменения в критически важных ветках, таких как `main`, `master`, `release/*`.

**На GitHub:**
1. Перейдите в **Settings** вашего репозитория → **Branches**.
2. Нажмите **Add branch ruleset** (рекомендуемый способ) или **Add rule** (legacy).
3. В разделе **Branch name pattern** укажите `main` (или `release/*`).
4. Настройте правила. **Обязательные для включения:**
   - **Require a pull request before merging** — запретить прямые пуши в ветку.
   - **Require approvals** — требовать хотя бы одно одобрение от другого члена команды.
   - **Require status checks to pass** — запретить слияние, если тесты CI/CD не прошли.
   - **Block force pushes** — предотвратить перезапись истории.
   - **Require signed commits** — принимать только подписанные GPG коммиты.
5. Сохраните правила.

Эти меры создают «карантинный барьер»: ни один разработчик не может в одиночку изменить защищённую ветку — каждое изменение должно пройти через пулл-реквест, код-ревью и автоматические проверки.

### 5. Аудит репозитория: Ищите то, что могло просочиться

Регулярный аудит истории Git — важная часть профилактики. Даже если вы уверены, что никогда не коммитили секреты, проверка не будет лишней.

**Базовые команды для поиска:**

```bash
# Поиск по всем коммитам с использованием git log и grep
git log -p | grep -i "password\|secret\|key\|token"

# Поиск строки "password" во всей истории файла
git log -S "password" --oneline

# Поиск строки по регулярному выражению (например, AWS ключи)
git log -p | grep -E "AKIA[0-9A-Z]{16}"
```

**Специализированные инструменты для аудита:**
- **`git-secrets --scan-history`** — просканирует всю историю репозитория на наличие известных паттернов секретов.
- **`truffleHog`** — проанализирует историю коммитов и найдет строки с высокой энтропией.
- **GitHub Secret Scanning** — для публичных репозиториев GitHub автоматически сканирует историю на предмет известных типов токенов и уведомляет владельца при обнаружении.

### Практическое задание: Настройка и проверка GPG-подписи

1.  **Создайте новый тестовый репозиторий:**
    ```bash
    mkdir gpg-security-lab && cd gpg-security-lab
    git init
    ```

2.  **Сгенерируйте GPG-ключ** (если у вас его ещё нет). Следуйте инструкциям из раздела 2. Убедитесь, что используете email, привязанный к GitHub.

3.  **Настройте Git для использования ключа:**
    ```bash
    git config --global user.signingkey <YOUR_KEY_ID>
    git config --global commit.gpgsign true
    ```

4.  **Создайте и подпишите коммит:**
    ```bash
    echo "# Security Test" > README.md
    git add README.md
    git commit -S -m "chore: initial signed commit"
    ```

5.  **Проверьте подпись локально:**
    ```bash
    git log --show-signature
    ```
    Вы должны увидеть строку `gpg: Good signature from ...`.

6.  **(Опционально) Отправьте репозиторий на GitHub.** Убедитесь, что публичный GPG-ключ добавлен в настройках аккаунта. После пуша рядом с коммитом должна появиться зелёная плашка "Verified".

### Вопросы для самопроверки

**1. Почему важно подписывать теги релизов с помощью GPG?**
> **Ответ:** Тег релиза (`v1.0.0`) является точкой входа для пользователей и CI/CD-систем. Подпись тега гарантирует, что релиз был создан доверенным лицом и не был подменён после создания. Это критически важно для цепочки поставок (supply chain security), особенно при распространении артефактов или публикации пакетов.

**2. Как правильно отозвать скомпрометированный GPG-ключ?**
> **Ответ:**
> 1. Сгенерируйте **сертификат отзыва (revocation certificate)**. Рекомендуется создать его сразу при генерации ключа (`gpg --gen-revoke <KEY_ID>`).
> 2. Импортируйте сертификат отзыва в свою связку ключей (`gpg --import revocation-certificate.asc`).
> 3. Отправьте отозванный ключ на сервер ключей (`gpg --send-keys <KEY_ID>`), чтобы другие пользователи узнали о компрометации.
> 4. Удалите скомпрометированный ключ из своего аккаунта GitHub/GitLab и замените его новым.

**3. Вы случайно запушили коммит с секретом в публичный репозиторий. Каков ваш немедленный план действий?**
> **Ответ:**
> 1. **Первое и самое важное:** Немедленно отзовите скомпрометированный секрет (сгенерируйте новый токен/пароль).
> 2. Используйте **BFG Repo-Cleaner** или `git filter-repo`, чтобы полностью удалить секрет из всей истории.
> 3. Выполните принудительный пуш (`git push --force`).
> 4. Свяжитесь со всеми, кто мог склонировать репозиторий, и попросите их переклонировать его или выполнить `git pull --rebase`.

**4. Чем GPG-подпись коммита отличается от SSH-подписи коммита, и что лучше использовать?**
> **Ответ:** Оба метода обеспечивают криптографическую верификацию. GPG — это более старый, но широко распространённый стандарт, который также используется для шифрования email и файлов. SSH-подпись — более новый метод, использующий ваш существующий SSH-ключ, что упрощает управление ключами. GitHub поддерживает оба. Если у вас уже есть SSH-ключи и вы не хотите управлять отдельным GPG-ключом, SSH-подпись может быть удобнее. Для проектов, требующих максимальной совместимости и устоявшихся практик, GPG остаётся стандартом де-факто.

**5. Какие настройки защищённой ветки (protected branch) являются минимально необходимыми для предотвращения случайных или злонамеренных изменений в `main`?**
> **Ответ:**
> - **Require a pull request before merging** (запрет прямых пушей).
> - **Require approvals** (минимум одно одобрение).
> - **Require status checks to pass before merging** (блокировка при падении CI).
> - **Block force pushes** (запрет перезаписи истории).
> - **Require signed commits** (приём только подписанных коммитов).
