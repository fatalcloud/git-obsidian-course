Приветствую, коллеги. Сегодня я надену каску специалиста по производительности и обслуживанию Git. Мы часто воспринимаем репозиторий как нечто само собой разумеющееся, но внутри папки `.git` происходит постоянная работа: объекты сжимаются, старые ссылки удаляются, целостность проверяется. Понимание этих процессов и умение ими управлять — признак профессионала, который заботится о долгосрочном здоровье проекта. Мы рассмотрим, как Git хранит данные «под капотом», как проводить профилактику и что делать, если репозиторий «распух» от бинарного мусора.

### 1. Как Git хранит объекты: свободные объекты и pack-файлы

Прежде чем говорить об обслуживании, нужно понять, с чем мы работаем. Git хранит все данные (коммиты, деревья, файлы) в виде **объектов** в папке `.git/objects`. Изначально каждый объект сохраняется в виде отдельного **свободного файла (loose object)**. Это быстро для записи, но неэффективно для хранения: тысячи маленьких файлов занимают больше места на диске из-за фрагментации и служебных областей файловой системы.

Git периодически (или при ручном запуске) упаковывает множество свободных объектов в **пак-файлы (packfiles)**. Это файлы с расширением `.pack` (сжатые данные) и соответствующие им `.idx` (индексы для быстрого поиска). Внутри pack-файла объекты хранятся в виде **дельт** (разницы между похожими версиями), что значительно экономит место.

**Метафора:** Представьте, что вы пишете черновики на отдельных листочках и разбрасываете их по столу (свободные объекты). В конце дня вы собираете их в папку-скоросшиватель, причём похожие черновики вкладываете друг в друга, чтобы сэкономить место (pack-файл).

### 2. `git gc`: Сборка мусора и оптимизация

**`git gc` (garbage collection)** — это главная команда обслуживания. Она запускает набор подкоманд, которые:

- Собирают свободные объекты в pack-файлы.
- Удаляют недостижимые объекты (те, на которые больше не ссылаются ветки, теги или reflog).
- Оптимизируют pack-файлы.

**Когда запускать?**
Git автоматически запускает `git gc --auto` при определённых условиях (например, когда количество свободных объектов превышает порог). В повседневной работе ручной запуск требуется редко. Однако **полезно запустить вручную**:

- После интенсивной работы (например, после большого `rebase` или `filter-branch`).
- Перед созданием архивной копии репозитория.
- Если вы заметили, что размер `.git` неоправданно велик.

**Базовое использование:**
```bash
git gc
```

**Флаг `--aggressive`**
```bash
git gc --aggressive
```
Эта команда заставляет Git потратить больше времени на поиск оптимальных дельт в pack-файлах. Она может дополнительно уменьшить размер репозитория, но выполняется значительно дольше. **Не рекомендуется запускать её часто** — достаточно раз в несколько месяцев на крупных проектах или перед архивацией.

### 3. `git fsck`: Проверка целостности репозитория

**`git fsck` (file system check)** — это диагностический инструмент, который проверяет целостность базы данных объектов Git. Он сканирует все объекты и ссылки, сообщая о проблемах:

- **Повреждённые объекты** (например, из-за сбоя диска).
- **Висячие объекты (dangling)**: объекты, на которые нет ссылок, но они ещё не удалены сборщиком мусора. Это не ошибка, а нормальное явление после переписывания истории.
- **Несоответствия в связях** (например, коммит ссылается на несуществующее дерево).

**Использование:**
```bash
git fsck
```

Если команда выводит `dangling commit` или `dangling blob` — это нормально. Если вы видите `error: inflate: data stream error` или `error: corrupt loose object` — это серьёзная проблема, указывающая на повреждение данных.

### 4. Очистка reflog и удаление недостижимых объектов

Даже после `git gc` некоторые объекты могут оставаться в репозитории, потому что на них ссылается **reflog** — локальный журнал перемещений `HEAD`. По умолчанию записи в reflog хранятся **90 дней** для достижимых коммитов и **30 дней** для недостижимых. Это ваша страховка от случайных `reset --hard`.

**Принудительная очистка reflog (для немедленного освобождения места):**
```bash
# Установить срок хранения записей, например, 1 неделю
git reflog expire --expire=1.week --all
```

**`git prune`**
Эта команда удаляет недостижимые объекты, на которые **не ссылается даже reflog**. Обычно `git gc` включает в себя `git prune`, но можно вызвать и отдельно:
```bash
git prune
```

**Важно:** После очистки reflog и `prune` восстановить случайно удалённые коммиты станет **невозможно**.

### 5. Уменьшение размера репозитория: удаление больших файлов из истории

Если размер `.git` огромен (гигабайты), то `git gc` может помочь лишь частично. Чаще всего причина — **случайно закоммиченные и позже удалённые большие бинарные файлы**. Поскольку Git хранит всю историю, эти файлы навсегда остаются в pack-файлах. Единственный способ радикально уменьшить размер — **переписать историю** и удалить эти файлы из всех коммитов.

**Предупреждение команды:** Переписывание истории, которая уже опубликована на удалённом сервере, **требует принудительного пуша (`git push --force`)** и **ломает репозитории всех ваших коллег**. Делайте это только в крайнем случае, после тщательной координации с командой, и только на ветках, с которыми работает ограниченный круг лиц. Идеально — делать это в изолированном клоне перед тем, как заменить старый репозиторий новым.

**Инструменты:**

- **`git filter-branch`** (встроенный, но медленный и сложный).
    ```bash
    # Пример удаления файла backup.tar из всей истории
    git filter-branch --force --index-filter \
      "git rm --cached --ignore-unmatch backup.tar" \
      --prune-empty --tag-name-filter cat -- --all
    ```

- **BFG Repo-Cleaner** (внешний инструмент, гораздо быстрее и проще).
    ```bash
    # Установите BFG (скачайте jar-файл)
    java -jar bfg.jar --delete-files backup.tar .git
    git reflog expire --expire=now --all
    git gc --prune=now --aggressive
    ```

После любой очистки истории обязательно выполните `git gc --aggressive` для переупаковки объектов.

### 6. Как посмотреть размер объектов и оценить «здоровье» репозитория

- **`git count-objects -vH`** — показывает количество свободных объектов и размер pack-файлов в человекочитаемом формате.
    ```bash
    git count-objects -vH
    ```
    Вывод:
    ```
    count: 152
    size: 1.2 MiB
    in-pack: 1520
    packs: 2
    size-pack: 12.5 MiB
    prune-packable: 0
    garbage: 0
    size-garbage: 0 bytes
    ```

- **Поиск самых больших объектов в истории (до переписывания):**
    ```bash
    # Вывести список всех объектов с размером, отсортировать по размеру
    git rev-list --objects --all | \
      git cat-file --batch-check='%(objecttype) %(objectname) %(objectsize) %(rest)' | \
      sed -n 's/^blob //p' | \
      sort --numeric-sort --key=2 | \
      tail -10
    ```
    Это покажет хеши и имена 10 самых больших файлов. Затем можно найти их имена: `git rev-list --objects --all | grep <хеш>`.

### Практическое задание: Измеряем эффективность `git gc`

1.  **Создайте новый репозиторий и сгенерируйте много объектов:**
    ```bash
    mkdir gc-lab && cd gc-lab
    git init
    
    # Создадим 100 коммитов с изменениями в файле, чтобы накопить объекты
    for i in {1..100}; do
      echo "Line $i" >> file.txt
      git add file.txt
      git commit -m "Commit $i"
    done
    ```

2.  **Посмотрите начальный размер `.git`:**
    ```bash
    du -sh .git
    # или в PowerShell: (Get-ChildItem -Recurse .git | Measure-Object -Property Length -Sum).Sum / 1MB
    ```

3.  **Запустите сборку мусора с агрессивной оптимизацией:**
    ```bash
    git gc --aggressive
    ```

4.  **Снова проверьте размер `.git`.**
    Он должен уменьшиться, так как свободные объекты упакованы в эффективный pack-файл.

5.  **Бонус: удалите файл из истории и посмотрите на эффект.**
    - Создайте большой файл (например, `dd if=/dev/zero of=big.dat bs=1M count=50`), закоммитьте.
    - Затем удалите его из истории с помощью `git filter-branch` или BFG.
    - Запустите `git gc --aggressive` и сравните размер до и после.

### Вопросы для самопроверки

**1. Что происходит с «висячими коммитами» (dangling commits) при запуске `git gc`?**
> **Ответ:** Если на висячий коммит больше не ссылается ни одна ветка, тег или запись в reflog, `git gc` удалит его и все связанные с ним недостижимые объекты. Если же запись в reflog ещё существует (и не истек её срок), коммит останется до тех пор, пока reflog не будет очищен или не истечёт срок хранения.

**2. Как посмотреть общий размер всех объектов в репозитории (и свободных, и упакованных)?**
> **Ответ:** Команда `git count-objects -vH` показывает размер pack-файлов (`size-pack`) и размер свободных объектов (`size`). Также можно использовать `du -sh .git` для общего размера папки, но это включает и другие служебные файлы.

**3. Вы нашли, что репозиторий весит 2 ГБ, и хотите уменьшить его, удалив случайно закоммиченный видеофайл. Какой подход безопаснее всего для команды, если репозиторий уже используется несколькими разработчиками?**
> **Ответ:** Безопаснее всего **не переписывать историю публичного репозитория**, а:
> 1. Удалить файл в новом коммите (`git rm`).
> 2. Настроить Git LFS для предотвращения повторения проблемы.
> 3. Если размер критичен, **создать новый репозиторий с очищенной историей** (например, с помощью BFG), и попросить команду переключиться на него, а старый заархивировать. Переписывание истории в общем репозитории с `--force` почти всегда ведёт к хаосу.

**4. Зачем нужен флаг `--aggressive` в `git gc`? Когда его стоит использовать?**
> **Ответ:** `--aggressive` заставляет Git тщательнее искать дельты в pack-файлах, что может дополнительно уменьшить размер репозитория за счёт большего времени выполнения. Используйте его **редко**: после большой чистки истории, перед архивацией проекта, или если обычный `git gc` не дал желаемого результата. Частый запуск не рекомендуется, так как он создаёт избыточную нагрузку.

**5. Что делает команда `git reflog expire --expire=now --all` и зачем она может понадобиться?**
> **Ответ:** Она немедленно удаляет **все** записи из reflog для всех ссылок. Это нужно для того, чтобы сборщик мусора (`git gc`) мог удалить все объекты, которые больше нигде не используются (например, после переписывания истории). Без очистки reflog старые коммиты будут висеть ещё 90 дней. Эта команда необратима, используйте с осторожностью.
