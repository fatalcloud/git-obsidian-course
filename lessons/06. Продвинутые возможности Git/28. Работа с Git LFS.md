Приветствую, коллеги. Сегодня я, как инженер данных, расскажу о решении одной из самых «тяжёлых» проблем Git — о **Git Large File Storage (LFS)**. В мире разработки и, особенно, в Data Science, мы постоянно сталкиваемся с огромными бинарными файлами: датасеты, обученные модели, видеоролики, PSD-макеты. Git, созданный для текстового кода, отчаянно «задыхается» при работе с ними. Git LFS — это «кислородная маска», которая позволяет Git оставаться быстрым и лёгким, даже когда в проекте гигабайты данных.

### 1. Проблема: Почему Git не дружит с большими бинарными файлами?

Git спроектирован так, чтобы хранить **полную историю изменений** каждого файла. Для текстовых файлов это работает гениально: Git сохраняет только разницу (дельту) между версиями. Но с бинарными файлами так не получится.

Представьте, что вы добавили видеофайл `video.mp4` размером 150 МБ и сделали коммит. Теперь вы вносите небольшое изменение в видео и снова коммитите. Git вынужден сохранить **ещё одну полную копию** всего файла. После десяти таких коммитов ваш репозиторий будет весить уже 1.5 ГБ.

**Результаты для команды:**
*   **Чудовищное время клонирования:** Каждый новый разработчик будет вынужден скачать всю эту многогигабайтную историю, даже если ему нужна только последняя версия файла.
*   **Раздувание репозитория:** Размер папки `.git` становится неконтролируемым, что замедляет все операции: `git status`, `git log`, `git pull`.
*   **Проблемы с хостингом:** Платформы вроде GitHub и GitLab вводят лимиты на размер репозитория (например, GitHub рекомендует держать его в пределах 1 ГБ, а мягкий лимит — 5 ГБ).

### 2. Решение: Как работает Git LFS?

Git LFS решает эту проблему, заменяя большие файлы в вашем репозитории на крошечные **текстовые указатели (pointer files)**.

**Вот как это выглядит:**

1.  **В репозитории хранится указатель:** Вместо самого 150-мегабайтного файла `video.mp4`, Git сохраняет небольшой текстовый файл, содержащий уникальный идентификатор (SHA-256 хеш) этого видеофайла.
2.  **Сами файлы хранятся отдельно:** Настоящий `video.mp4` отправляется на специальный LFS-сервер (например, на сервер GitHub или GitLab).
3.  **Прозрачная работа:** Когда вы клонируете репозиторий, Git скачивает указатели. А когда вы переключаетесь на ветку, где нужен `video.mp4`, Git LFS автоматически «подтягивает» реальный файл с LFS-сервера и подменяет им указатель.

**Аналогия:** Git LFS — это как библиотечный каталог. Вместо того чтобы хранить все книги в самом каталоге, вы храните только карточки с их названиями и местоположением на полке. Сам каталог остаётся лёгким, а книги (файлы) находятся на своих местах.

### 3. Первый запуск: Установка и инициализация

Для работы с LFS необходимо установить специальное расширение.

*   **Установка:**
    *   **macOS:** `brew install git-lfs`
    *   **Windows:** Скачайте установщик с [git-lfs.com](https://git-lfs.com)
    *   **Linux (Debian/Ubuntu):** `sudo apt install git-lfs`
*   **Инициализация в системе (делается один раз):** После установки выполните команду, которая настраивает глобальные хуки Git.
    ```bash
    git lfs install
    ```
    Эта команда добавит необходимые настройки в ваш глобальный файл `~/.gitconfig`.

### 4. Отслеживание файлов: `.gitattributes` и `git lfs track`

Теперь нужно указать Git LFS, за какими типами файлов ему следить.

*   **Команда `git lfs track`:** Она добавляет правила отслеживания в специальный файл `.gitattributes`.
    ```bash
    # Отслеживать все PSD-файлы
    git lfs track "*.psd"

    # Отслеживать конкретный файл
    git lfs track "datasets/huge_data.csv"

    # Отслеживать все ISO-образы
    git lfs track "*.iso"
    ```
    После выполнения этой команды Git создаст (или обновит) файл **`.gitattributes`** в корне вашего репозитория. Внутри него будет строка, например: `*.psd filter=lfs diff=lfs merge=lfs -text`.

*   **Важное действие:** Файл `.gitattributes` **обязательно нужно закоммитить** в репозиторий. Это гарантирует, что все члены команды будут использовать одинаковые правила LFS.
    ```bash
    git add .gitattributes
    git commit -m "Настроил LFS для *.psd и *.iso"
    ```

### 5. Основные команды для работы с LFS

*   **`git lfs track` (без аргументов):** Показывает список всех шаблонов файлов, которые сейчас отслеживаются LFS в этом репозитории.
*   **`git lfs ls-files`:** Показывает список конкретных файлов, которые уже находятся под управлением LFS в вашем текущем рабочем состоянии.
*   **`git lfs pull`:** Принудительно скачивает все LFS-объекты для текущего состояния репозитория.
*   **`git lfs push`:** Отправляет LFS-объекты на удалённый сервер. Обычно это происходит автоматически при `git push`, но иногда может потребоваться ручной запуск.

### 6. Пример: Добавляем видеофайл > 100 МБ

1.  Убедитесь, что вы инициализировали LFS: `git lfs install`.
2.  Настройте отслеживание для видеофайлов: `git lfs track "*.mp4"`.
3.  Добавьте ваш большой видеофайл в репозиторий, как обычно:
    ```bash
    cp /path/to/big_video.mp4 .
    git add big_video.mp4 .gitattributes
    git commit -m "Добавил обучающее видео"
    git push origin main
    ```
    При выполнении `git push` вы увидите, как Git LFS загружает файл на сервер.

### 7. Лимиты и ограничения на GitHub и GitLab

Как инженер данных, вы должны знать о существующих лимитах, чтобы не столкнуться с ними в неподходящий момент.

| Платформа | Лимит для Free-аккаунта |
| :--- | :--- |

### 8. Миграция существующего большого файла в LFS

Если большой файл уже был случайно закоммичен в историю Git, простое добавление его в отслеживание LFS не решит проблему — его копия навсегда останется в истории. `git lfs track` не работает ретроспективно. Чтобы это исправить, нужно переписать историю.

Для этого используется команда **`git lfs migrate`**.

**Пример:** Миграция всех существующих PSD-файлов в истории.
```bash
# Убедитесь, что у вас есть резервная копия репозитория!
git lfs migrate import --include="*.psd" --everything
```
Флаг `--everything` указывает, что нужно переписать всю историю во всех ветках. После этого старые версии файлов будут заменены на LFS-указатели, и вам потребуется выполнить `git push --force`, что допустимо только в личных ветках или при полной уверенности и координации с командой.

**Более простой путь для новых файлов:** Если файл ещё не был запушен, можно использовать `git add --renormalize` после настройки трекинга.

### Практическое задание: Осваиваем Git LFS

1.  **Установите Git LFS:** Следуйте инструкциям для вашей ОС и выполните `git lfs install`.
2.  **Создайте новый репозиторий:** `mkdir lfs-lab && cd lfs-lab && git init`.
3.  **Настройте отслеживание:** `git lfs track "*.iso"`.
4.  **Создайте «большой» файл:** Создайте файл-заглушку большого размера, например, с помощью `dd` (на Linux/macOS) или `fsutil` (на Windows). Для простоты можно создать файл в 10 МБ:
    ```bash
    # Linux/macOS
    dd if=/dev/zero of=bigfile.iso bs=1M count=10
    ```
5.  **Закоммитьте изменения:** `git add bigfile.iso .gitattributes && git commit -m "Добавил большой ISO-образ"`.
6.  **Проверьте работу LFS:** Выполните `git lfs ls-files`. Вы должны увидеть `bigfile.iso` в списке.

### Вопросы для самопроверки

1.  **Что хранится в репозитории Git при использовании LFS — сами файлы или ссылки?**
    > **Ответ:** В репозитории Git хранятся только **текстовые указатели (pointer files)** на большие файлы. Сами бинарные файлы хранятся на отдельном LFS-сервере.

2.  **Как правильно мигрировать существующий большой файл, который уже находится в истории Git, в LFS?**
    > **Ответ:** С помощью команды `git lfs migrate import`. Например, для всех `*.zip` файлов: `git lfs migrate import --include="*.zip" --everything`. Это перепишет историю, заменив файлы на указатели LFS. После этого потребуется принудительный пуш (`git push --force`).

3.  **Вы настроили отслеживание `git lfs track "*.png"`, но Git всё равно показывает, что `image.png` — обычный файл. Что вы забыли сделать?**
    > **Ответ:** Нужно не только выполнить команду `git lfs track`, но и закоммитить изменения в файле `.gitattributes` **до** того, как добавлять `*.png` файлы в коммит. Если файл уже был в индексе, его нужно переиндексировать.

4.  **Как посмотреть, какие типы файлов сейчас отслеживаются LFS в вашем репозитории?**
    > **Ответ:** Выполнить команду `git lfs track` (без аргументов). Она покажет список всех шаблонов из `.gitattributes`. Чтобы увидеть конкретные файлы, используйте `git lfs ls-files`.
