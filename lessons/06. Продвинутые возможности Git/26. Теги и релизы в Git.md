Приветствую, коллеги. Сегодня я надену шляпу релиз-инженера, чтобы рассказать вам об одном из самых важных инструментов в арсенале команды, выпускающей продукт в мир, — о **тегах (tags)**. Если коммиты — это точки на временной шкале разработки, то теги — это зарубки с надписью «Здесь была версия 1.0.0». Они позволяют мгновенно вернуться к состоянию кода на момент релиза, сравнить версии или начать срочный хотфикс. Без тегов управление релизами превращается в хаотичный поиск «того самого коммита, который мы залили на прод в прошлый вторник».

### Зачем нужны теги?

В отличие от веток, которые постоянно двигаются вперёд с новыми коммитами, теги — это **неподвижные указатели** на конкретный коммит. Они созданы для того, чтобы навсегда зафиксировать важные моменты истории проекта, прежде всего — **публичные релизы** (`v1.0.0`, `v2.1.3`).

**Основные сценарии использования:**
- Отметка стабильных версий для пользователей.
- Быстрое переключение кода на состояние конкретного релиза для отладки или сборки.
- Автоматизация процессов CI/CD (триггеры на создание тега).

### Два типа тегов: лёгкие и аннотированные

Git предлагает два вида тегов, и выбор между ними важен.

#### Лёгкий тег (Lightweight Tag)

Это просто **указатель на коммит**, аналогичный ветке, но без возможности добавления новых коммитов. Он хранится как файл в `.git/refs/tags/`, содержащий только хеш коммита.

**Создание:**
```bash
git tag v1.0-light
```

**Когда использовать:** Для временных меток «для себя», когда не нужны дополнительные метаданные. Например, отметить промежуточное состояние перед экспериментом.

#### Аннотированный тег (Annotated Tag)

Это **полноценный объект Git**, который хранится в базе данных отдельно от коммита и содержит:
- Имя автора тега.
- Дату создания.
- Сообщение (аналогично коммиту).
- Возможность GPG-подписи для верификации подлинности.

**Создание:**
```bash
git tag -a v1.0.0 -m "Релиз версии 1.0.0: добавлена интеграция с API"
```

**Когда использовать:** **Всегда для публичных релизов.** Аннотированные теги — это стандарт индустрии. Именно они используются на GitHub/GitLab для создания Releases. Они содержат информацию о том, кто и когда официально выпустил версию.

**Как посмотреть информацию о теге:**
```bash
git show v1.0.0
```
Вывод будет содержать информацию о теге (Tagger, Date, Message), а затем информацию о коммите, на который он указывает. Для лёгкого тега покажется только информация о коммите.

### Работа с тегами: основные команды

#### Просмотр списка тегов

```bash
# Все теги
git tag

# Фильтрация по шаблону (например, только версии 1.x)
git tag -l "v1.*"
```

#### Переход на состояние, отмеченное тегом

Вы можете переключить рабочий каталог на состояние тега. Это приведёт вас в состояние **detached HEAD** (отсоединённый HEAD), так как тег не является веткой и не предполагает дальнейших коммитов.

```bash
git checkout v1.0.0
```

Если вам нужно внести исправления на основе тега, следует создать новую ветку:

```bash
git checkout -b hotfix/1.0.1 v1.0.0
```

#### Отправка тегов на удалённый репозиторий

По умолчанию `git push` **не отправляет теги**. Их нужно пушить явно.

**Отправить конкретный тег:**
```bash
git push origin v1.0.0
```

**Отправить все локальные теги разом:**
```bash
git push --tags
```
**Важно:** `git push --tags` отправит **все** теги, включая старые и возможно уже неактуальные. В больших проектах это может привести к замусориванию удалённого репозитория. Рекомендуется пушить теги выборочно.

#### Удаление тегов

**Локально:**
```bash
git tag -d v1.0.0
```

**Удалённо (на сервере):**
```bash
git push origin --delete v1.0.0
# или альтернативный синтаксис
git push origin :refs/tags/v1.0.0
```

### Перемещение тегов (или почему это плохая идея)

Технически Git позволяет **переместить существующий тег** на другой коммит. Для этого нужно сначала удалить тег локально, а затем создать его заново на нужном коммите, после чего принудительно запушить:

```bash
git tag -d v1.0.0
git tag -a v1.0.0 -m "Новое сообщение" <новый_хеш>
git push --force origin v1.0.0
```

**Предостережение релиз-инженера:**
**Никогда не перемещайте теги, которые уже были опубликованы и могли быть использованы другими!** Теги — это якоря стабильности. Если вы передвинете `v1.0.0` на другой коммит, любой, кто уже скачал эту версию или использует её в зависимостях, столкнётся с несоответствием. Если вам нужно выпустить исправление, создайте **новый тег** `v1.0.1`.

### Связь с релизами на GitHub/GitLab

На платформах хостинга репозиториев теги тесно интегрированы с функцией **Releases**.

- Когда вы пушите аннотированный тег, GitHub автоматически предлагает создать на его основе **Release**.
- Release — это страница, которая помимо самого тега содержит описание изменений (Changelog), ссылки на скачивание архива (`.zip`, `.tar.gz`) и возможность прикрепить скомпилированные бинарные файлы (артефакты).
- Именно Releases пользователи видят в разделе «Releases» вашего проекта. Без тега не будет и релиза.

### Практическое задание: Готовим релиз 1.0.0

1.  **Создайте новый репозиторий и сделайте несколько коммитов:**
    ```bash
    mkdir release-lab && cd release-lab
    git init
    echo "# My Awesome Project" > README.md
    git add . && git commit -m "Initial commit"
    
    echo "console.log('Hello');" > app.js
    git add . && git commit -m "feat: add main app logic"
    
    echo "// Helper functions" >> app.js
    git add . && git commit -m "feat: add utilities"
    ```

2.  **Дойдите до состояния, готового к релизу, и поставьте аннотированный тег:**
    ```bash
    git tag -a v1.0.0 -m "Первый официальный релиз проекта. Основная функциональность готова."
    ```

3.  **Проверьте список тегов и информацию о нём:**
    ```bash
    git tag
    git show v1.0.0
    ```

4.  **Создайте удалённый репозиторий на GitHub (пустой, без README) и отправьте туда код с тегом:**
    ```bash
    git remote add origin <ваш-url>
    git push -u origin main
    git push origin v1.0.0
    ```

5.  **Зайдите на GitHub во вкладку «Releases».** Вы увидите свой тег и сможете на его основе создать Release, заполнив описание.

6.  **Бонус: имитация хотфикса.**
    - Создайте ветку `hotfix/1.0.1` от тега: `git checkout -b hotfix/1.0.1 v1.0.0`
    - Внесите исправление: `echo "// Fix critical bug" >> app.js`
    - Закоммитьте: `git commit -am "fix: resolve critical issue"`
    - Поставьте новый тег: `git tag -a v1.0.1 -m "Hotfix: исправлена критическая ошибка"`
    - Отправьте ветку и тег: `git push -u origin hotfix/1.0.1` и `git push origin v1.0.1`

### Вопросы для самопроверки

**1. Как перейти на состояние, отмеченное тегом `v2.0.0`, чтобы просто посмотреть код, не создавая новую ветку?**
> **Ответ:** Использовать команду `git checkout v2.0.0`. Это переведёт репозиторий в состояние **detached HEAD**. Для возврата обратно на ветку (например, `main`) выполнить `git switch main`.

**2. Можно ли переместить уже существующий тег на другой коммит? Стоит ли это делать?**
> **Ответ:** Технически да, удалив тег локально и создав заново, затем принудительно запушив (`--force`). Однако **категорически не рекомендуется** перемещать опубликованные теги. Это нарушает принцип неизменности истории и ломает зависимости у тех, кто уже использует этот тег. Всегда создавайте новый тег с инкрементированной версией (например, `v1.0.1`).

**3. Чем аннотированный тег отличается от лёгкого? Какой тип следует использовать для публичных релизов и почему?**
> **Ответ:** Лёгкий тег — просто указатель на коммит. Аннотированный тег — отдельный объект в базе Git, хранящий автора, дату создания, сообщение и опционально GPG-подпись. Для публичных релизов следует использовать **аннотированные теги**, так как они содержат метаданные, интегрируются с GitHub Releases и являются стандартом индустрии.

**4. Вы выполнили `git push`, но тег `v1.0.0` не появился на GitHub. Почему? Как это исправить?**
> **Ответ:** По умолчанию `git push` не отправляет теги. Нужно явно запушить тег: `git push origin v1.0.0`. Для отправки всех локальных тегов разом: `git push --tags`, но лучше пушить выборочно.

**5. Как полностью удалить тег `v1.0.0` и локально, и с удалённого сервера?**
> **Ответ:**
> Локально: `git tag -d v1.0.0`
> Удалённо: `git push origin --delete v1.0.0`
